##  해결 전략 (Code Before Think)

> `transactions` 테이블을 이용하여 2023년 11월, 12월 주문에 대해 다음 데이터를 추출하는 쿼리를 작성해 주세요.

1. **타겟 데이터:** (어떤 테이블에서 무엇을 뽑아야 하는가?)
   - `transactions` 테이블
	   - `transaction_id` 거래 ID
	   - `purchased_at` 상품 구매시각
	   - `is_online_order` 온라인 구매 여부
1. **조건 분석:**
   - `2023-11-01` 이상 `2024-01-01` 미만
   - is_online_order = true
   - DATE_TRUNC('day', purchased_at)
1. **사용할 문법:**
   - `DATE_TRUNC('day', purchased_at)::date`
   - `TO_CHAR(order_date, 'FMDay')`
   - `LAG() OVER (ORDER BY order_date)`
---
## 정답 쿼리 (Solution)

```sql
-- 여기에 작성한 쿼리를 붙여넣으세요.
WITH daily_orders AS (
    SELECT
        DATE_TRUNC('day', purchased_at)::date AS order_date,
        COUNT(*) AS num_orders_today
    FROM transactions
    WHERE purchased_at >= '2023-11-01'
      AND purchased_at < '2024-01-01'
      AND is_online_order = true
    GROUP BY DATE_TRUNC('day', purchased_at)
)

SELECT
    order_date,
    TO_CHAR(order_date, 'FMDay') AS weekday,
    num_orders_today,
    num_orders_today
      + COALESCE(LAG(num_orders_today) OVER (ORDER BY order_date), 0)
      AS num_orders_from_yesterday
FROM daily_orders
ORDER BY order_date;
```

##  오답 노트 & 배운 점 (Retrospective)

-  **내가 실수한 부분:**
    - 처음에는 `transaction_id` 기준으로 `LAG()`를 사용함
    - >“주문 건수”가 아니라 “행 단위 비교”가 되어버리는 현상이 발생
    - 첫 번째 행에서 `NULL` 발생 → 결과 틀림>LAG / LEAD 사용 시 COALESCE는 거의 필수
-  **새로 알게 된 함수/꿀팁:**
	- 윈도우 함수는 집계 이후에 적용해야 한다!!!!!
	- `FMDay` → 공백 제거된 깔끔한 형태
	- `DATE_TRUNC('day', purchased_at)::date` >“날짜 기준 집계” 할 때 가장 안전한 방식
