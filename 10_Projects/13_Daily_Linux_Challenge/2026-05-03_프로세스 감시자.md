## 📌 랩 목표

Linux 환경에서 시스템의 자원을 사용하는 프로세스를 모니터링하고, 특정 프로세스를 찾아 안전하게 제어하거나, 장시간 실행되는 작업을 백그라운드로 안전하게 넘기는 필수 프로세스 관리 기술을 실습

### 🔷 섹션 1 — 실행 중인 시스템 프로세스 나열하기

**개념 요약**

- **`ps aux` 명령어:** 시스템에서 현재 실행 중인 모든 프로세스의 정적인 상태(스냅샷)를 상세하게 출력합니다.
- 주요 출력 컬럼 의미:
	- `USER`: 프로세스를 실행시킨 사용자 계정
	- `PID`: 프로세스에 부여된 고유 식별 번호 (주민번호와 같은 역할)
	- `%CPU` / `%MEM`: 프로세스가 현재 사용 중인 CPU와 메모리 점유율
	- `COMMAND`: 이 프로세스를 실행할 때 사용된 실제 명령어

**명령어 / 문법**

```bash
ps aux
```

**실습 메모**

`ps aux`만 단독으로 치기보다, 앞서 배운 파이프(`|`)와 조합하여 `ps aux | grep nginx` 처럼 내가 찾고 싶은 특정 프로그램이 켜져 있는지 확인하는 용도로 가장 많이 사용

### 🔷 섹션 2 — 프로세스 리소스 사용량 모니터링

**개념 요약**

- **`top` 명령어:** 프로세스들의 CPU와 메모리 사용량을 실시간(동적)으로 갱신하며 보여주는 표준 모니터링 도구입니다.
- 기본적으로 CPU 사용량이 높은 순서대로 정렬되어 상단에 나타납니다.
-  **`htop` 명령어:** `top`을 시각적으로 더 보기 편하게 개선한 유틸리티입니다. (게이지 바 제공, 마우스 클릭 지원 등)

**명령어 / 문법**

```bash
top
# (추가 패키지 설치 시) `htop`
```

**실습 메모**

장애가 발생해서 서버가 먹통이 되려 할 때 제일 먼저 켜보는 화면이 바로 `top`입니다. 맨 위에 위치한 프로세스가 범인일 확률이 99%이므로, 리소스 낭비의 주범을 즉각적으로 색출할 수 있습니다.

### 🔷 섹션 3 — 주요 프로세스 식별하기

**개념 요약**

-  무턱대고 프로세스를 죽이는 것은 위험합니다. 종료하기 전에 대상이 맞는지 정확히 식별해야 합니다.
- **`pgrep` 명령어:** 프로세스 이름으로 검색하여 해당 프로세스의 **PID 숫자만** 깔끔하게 찾아줍니다.
	-  `-f` 옵션: 프로세스의 이름뿐만 아니라, 실행할 때 함께 입력한 인자(전체 명령줄)까지 포함해서 검색합니다.
- **`ps -p` 명령어:** 특정 PID 번호를 지정하여 그 프로세스만의 상세 정보를 뽑아냅니다.
	-  `-o` 옵션: 내가 보고 싶은 특정 정보(예: `pid`, `ppid`(부모 프로세스 ID), `cmd`)만 골라서 출력하도록 포맷을 지정합니다.

**명령어 / 문법**

```bash
pgrep -f critical_service.sh # 200
ps -p 200 -o pid,ppid,cmd #(위에서 찾은 PID가 200일 경우)
```

**실습 메모**

- 시스템 엔지니어의 핵심 덕목은 '신중함'입니다. 비슷한 이름의 중요 서비스(`critical_service`)를 실수로 끄지 않도록, 죽이기 전 대상의 멱살(PID)을 정확히 쥐고 신원을 한 번 더 확인하는 아주 좋은 습관입니다.

### 🔷 섹션 4 — 오작동하는 프로세스 종료하기

**개념 요약**

- 확인을 마쳤다면, 자원을 갉아먹는 프로세스를 강제로 종료하여 서버를 정상화해야 합니다.
- **`pkill` 명령어:** PID 번호를 몰라도 **프로세스의 이름이나 패턴**으로 지정하여 즉시 종료시키는 명령어입니다.
	- `-f` 옵션: 파일 이름뿐만 아니라 전체 경로 및 명령줄 매개변수까지 일치하는 것을 찾아 종료합니다.
- 확인: `ps aux | grep [스크립트명]`을 다시 쳐봤을 때 검색 명령어(`grep`) 자신만 나오고 원래 스크립트는 안 나온다면 성공적으로 죽은 것입니다.

**명령어 / 문법**

```bash
pkill -f resource_hog.sh
ps aux | grep resource_hog.sh
```

**실습 메모**

`kill [PID]`는 번호를 명시하므로 가장 안전하지만, 번호를 찾는 과정이 귀찮을 때 `pkill [이름]`을 자주 씁니다. 단, `pkill python` 같은 식으로 너무 광범위한 이름을 입력하면 정상적으로 돌고 있던 다른 파이썬 프로그램들까지 몰살당할 수 있으니 극도로 주의해야 합니다.

### 🔷 섹션 5 — 백그라운드 프로세스 시작 및 관리

**개념 요약**

- 긴 시간이 걸리는 작업은 내 SSH 접속이 끊기더라도 서버에서 계속 돌아가도록 **백그라운드 처리**를 해야 합니다.
- **`nohup` (No Hang Up):** 사용자가 로그아웃하거나 터미널 창을 닫아도 프로세스가 죽지 않도록 방어해 줍니다.
- **`&` 연산자:** 명령을 터미널 앞단(포그라운드)이 아닌 백그라운드에서 실행하게 하여, 내가 터미널 프롬프트를 계속 쓸 수 있게 해줍니다.
- `> processor.log 2>&1`의 의미:
	- 기본 출력 결과(stdout)를 `processor.log` 파일에 저장합니다.
	- `2>&1`: 에러 메시지(stderr=2)도 기본 출력(stdout=1)과 같은 곳(즉, 동일한 로그 파일)으로 묶어서 보내라는 뜻입니다.


**명령어 / 문법**

```bash
nohup ./data_processor.sh > processor.log 2>&1 &
cat processor.log
ps aux | grep data_processor
```

**실습 메모**

이건 실무에서 **가장 중요한 마법의 주문** 중 하나입니다. AI 모델 학습, DB 마이그레이션 등 몇 시간이 걸리는 작업을 그냥 실행해뒀다가 노트북 덮는 순간 네트워크가 끊기며 작업이 날아가는 대참사를 막아주는 필수 생존 스킬입니다. 백그라운드 실행 후엔 잊지 말고 `tail -f processor.log`로 로그가 잘 찍히는지 구경하는 재미도 있습니다.
