## 📌 랩 목표

Linux 시스템의 핵심인 커널을 재컴파일하지 않고도 기능을 동적으로 확장할 수 있는 '커널 모듈'의 개념을 이해하고, 모듈의 조회, 의존성 관리, 수동 적재/제거 및 부팅 시 자동 로드 설정 방법을 실습합니다.

### 🔷 섹션 1 — lsmod 와 modinfo 를 이용한 커널 모듈 목록 확인 및 조사

**개념 요약**

- `lsmod` 명령어를 사용하면 깔끔하고 읽기 쉬운 목록을 볼 수 있습니다. 목록이 길어질 수 있으므로 `less` 유틸리티로 출력을 전달하여 편리하게 탐색하겠습니다.
- 모듈 목록이 나타납니다. 출력 결과는 `Module`, `Size`, `Used by` 세 개의 열로 구성됩니다.
	- `Module`: 커널 모듈의 이름입니다.
	- `Size`: 모듈의 크기 (바이트 단위) 입니다.
	- `Used by`: 현재 이 모듈을 사용 중인 다른 모듈이나 프로세스의 수이며, 그 뒤에 해당 모듈에 의존하는 모듈 목록이 표시
-  화살표 키를 사용하여 목록을 스크롤할 수 있습니다. `q`를 누르면 `less` 뷰어를 종료하고 명령 프롬프트로 돌아갑니다
- `modinfo` 명령어는 커널 모듈의 파일 경로, 라이선스, 설명, 작성자 및 의존성과 같은 다양한 세부 정보를 표시합니다.
- 병렬 포트와 관련된 `parport` 모듈을 조사해 보겠습니다. `depends` 필드가 비어 있는 것을 확인할 수 있는데, 이는 이 모듈이 다른 모듈에 의존하지 않음을 의미합니다.
-  모듈은 종종 다른 모듈에 의존하기도 합니다. `depmod` 명령어는 `/lib/modules/$(uname -r)` 디렉토리의 모듈을 분석하여 모듈 의존성 목록을 생성합니다. 이 명령어는 루트 권한이 필요하므로 `sudo`를 사용합니다.
-  커널의 모듈 디렉토리에 `modules.dep`라는 파일을 생성하거나 업데이트합니다. 경로의 `$(uname -r)` 부분은 현재 커널의 릴리스 버전을 삽입하는 명령어 치환 방식입니다.

**명령어 / 문법**

```bash
lsmod | less
modinfo partport
sudo depmod
less /lib/modules/$(uname -r)/modules.dep
```

**실습 메모**

- 시스템에 새로운 하드웨어(예: 그래픽 카드, 특수 네트워크 랜카드 등)를 장착했는데 동작하지 않을 때, 가장 먼저 해당 장치의 드라이버(커널 모듈)가 정상적으로 인식되어 올라와 있는지 점검하는 매우 중요한 기초 트러블슈팅 단계

### 🔷 섹션 2 — rmmod 를 이용한 커널 모듈 제거

**개념 요약**

- `rmmod` 명령어를 사용하여 실행 중인 커널에서 커널 모듈을 제거 (언로드) 하는 방법을 배웁니다. 모듈 제거는 하드웨어 문제를 해결하거나, 시스템 메모리를 확보하거나, 더 이상 필요하지 않은 모듈을 정리할 때 유용합니다.
- 중요한 점은 시스템이나 다른 모듈에서 현재 사용 중이지 않은 모듈만 제거할 수 있다는 것입니다.
- 조이스틱 지원을 제공하는 `joydev` 모듈로 실습을 이어가겠습니다. 먼저 이 모듈이 로드되어 있는지 확인해 봅시다. `lsmod`와 `grep`을 조합하여 출력을 필터링할 수 있습니다.
-  `Used by` 열의 `0`은 모듈이 로드되어 있지만 현재 어떤 프로세스나 다른 모듈에서도 사용하고 있지 않음을 나타냅니다. 만약 이 숫자가 `0` 보다 크다면 `rmmod`는 실패할 가능성이 높으며, 시스템이 의존하고 있는 모듈이 제거되는 것을 방지합니다.
- 이제 `joydev` 모듈을 제거해 보겠습니다. 이 작업은 실행 중인 커널을 수정하므로 `sudo`를 통해 루트 권한을 얻어야 합니다 
- `rmmod` 명령어는 성공 시 보통 아무런 출력을 내보내지 않습니다. 모듈이 성공적으로 제거되었는지 확인하기 위해 다시 한번 `lsmod | grep joydev` 명령어를 실행합니다.

**명령어 / 문법**

```bash
lsmod | grep joydev # joydev                 32768  0
sudo rmmod joydev
lsmod | grep joydev
```

**실습 메모**

- 서버 환경에서는 오작동을 일으키거나 보안상 취약점이 될 수 있는 모듈(예: 안 쓰는 블루투스 모듈, 플로피 디스크 모듈 등)을 강제로 언로드하여 시스템 자원을 최적화하고 공격 표면(Attack Surface)을 줄이는 데 자주 활용

### 🔷 섹션 3 — insmod 를 이용한 커널 모듈 로드

**개념 요약**

- `insmod` 명령어를 사용하여 실행 중인 커널에 커널 모듈을 수동으로 로드하는 방법을 배웁니다. 이는 `rmmod`와 반대되는 작업입니다.
- 최신 시스템에서는 의존성을 자동으로 처리해 주는 `modprobe`라는 더 발전된 도구를 주로 사용하지만, `insmod`를 이해하는 것은 커널 모듈의 작동 원리를 배우는 데 있어 기초가 됩니다.
- 이전 단계에서 제거했던 `joydev` 모듈을 다시 로드해 보겠습니다. `insmod` 명령어를 사용할 때는 `.ko`(Kernel Object) 확장자를 가진 모듈 객체 파일의 **전체 절대 경로**를 제공해야 합니다.
- 커널 모듈의 표준 디렉토리는 `/lib/modules/$(uname -r)/kernel/`이며, `joydev` 모듈은 일반적으로 `drivers/input/` 하위 디렉토리에 위치합니다.
- 이 작업에는 루트 권한이 필요합니다. `rmmod`와 마찬가지로 `insmod`도 성공하면 아무런 출력을 내보내지 않습니다.

**명령어 / 문법**

```bash
sudo insmod /lib/modules/$(uname -r)/kernel/drivers/input/joydev.ko
lsmod | grep joydev #joydev                 32768  0
```

**실습 메모**

- 제조사에서 제공하는 특정 커스텀 장치 드라이버를 테스트하거나, 커널 모듈 개발자가 자신이 방금 컴파일한 `.ko` 파일을 시스템 디렉토리에 복사하지 않고 현재 폴더에서 즉시 로드해 볼 때 주로 사용됩니다.

### 🔷 섹션 4 — 부팅 시 커널 모듈이 자동으로 로드되도록 설정하기

**개념 요약**

- 이 단계에서는 시스템이 시작될 때마다 커널 모듈이 자동으로 로드되도록 설정하는 방법을 배웁니다. `insmod`를 사용하여 수동으로 모듈을 로드하는 것은 일시적이며 재부팅 후에는 날아갑니다.
-  변경 사항을 영구적으로 적용하려면, `systemd`를 사용하는 최신 Linux 시스템의 표준 방법인 `/etc/modules-load.d/` 디렉토리에 설정 파일을 추가해야 합니다.
- 이 디렉토리에 있는 `.conf` 확장자를 가진 모든 파일을 시스템이 읽어 들여, 파일 안에 한 줄에 하나씩 나열된 모든 모듈을 로드하려고 시도합니다.
- `joydev` 모듈이 자동으로 로드되도록 설정해 보겠습니다. `echo`와 `tee` 명령어를 `sudo`와 함께 사용하여 한 번에 처리할 수 있습니다.
	- `echo joydev`: 'joydev'라는 텍스트를 출력합니다.
	- `|`: 파이프는 이 텍스트를 다음 명령어의 입력으로 전달합니다.
	- `sudo tee /etc/modules-load.d/joydev.conf`: 화면에 출력하는 동시에 지정된 파일에 내용을 기록합니다.
- 파일이 올바르게 생성되었는지 확인하기 위해 `cat` 명령어로 내용을 출력해 봅니다.
- 시스템 인프라를 구축할 때, 데이터베이스 서버용 특수 스토리지 드라이버나 방화벽 기능을 위한 커널 모듈 등이 서버 재시작 후에도 무조건 올라와 있도록 설정할 때 필수적으로 거쳐야 하는 작업입니다.

**명령어 / 문법**

```bash
echo joydev | sudo tee /etc/modules-load.d/joydev.conf #joydev
cat /etc/modules-load.d/joydev.conf #joydev
```
