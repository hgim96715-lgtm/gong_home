### 🔷 섹션 1 —  `find` 와 와일드카드를 사용한 기본 파일 검색

**개념 요약**

-  **기본 구조:** `find [경로] [표현식]` (예: `.`은 현재 디렉토리부터 검색)
- **이름 검색:** `-name` (대소문자 구분), `-iname` (대소문자 무시)
- **와일드카드(`*`):** 쉘의 사전 해석을 방지하기 위해 패턴을 큰따옴표(`"*"`)로 묶는 것을 권장.
- **유형 검색:** `-type f` (일반 파일), `-type d` (디렉토리)

**명령어 / 문법**

```bash
find . -name "report.txt"       # 정확한 파일명 검색
find . -name "*.txt"            # 확장자가 txt인 모든 파일 검색
find . -iname "*.txt"           # 대소문자 구분 없이 txt 파일 검색
find . -type d                  # 디렉토리만 검색
find . -type f -name "*.log"    # 일반 파일 중 확장자가 log인 파일 검색
```

**실습 메모**

- **따옴표의 중요성:** `find . -name *.txt`처럼 따옴표를 생략하면, `find`가 실행되기 전에 현재 디렉토리의 파일들로 쉘이 먼저 확장해버려 엉뚱한 결과가 나올 수 있습니다. 패턴에는 항상 `""`를 써야한다.
- **루트 검색 주의:** `find /`처럼 루트부터 검색하면 시스템 전체를 훑느라 성능에 부하를 줄 수 있습니다. 가능하면 특정 하위 경로를 지정

### 🔷 섹션 2 —  `find -exec`와 `xargs` 를 사용하여 검색 결과에 작업 실행하기

**개념 요약**
-  **-exec 옵션:** `find`로 찾은 각 파일에 특정 명령어 실행.
    - 구조: `-exec [명령어] {} \;` (`{}`는 찾은 파일의 경로, `\;`는 명령의 끝을 의미)
- **-ok 옵션:** `-exec`와 동일하나, 실행 전 사용자에게 확인을 요청함. 파일 삭제(`rm`) 등 파괴적인 작업 시 권장.
- **xargs:** 표준 입력(파이프)을 통해 전달받은 항목을 인자로 사용하여 명령어를 실행. `-p` 옵션으로 실행 전 확인 모드 사용 가능.

**명령어 / 문법**

```bash
find . -name "*.txt" -exec ls -l {} \; # -rw-rw-r-- 1 labex labex 0 Jun 26 09:45 ./file1.txt -rw-rw-r-- 1 labex labex 0 Jun 26 09:45 ./report.txt -rw-rw-r-- 1 labex labex 0 Jun 26 09:45 ./subdir/file3.txt

find . -name "*.log" -ok rm {} \; # < rm ... ./file2.log > ? y < rm ... ./subdir/another.log > ? y

ls -R # 하위 디렉토리까지 모두 나열

touch file2.log subdir/another.log # 테스트용 파일 생성
find . -name "*.log" | xargs ls -l # -rw-r--r-- 1 labex labex 0 <date> <time> ./file2.log -rw-r--r-- 1 labex labex 0 <date> <time> ./subdir/another.log

find . -name "*.log" | xargs -p rm
# rm ./file2.log ./subdir/another.log ?...y
```

**실습 메모**

- **공백이 포함된 파일명:** 파일 이름에 공백이 있으면 `xargs`가 이를 별개의 인자로 오해할 수 있습니다. 이럴 땐 `find -print0 | xargs -0` 조합을 사용하면 안전합니다.
- **성능 차이:** `-exec`는 찾은 파일마다 프로세스를 새로 띄우지만, `xargs`는 여러 파일을 한꺼번에 명령어로 넘기기 때문에 대량의 파일을 처리할 때 훨씬 빠릅니다.

### 🔷 섹션 3 —  빠른 데이터베이스 기반 검색을 위해 `locate` 와 `updatedb` 사용하기

**개념 요약**
  - **locate:** 미리 구축된 파일 경로 데이터베이스를 검색하므로 매우 빠름. 단, 데이터베이스 업데이트 이후 생성된 최신 파일은 찾지 못하는 단점이 있음.
- **updatedb:** 파일 데이터베이스를 최신 상태로 수동 갱신하는 명령어(`sudo` 권한 필요).

**명령어 / 문법**

```bash
sudo apt-get update && sudo apt-get install -y mlocate # mlocate 패키지 설치
cd ~/project/find_lab
locate report.txt # /home/labex/project/find_lab/report.txt

sudo updatedblo
locate report.txt # /home/labex/project/find_lab/report.txt

touch special_report.pdf
locate special_report.pdf

sudo updatedb
locate special_report.pdf
```

**실습 메모**

- **데이터베이스 업데이트 빈도:** 시스템은 보통 cron 작업을 통해 `locate` DB를 매일 자동 갱신함.
- **검색 도구 선택:** 즉각적인 결과나 방금 생성된 파일은 실시간 스캔인 `find`가 신뢰성 높음. 빠르고 전반적인 시스템 검색이 필요할 때는 `locate`가 적합.

### 🔷 섹션 4 — whereis 로 명령어 바이너리 및 매뉴얼 위치 찾기

**개념 요약**

- **목적:** 일반 파일이 아닌, 시스템 명령어의 바이너리(실행 파일), 소스 코드, 매뉴얼 페이지 위치를 찾는 데 특화됨.
- **옵션:** `-b` (바이너리 파일만 검색), `-m` (매뉴얼 페이지만 검색)
- **한계:** 표준 시스템 디렉토리 내에서만 검색하므로, 개인 프로젝트 파일(`report.txt` 등)을 찾는 용도로는 부적합함

**명령어 / 문법**

```bash
cd ~/project
whereis passwd #passwd: /usr/bin/passwd /etc/passwd /usr/share/man/man1/passwd.1.gz /usr/share/man/man5/passwd.5.gz

whereis -b passwd # 바이너리 위치만 출력

whereis -m passwd # 매뉴얼 페이지 위치만 출력

whereis report.txt # 표준 디렉토리에 없으므로 경로를 반환하지 못함
```

**실습 메모**
- **환경 변수와의 관계:** `whereis`는 `$PATH` 경로뿐만 아니라 표준적인 시스템 디렉토리들을 미리 정의된 대로 검색합니다. 따라서 내가 직접 만든 커스텀 스크립트 위치는 못 찾을 수도 있습니다.
- **설정 파일 찾기:** 명령어가 참조하는 기본 설정 파일이 어디 있는지 헷갈릴 때 빠르게 체크하기 좋습니다.

### 🔷 섹션 5 —  `alias`, `which`, `type` 으로 명령어 경로 분석하기

**개념 요약**

- - **type:** 명령어의 정체(별칭, 내장 명령어, 외부 프로그램 등)를 확인. `-a` 플래그는 일치하는 모든 명령어와 쉘의 실행 우선순위를 보여줌.
    - _우선순위:_ 별칭(Alias) ➔ 쉘 내장 명령어(Builtin) ➔ 환경변수 경로의 외부 프로그램
- **alias / unalias:** 특정 명령어를 다른 이름이나 옵션이 포함된 형태로 단축 지정(`alias`)하거나 제거(`unalias`).
- **which:** 쉘 환경에서 실행될 명령어의 실제 바이너리 경로(또는 별칭 여부)를 확인.

**명령어 / 문법**

```bash
alias pwd='date'
pwd # <현재 날짜와 시간>


which pwd #pwd: aliased to date

type pwd # pwd is an alias for date

type -a pwd # 이 출력은 쉘의 우선순위를 알려줍니다 pwd is an alias for date pwd is a shell builtin pwd is /usr/bin/pwd pwd is /bin/pwd

unalias pwd
pwd #/home/labex/project

type pwd #pwd is a shell builtin
```

**실습 메모**

- **가장 믿음직한 type:** `which`는 별칭(alias)이나 쉘 내장 명령어(builtin)를 제대로 구별하지 못할 때가 있습니다. 명령어가 어떻게 작동하는지 궁금할 땐 `type -a`가 가장 정확한 정보를 줍니다.
-  **우선순위 역전:** `alias`를 설정하면 기존 시스템 명령어를 덮어쓰게 되므로 매우 조심해야 합니다. (예: `alias rm='rm -i'`) 만약 별칭을 무시하고 원래 명령어를 쓰고 싶다면 명령어 앞에 백슬래시(`\rm`)를 붙이면 됩니다.
