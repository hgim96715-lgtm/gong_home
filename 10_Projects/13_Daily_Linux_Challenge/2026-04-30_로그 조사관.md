## 📌 랩 목표

리눅스의 텍스트 처리 및 비교 도구(`grep`, `diff`)와 출력 리다이렉션(`>`, `>>`, `|`)을 활용하여, 방대한 로그에서 에러를 추출하고 서버 환경 간의 설정 차이를 분석하는 트러블슈팅 능력을 기릅니다.

### 🔷 섹션 1 — 애플리케이션 로그 파일 내용 검토

**개념 요약**

- **`grep` 명령어:** 텍스트 파일 내에서 특정 패턴(문자열)이 포함된 줄만 검색하여 출력하는 강력한 도구입니다.
- **출력 리다이렉션 (`>`):** 화면에 출력될 결과를 파일로 저장합니다. (주의: 기존 파일이 있다면 덮어씁니다.)
- **작동 원리:** `grep "검색어" [대상 파일] > [저장할 파일]`
	- 예시에서 `grep "ERROR"`는 로그 파일에서 에러가 발생한 줄만 골라내어 `error_report.txt`라는 새 파일로 깔끔하게 모아줍니다.

**명령어 / 문법**

```bash
grep "ERROR" ~/project/logs/app.log > ~/project/error_report.txt
```

### 🔷 섹션 2 — 시스템 부팅 메시지 조사

**개념 요약**

- **`dmesg` 명령어:** 시스템 부팅 과정이나 하드웨어와 관련된 커널 메시지를 출력합니다.
- **파이프 (`|`):** 앞 명령어의 결과를 뒤 명령어의 입력으로 전달해 주는 연결 고리입니다.
- `grep`의 고급 옵션:
	- `-i`: 대소문자를 구분하지 않고 검색합니다. (Error, ERROR, error 모두 검색)
	-  `-E`: 정규표현식을 사용하여 여러 패턴을 한 번에 검색합니다. (`'fail|error'`는 fail 또는 error를 찾으라는 의미)
- **권한:** 하드웨어 메시지 접근 시 권한 오류("Operation not permitted")가 나면 `sudo`를 앞에 붙여 관리자 권한으로 실행해야 합니다.

**명령어 / 문법**

```bash
sudo dmesg | grep -E 'fail|error' > ~/project/boot_issue
```

**실습 메모**

- 리눅스의 진정한 강력함은 파이프(`|`)에 있습니다. 각자 단순한 역할을 하는 작은 명령어들을 레고 블록처럼 연결하여 복잡한 데이터를 입맛에 맞게 가공할 수 있습니다.

### 🔷 섹션 3 — 웹 서버 설정 파일 검토

**개념 요약**

- **설정 파일 분석의 중요성:** 애플리케이션 장애는 코드 문제뿐만 아니라, Nginx 같은 웹 서버의 설정(예: `worker_processes` 수치) 문제로도 자주 발생합니다.
- **추가 리다이렉션 (`>>`):** 파일의 기존 내용을 지우지 않고 **맨 끝에 새로운 내용을 덧붙여(Append) 기록**하는 연산자입니다.
- **비교 (`>` vs `>>`):** `>`는 기존 내용을 날려버리지만, `>>`는 안전하게 내용을 누적시킵니다.

**명령어 / 문법**

```bash
 grep "worker_processes" ~/project/config/nginx.conf >> ~/project/error_report.txt
```

**실습 메모**

- 조사 과정에서 수집한 여러 단서들을 하나의 보고서 파일로 취합할 때 `>>` 연산자는 필수적입니다. 실수로 `>`를 사용하여 앞서 고생해서 찾은 에러 로그들을 날려버리는 대참사를 겪지 않도록 주의해야 합니다.

### 🔷 섹션 4 — 스테이징 및 프로덕션 설정 파일 비교

**개념 요약**

- **환경 불일치 문제:** 테스트 환경(스테이징)에서는 잘 되던 코드가 운영 환경(프로덕션)에서 죽는다면, 가장 의심해야 할 것은 두 환경의 '설정 파일 차이'입니다.
- **`diff` 명령어:** 두 텍스트 파일을 줄(Line) 단위로 꼼꼼하게 비교해 줍니다. (`diff file1 file2`)
- 출력 기호 이해하기:
	-  `<`: 첫 번째 파일(file1, 스테이징)에만 있는 내용입니다.
	-  `>`: 두 번째 파일(file2, 프로덕션)에만 있는 내용입니다.
- **주의:** 비교 기준이 되므로 입력하는 **파일의 순서가 매우 중요**합니다.

**명령어 / 문법**

```bash
diff ~/project/config/staging/app.conf ~/project/config/production/app.conf > ~/project/config_diff.txt
 cat ~/project/config_diff.txt 
```

**실습 메모**

- "제 PC에서는 잘 되는데요?" 라는 개발자의 변명을 가장 확실하게 검증하는 도구가 바로 `diff`입니다. 배포 장애 시 두 서버의 `.conf` 파일을 `diff`로 찍어보는 것은 엔지니어의 반사신경과도 같아야 합니다.

### 🔷 섹션 5 — 서버 간 디렉터리 일관성 확인

**개념 요약**

- **디렉터리 비교:** 설정 파일 내부의 텍스트뿐만 아니라, "스테이징에는 있는데 프로덕션에는 빠진 파일"이 있는지 폴더 전체를 비교할 수 있습니다.
- **`-r` (Recursive) 옵션:** 디렉터리 안에 있는 하위 폴더와 파일들까지 전부 재귀적으로 파고들어가며 비교하라는 핵심 옵션입니다.
- **순서에 따른 관점:** `diff -r [스테이징] [프로덕션]` 순서로 입력하면, "스테이징 디렉터리 기준"으로 누락되거나 다른 파일을 찾아줍니다.
- **결과 해석:** `Only in /경로/스테이징: asset2.js` 라고 뜬다면, 프로덕션 서버에 해당 파일이 복사되지 않았다는 강력한 장애 단서가 됩니다.

**명령어 / 문법**

```bash
diff -r /home/labex/project/server1_files /home/labex/project/server2_files >/home/labex/project/missing_files.txt
cat /home/labex/project/missing_files.txt Only in /home/labex/project/server1_files: asset2.js
```

**실습 메모**

- 실무 배포 과정에서 특정 파일 전송이 누락되어 웹페이지가 깨지는 일이 빈번합니다. 수백 개의 파일이 담긴 폴더를 일일이 대조하는 대신, `diff -r` 한 줄이면 누락된 1개의 파일을 1초 만에 찾아낼 수 있습니다.
