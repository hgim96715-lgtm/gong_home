## 📌 랩 목표

Linux 파일 시스템의 핵심 보안 체계인 소유권(Ownership)과 허가권(Permissions)을 이해하고, `chmod`, `chown`, `setgid` 등을 활용하여 실제 협업 환경에 맞는 안전한 디렉터리 접근 제어 기술을 실습합니다.

---

## 📖 학습 내용

> 섹션별로 복사해서 사용

---

### 🔷 섹션 1 — 새 프로젝트를 위한 보안 파일 생성

**개념 요약**

- **`touch` 명령어:** 내용이 없는 빈 파일을 빠르게 생성할 때 사용합니다.
- 권한의 숫자 계산 (Octal mode):
	- 읽기(Read, r) = 4
	- 쓰기(Write, w) = 2
	- 실행(Execute, x) = 1
- `chmod 600`의 의미:
	- **6 (소유자):** 읽기(4) + 쓰기(2). 소유자는 파일을 읽고 쓸 수 있습니다.
	-  **0 (그룹), 0 (기타 사용자):** 권한 없음. 다른 사람은 아예 접근할 수 없습니다.
	- **출력 결과:** `-rw-------` 형태로 가장 강력한 파일 보안 상태가 됩니다.

**명령어 / 문법**

```bash
touch project_keys.txt
chmod 600 project_keys.txt
ls -l ~/project/phoenix_project/
```

**실습 메모**

- 실무에서 `.env` (환경변수 파일)나 `id_rsa` (프라이빗 SSH 키)를 다룰 때 필수적인 설정입니다. 만약 이런 파일들의 권한이 너무 열려있으면(예: 777), 시스템이나 Git과 같은 프로그램이 보안 취약점으로 판단하여 파일 사용 자체를 강제로 거부하기도 합니다.

### 🔷 섹션 2 — 프로젝트 리소스의 소유권 할당

**개념 요약**

-  **`chown` (Change Owner):** 파일이나 디렉터리의 주인을 바꾸는 명령어입니다.
- **사용법:** `chown [사용자]:[그룹] [대상경로]` 형태로 사용자와 소속 그룹을 한 번에 변경할 수 있습니다.
- **`-R` 옵션 (Recursive):** 지정한 폴더뿐만 아니라 그 안에 들어있는 모든 하위 파일과 폴더의 소유권까지 한꺼번에 변경합니다.
- 파일 소유권을 넘기는 강력한 작업이므로 반드시 관리자 권한(`sudo`)이 필요합니다.

**명령어 / 문법**

```bash
sudo chown -R dev_lead:developers ~/project/phoenix_project
ls -ld ~/project/phoenix_project/
ls -l ~/project/phoenix_project/
```

**실습 메모**

- 웹 서버가 이미지를 띄우지 못하거나 로그 파일을 저장하지 못하는 장애의 원인은 '권한(chmod)' 문제보다 '소유권(chown)' 불일치 문제인 경우가 훨씬 많습니다. `-R` 옵션은 수백 개의 파일 소유권을 한 번에 맞출 때 엔지니어가 가장 자주 쓰는 치트키입니다.

### 🔷 섹션 3 — 메인 프로젝트 디렉터리 보안 설정

**개념 요약**

- **디렉터리의 "실행(x)" 권한:** 파일의 실행 권한과는 뜻이 다릅니다. 폴더에서의 실행 권한은 **해당 폴더 안으로 진입(`cd`)할 수 있는지 여부**를 의미합니다.
- `chmod 750`의 의미:
	- **7 (소유자):** 읽기, 쓰기, 진입 가능 (rwx)
	-  **5 (그룹):** 읽고 진입은 가능하지만, 쓰기(파일 생성/수정)는 불가능 (r-x)
	- **0 (기타):** 아예 접근 불가능 (---)
- 이번 섹션에서는 `-R` 옵션을 쓰지 않았기 때문에, 안에 있는 파일들은 건드리지 않고 '폴더 자체'의 문단속만 단단히 합니다.

**명령어 / 문법**

```bash
sudo chmod 750 ~/project/phoenix_project
ls -ld ~/project/phoenix_project/
```

**실습 메모**

 - 서버 보안 감사 시 가장 중요하게 보는 포인트입니다. 프로젝트의 최상위 폴더를 `750`으로 막아두면, 그 안쪽에 있는 파일이 실수로 `777`로 활짝 열려있다 하더라도, 외부인은 최상위 폴더의 문턱조차 넘을 수 없기 때문에 시스템이 안전하게 보호됩니다.

### 🔷 섹션 4 — 개발팀을 위한 협업 권한 설정

**개념 요약**

- **`setgid` (특수 권한 숫자 2):** 이 권한이 설정된 폴더 안에서 새로운 파일을 만들면, 파일을 만든 사람의 그룹이 아니라 **해당 폴더의 그룹 소유권을 강제로 상속**받습니다.
- **`chmod 2770`의 의미:**
    - `2`: setgid 특수 권한 활성화
    - `770`: 소유자(rwx)와 그룹(rwx)은 완전한 권한을 갖고, 기타 사용자는 접근 불가(---)
- **확인 방법:** 권한을 확인했을 때 그룹의 실행 자리(`x`)가 소문자 `s`로 바뀌어 있으면(예: `drwxrws---`), `setgid`가 성공적으로 적용된 것입니다.

**명령어 / 문법**

```bash
sudo chmod 2770 ~/project/phoenix_project/src
ls -ld ~/project/phoenix_project/src
```

**실습 메모**

- 다수의 개발자가 리눅스 서버 하나에서 소스 코드를 공유할 때 반드시 마주치는 "너 파일이라서 내가 수정이 안 돼!"라는 권한 충돌 문제를 아주 우아하게 해결하는 실무 핵심 노하우입니다. 폴더 권한의 `s` 마크 하나가 팀 전체의 작업 생산성을 결정합니다.
