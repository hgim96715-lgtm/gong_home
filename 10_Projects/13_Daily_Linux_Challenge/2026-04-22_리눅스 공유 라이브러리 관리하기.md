## 📌 랩 목표

Linux 시스템에서 프로그램이 실행되는 데 필요한 공유 라이브러리(Shared Library)의 의존성을 파악하고, 라이브러리의 물리적 위치 검색 및 시스템 캐시(`ldconfig`)를 관리하는 방법을 실습하여 동적 링킹(Dynamic Linking) 구조를 이해합니다.

---

## 📖 학습 내용

---

### 🔷 섹션 1 — ldd 를 이용한 공유 라이브러리 의존성 식별

**개념 요약**

- `ldd` (List Dynamic Dependencies) 명령어는 실행 파일이나 라이브러리가 작동하기 위해 의존하고 있는 공유 라이브러리(`.so` 파일) 목록을 출력합니다.
- 출력 결과의 `{text}=>` 기호는 시스템의 동적 링커가 해당 라이브러리를 실제 파일 시스템의 어느 경로에서 찾았는지를 보여줍니다.
- 대표적인 라이브러리 역할:
	- `libc.so.6`: 파일, 메모리 등 운영체제 기본 기능을 제공하는 표준 C 라이브러리
	- `linux-vdso.so.1`: 시스템 콜 속도 향상을 위해 커널이 메모리에 매핑하는 가상 라이브러리
	- `ld-linux-x86-64.so.2`: 다른 공유 라이브러리를 로드하는 프로그램 인터프리터 역할

**명령어 / 문법**

```bash
ldd /bin/ping

```

**실습 메모**

-  특정 프로그램을 실행했는데 "cannot open shared object file: No such file or directory"라는 에러가 발생한다면, 가장 먼저 `ldd [실행파일명]`을 입력해 보세요. 출력 결과 중 `not found`라고 뜨는 항목이 바로 시스템에 누락된 라이브러리이므로 신속한 트러블슈팅이 가능

### 🔷 섹션 2 — find 를 이용한 특정 공유 라이브러리 위치 찾기

**개념 요약**

- `find` 명령어는 디렉토리 계층 구조 내에서 지정한 조건(이름 등)과 일치하는 파일을 물리적으로 검색하는 강력한 도구입니다. (`find [경로] -name [파일명]`)
- 라이브러리 파일은 주로 `/usr`나 `/lib` 계열 디렉토리에 보관됩니다.
- 전체 파일 시스템(`/`)을 대상으로 검색할 수도 있으나, 보호된 시스템 디렉토리를 탐색할 때 발생하는 "Permission denied" 오류를 피하려면 `sudo`를 함께 사용해야 합니다.
- `find`는 정확하지만 디스크 전체를 뒤지기 때문에 속도가 느린 편입니다.

**명령어 / 문법**

```bash
sudo find /usr -name libc.so.6 # /usr/lib/x86_64-linux-gnu/libc.so.6
sudo find / -name libc.so.6
```

**실습 메모**

- 개발자가 비표준 경로(예: `/opt` 또는 사용자 홈 디렉토리)에 커스텀 라이브러리를 설치해 둔 경우, 시스템이 기본적으로는 찾지 못할 수 있습니다. 이때 `find /` 명령어로 파일의 실제 위치를 먼저 찾아낸 뒤, 환경 변수(`LD_LIBRARY_PATH`)를 설정해주어야 한다.

### 🔷 섹션 3 — ldconfig -p 를 이용한 공유 라이브러리 캐시 확인

**개념 요약**

- 리눅스는 프로그램을 실행할 때마다 느린 `find` 방식으로 라이브러리를 찾는 대신, `/etc/ld.so.cache` 파일에 미리 색인(캐시)을 만들어 두고 빠르게 참조합니다.
- `ldconfig -p` 옵션(print)을 사용하면 현재 시스템 캐시에 등록된 모든 공유 라이브러리 목록을 화면에 출력할 수 있습니다.
- 목록이 너무 길기 때문에 파이프(`|`)와 `grep` 명령어를 결합하여 원하는 라이브러리만 빠르게 필터링해 확인합니다.

**명령어 / 문법**

```bash
ldconfig -p
ldconfig -p | grep libc.so.6 # libc.so.6 (libc6,x86-64, OS ABI: Linux 3.2.0) => /lib/x86_64-l
```

**실습 메모**

- 파일 시스템에 파일이 분명히 존재(`find`로는 나옴)하는데도 프로그램이 라이브러리를 못 찾는 에러를 뿜어낼 때가 있습니다. 이 경우 `ldconfig -p | grep`으로 조회해 보면 십중팔구 캐시에 등록되지 않은 상태입니다. OS가 해당 라이브러리의 존재를 '인지'하고 있는지 검증하는 필수 단계

### 🔷 섹션 4 — ldconfig -v 를 이용한 공유 라이브러리 캐시 재구성

**개념 요약**

- `ldconfig` 명령어를 옵션 없이(또는 `-v`와 함께) 실행하면 `/etc/ld.so.conf` 파일 등에 정의된 신뢰할 수 있는 디렉토리를 쫙 스캔하여 라이브러리 캐시를 최신 상태로 갱신합니다.
  - `-v` (verbose) 옵션을 주면 어떤 디렉토리를 스캔하고 있고 어떤 심볼릭 링크를 만들고 있는지 상세 과정을 화면에 출력해 줍니다.
  - 시스템 핵심 설정 파일을 수정하는 작업이므로 반드시 `sudo` (관리자 권한)로 실행

**명령어 / 문법**

```bash
sudo ldconfig -v
```

**실습 메모**

- 실무에서 패키지 매니저(`yum`, `apt`)를 통하지 않고 소스 코드를 직접 컴파일(`make install`)하여 소프트웨어를 설치한 경우, 라이브러리 파일이 시스템에 복사되더라도 즉시 인식되지 않습니다. 소스 설치 후에는 반드시 `sudo ldconfig`를 한번 실행해 주어야 하는 것이 시스템 관리자의 기본
