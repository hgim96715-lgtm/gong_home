## 📌 랩 목표

Linux 시스템에서 하드웨어 장치(블록 장치, PCI/SCSI 버스 등)를 인식하고 관리하는 구조를 이해합니다. `lsblk`, `lspci`, `udevadm` 같은 필수 명령어와 가상 파일 시스템(`/proc`, `/sys`)을 탐색하여 시스템의 로우 레벨 하드웨어 리소스를 진단하는 방법을 실습합니다.

### 🔷 섹션 1 — lsblk 및 lshw 를 사용하여 블록 장치 나열하기

**개념 요약**

- **블록 장치:** 하드 드라이브, SSD, USB 등 고정된 크기의 블록 단위로 데이터를 전송하는 저장 장치입니다.
- `lsblk`: `sysfs`와 `udev` 데이터베이스를 읽어 사용 가능한 모든 블록 장치를 트리 구조로 한눈에 보여줍니다.
	- `NAME` (장치명), `MAJ:MIN` (커널 식별 번호), `RM` (이동식 여부), `SIZE` (크기), `RO` (읽기 전용 여부), `TYPE` (장치 유형), `MOUNTPOINTS` (마운트 경로) 등의 정보를 제공합니다.
- `lshw`: `lsblk`보다 훨씬 상세한 하드웨어 스펙 정보를 제공합니다. (기본 설치가 안 되어 있을 수 있어 설치 필요)
	- `-class storage`: 스토리지 컨트롤러 정보 확인
	- `-class volume`: 파티션 및 논리 볼륨의 상세 정보 확인

**명령어 / 문법**

```bash
lsblk
sudo apt-get update && sudo apt-get install -y lshw
sudo lshw -class storage
sudo lshw -class volume
```

**실습 메모**

-  새로운 디스크를 서버에 꽂았을 때 가장 먼저 입력하는 명령어가 바로 `lsblk`입니다. 디스크가 커널에 정상적으로 인식되었는지, 마운트는 어디에 되어 있는지 직관적으로 파악할 수 있어 스토리지 관리의 0순위 도구로 활용됩니다.

### 🔷 섹션 2 — udevadm 을 사용하여 장치 속성 점검하기

**개념 요약**

- **udev:** 리눅스 커널의 장치 관리자로, `/dev` 디렉토리 내의 장치 파일(노드)을 동적으로 생성하고 관리합니다.
- `udevadm info`: `udev` 데이터베이스를 조회하여 특정 장치의 상세 속성을 확인하는 명령어입니다.
	- `-q property -n [장치명]`: 모델명(`ID_MODEL`), 제조사(`ID_VENDOR`), 고유 시리얼(`ID_SERIAL`) 등 장치의 고유 속성을 출력합니다. 이 정보는 고정된 장치명(udev 규칙)을 맵핑할 때 필수적입니다.
	- `-q path`: 해당 장치의 `sysfs` 내 실제 경로를 추적합니다.
	- `-a -p [경로]`: 지정된 장치부터 최상위 상위 장치(부모 장치)까지의 모든 계층적 속성을 한 번에 보여줍니다.

**명령어 / 문법**

```bash
udevadm info -q property -n /dev/vda
udevadm info -q path -n /dev/vda # /devices/pci0000:00/0000:00:04.0/virtio1/block/vda
udevadm info -a -p $(udevadm info -q path -n /dev/vda)
```

**실습 메모**

- 외장 하드나 USB를 꽂을 때마다 `/dev/sdb`, `/dev/sdc` 등으로 이름이 바뀌어 스크립트나 마운트가 꼬이는 경우가 많습니다. 이때 `udevadm`으로 고유 시리얼을 추출하여 영구적인 심볼릭 링크(예: `/dev/my_usb`)를 만드는 커스텀 규칙(udev rule)을 작성할 때 핵심적으로 쓰입니다.

### 🔷 섹션 3 — PCI 및 SCSI 버스 계층 구조 보기

**개념 요약**

- **PCI 및 SCSI 버스:** 서버나 PC 내에서 하드웨어 부품들이 통신하는 핵심 데이터 통로입니다. 이 구조를 알면 장치의 물리적 연결 상태를 진단할 수 있습니다.
- `lspci` (`pciutils` 패키지): PCI 버스에 연결된 장치 목록을 출력합니다.
	- `-t`: 연결 상태를 트리(Tree) 구조로 시각화하여 보여줍니다.
	- `-tv`: 트리 구조와 함께 장치 이름 및 사용 중인 커널 드라이버 등 상세(verbose) 정보를 표시합니다.
- `systool` (`sysfsutils` 패키지): SCSI 기반 장치들을 사용자 친화적으로 조회할 때 사용합니다.
	-  `-b scsi`: SCSI 버스에 물려있는 장치들만 나열합니다.

**명령어 / 문법**

```bash
sudo apt-get update && sudo apt-get install -y pciutils
lspci -t
lspci -tv | more
sudo apt-get install -y sysfsutils
systool에 -b scsi 
```

**실습 메모**

- 고가의 GPU나 10G 고속 네트워크 카드를 서버에 장착했는데 인식이 안 될 때, 해당 장치가 메인보드 버스 슬롯에 물리적으로 잘 꽂혔는지, 호환되는 커널 드라이버가 제대로 잡혔는지를 `lspci`로 1차 검증합니다.

### 🔷 섹션 4 — /proc 에서 로우 레벨 시스템 리소스 점검하기

**개념 요약**

- **`/proc` 가상 파일 시스템:** 디스크에 존재하는 실제 파일이 아니라, 커널과 프로세스가 사용하는 하드웨어 리소스를 실시간 텍스트 형태로 노출해 주는 메모리 기반 가상 디렉토리입니다.
- `cat /proc/dma`: 현재 사용 중인 하드웨어 DMA(Direct Memory Access) 채널을 확인합니다.
- `cat /proc/interrupts`: 각 CPU 코어별로 수신된 하드웨어 인터럽트(IRQ) 횟수와 이를 처리하는 장치 드라이버를 보여줍니다. 하드웨어 충돌을 디버깅할 때 핵심적인 지표입니다.
- `less /proc/ioports`: CPU가 특정 장치와 통신하기 위해 할당받은 메모리 주소(I/O 포트) 영역을 나열합니다.

**명령어 / 문법**

```bash
cat /proc/dma
cat /proc/interrupts
less /proc/ioports
```

**실습 메모**

- 특정 네트워크 카드나 디스크 컨트롤러의 성능이 안 나올 때 `/proc/interrupts`를 확인해 보면, 특정 CPU 코어에만 과도하게 인터럽트 부하가 몰려 있는 병목 현상을 발견할 수 있습니다. 이를 분산시키는 튜닝(IRQ Affinity 설정)의 기초 자료로 쓰입니다.

### 🔷 섹션 5 — /sys 파일 시스템에서 장치 정보 탐색하기

**개념 요약**

- **`/sys` 파일 시스템 (sysfs):** `/proc`이 프로세스/메모리 중심이라면, `/sys`는 커널이 인식한 **하드웨어 장치 모델의 계층 구조**를 명확하게 분류해 놓은 가상 파일 시스템입니다.
- **`/sys/block`**: 블록 장치들이 모여 있습니다. 대부분 `/sys/devices` 하위의 실제 물리적 하드웨어 경로를 가리키는 심볼릭 링크 형태로 존재합니다.
- 장치 정보 직접 읽기: 특정 장치 폴더(예: `/sys/block/vda`) 내부의 텍스트 파일을 읽어 장치 스펙을 가져옵니다.
	- 예: `cat /sys/block/vda/size` → 장치의 총 크기 (512바이트 섹터 단위) 출력. 파티션(`vda1`, `vda2` 등)도 각각 독립된 디렉토리를 가집니다.
- **`/sys/bus`**: 장치가 연결된 통신 방식(버스 유형)별로 분류해 둔 디렉토리입니다. (예: `/sys/bus/virtio/devices` 내부에는 가상화 장치인 virtio 기반 장치들만 나열됨)
- 상위 도구(`lsblk`, `lspci` 등)가 시스템 정보를 표시할 때 사용하는 원천 데이터(Raw data)가 바로 이곳입니다.

**명령어 / 문법**

```bash
ls -l /sys/block
ls /sys/block/vda
cat /sys/block/vda/size
ls /sys/bus
ls -l /sys/bus/virtio/devices
```

**실습 메모**

- 셸 스크립트로 자동화 시스템을 구축할 때 텍스트가 복잡한 `lsblk`나 `lshw`의 출력을 `grep`이나 `awk`로 억지로 자를 필요가 없습니다. 그냥 `cat /sys/block/vda/size`처럼 특정 파일 하나만 직접 읽어오는 것이 가장 빠르고 오류가 적은 고급 스크립팅 방식입니다.
