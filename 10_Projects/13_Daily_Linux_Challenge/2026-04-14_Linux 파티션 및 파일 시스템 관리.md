## 📌 랩 목표

Linux 환경에서 새로운 디스크를 인식시키고 파티션 생성(`fdisk`), 파일 시스템 포맷(`mkfs`), 마운트(`mount`) 및 재부팅 후에도 유지되는 영구 마운트(`fstab`)를 구성하며, 가상 메모리인 스왑(Swap) 파티션을 추가하고 관리하는 전체 스토리지 구성 라이프사이클을 실습합니다.

### 🔷 섹션 1 —  디스크 점검 및 fdisk 를 이용한 새 Linux 파티션 생성

**개념 요약**

- `fdisk` 유틸리티를 사용하여 보조 디스크에 새 파티션을 생성합니다. (루트 권한 필요)
- `lsblk` 명령은 디스크 장치를 명확한 트리 구조로 보여줍니다. `-l` 플래그는 장치의 파티션 테이블을 나열합니다.
- `disk` 내에서는 `n`(생성), `p`(주 파티션), `1`(파티션 번호), `+500M`(크기 지정), `p`(확인), `w`(저장 및 종료)의 대화형 명령어를 사용합니다.
- 파티션 테이블을 기록한 후 커널이 즉시 파티션을 인식하지 못할 경우, `partprobe` 명령을 사용하여 운영 체제 커널에 파티션 테이블을 다시 읽도록 요청합니다.

**명령어 / 문법**

```bash
lsblk
sudo fdisk -l /dev/sdb

sudo fdisk /dev/sdb

sudo partprobe
lsblk /dev/sdb

PARTITION_DEVICE=$(lsblk -lno NAME /dev/sdb | grep p1 | head -1) 
echo "Partition device: /dev/$PARTITION_DEVICE"

sudo ln -s /dev/$PARTITION_DEVICE /dev/sdb1
lsblk /dev/sdb1
```

**실습 메모**

- `fdisk`는 디스크의 물리적 구역을 나누는 가장 기초적인 작업입니다. 대화형 인터페이스이므로 실수하더라도 `w` (Write)를 누르기 전까지는 디스크에 실제 반영되지 않습니다. 변경 사항이 의심스러울 때는 `q`를 눌러 저장하지 않고 빠져나올 수 있습니다. `partprobe`는 서버 재부팅 없이 파티션 테이블 갱신을 가능하게 해주는 매우 유용한 실무 명령어입니다.

### 🔷 섹션 2 — mkfs.ext4 를 이용한 ext4 파일 시스템 생성 및 포맷

**개념 요약**

- 파일 시스템을 생성하는 명령은 "make filesystem"의 약자인 `mkfs`입니다.
- 파티션의 기존 데이터를 모두 삭제하는 파괴적인 작업이므로 `sudo` 권한이 필요합니다.
- `blkid` 명령은 블록 장치의 파일 시스템 유형과 UUID 등 속성을 출력하는 데 유용합니다
- `dumpe2fs -h` 명령으로 슈퍼블록(파일 시스템 메타데이터) 정보를 표시할 수 있으며, 여기서 `Filesystem magic number` 및 `Filesystem state` 등을 확인할 수 있습니다.

**명령어 / 문법**

```bash
sudo mkfs.ext4 /dev/sdb1
sudo blkid /dev/sdb1
sudo dumpe2fs -h /dev/sdb1
```

**실습 메모**

- 파티션을 나누는 것(`fdisk`)이 땅에 금을 긋는 것이라면, 포맷(`mkfs`)은 그 땅에 건물을 지을 수 있도록 평탄화하고 도로를 까는 작업입니다. `blkid` 명령어를 통해 확인되는 UUID는 이 파티션의 고유한 '주민등록번호'와 같으며, 향후 시스템이 장치를 정확히 식별하는 데 매우 중요하게 사용됩니다.

### 🔷 섹션 3 — 파일 시스템 마운트, 테스트 및 언마운트

**개념 요약**

- 포맷된 파티션을 운영 체제의 파일 시스템 트리(디렉토리)에 연결하는 과정을 "마운트(mount)"라고 하며, 연결되는 디렉토리를 "마운트 포인트(mount point)"라고 부릅니다
- 마운트가 완료되면 해당 파티션을 일반 디렉토리처럼 사용할 수 있습니다. `df -h` 및 `mountpoint` 명령으로 상태를 확인합니다.
- `lost+found` 디렉토리는 ext4의 표준 기능으로 파일 시스템 손상 복구 시 사용됩니다
- `umount` 명령으로 마운트를 해제합니다. 현재 작업 디렉토리가 마운트 포인트 내부인 경우 "target is busy" 오류가 발생하며, 해제하려면 먼저 그 디렉토리 밖으로 나가야 합니다.

**명령어 / 문법**

```bash
sudo mkdir /mnt/data
sudo mount /dev/sdb1 /mnt/data

# 마운트 포인트에 파일 시스템이 마운트되어 있는지 확인 
mountpoint /mnt/data

df -h /mnt/data
mount | grep /mnt/data


touch /mnt/data/testfile
sudo chown labex:labex /mnt/data
touch /mnt/data/testfile
ls -l /mnt/data

cd /mnt/data
sudo umount /mnt/data # 사용중 > 오류 umount: /mnt/data: target is busy.

cd ~
sudo umount /mnt/data
mountpoint /mnt/data
sudo rmdir /mnt/data
```

**실습 메모**

- 윈도우의 'D드라이브' 개념과 달리, 리눅스는 모든 저장소를 디렉터리 트리의 일부로 편입시킵니다. `target is busy` 오류는 실무에서 가장 흔히 겪는 실수 중 하나입니다. 내가 지금 해제하려는 폴더 안에 들어가 있거나, 다른 프로세스가 해당 폴더의 파일을 열고 있지 않은지 항상 확인해야 합니다.

### 🔷 섹션 4 —  /etc/fstab 에서 영구 마운트 구성

**개념 요약**

- `mount` 명령은 일시적이며 재부팅 시 초기화됩니다. 영구 마운트를 위해 `/etc/fstab` 파일에 항목을 추가합니다.
- /dev/sdb1 같은 장치 이름은 부팅 순서에 따라 바뀔 수 있으므로, 재부팅 시에도 변하지 않는 고유 식별자인 `UUID`를 사용하는 것이 권장됩니다.
- fstab 편집 전에는 반드시 `cp` 명령으로 백업을 생성해야 합니다.
- 편집을 마친 후 `mount -a` 명령을 실행하면 재부팅 없이 fstab에 나열된 설정을 테스트하고 마운트할 수 있습니다.

**명령어 / 문법**

```bash
sudo cp /etc/fstab /etc/fstab.bak

sudo mkdir /data
sudo blkid /dev/sdb1 
sudo mount -a

df -h | grep /data
```

**실습 메모**

- `/etc/fstab`은 리눅스 부팅에 치명적인 영향을 미치는 핵심 파일입니다. 여기에 오타가 있거나 존재하지 않는 UUID를 적어두면 부팅 시 응급 복구 모드(Emergency Mode)로 빠지게 됩니다. 따라서 설정 추가 후 무작정 재부팅하지 않고, `mount -a` 명령어로 문법적 오류나 마운트 이상이 없는지 먼저 검증하는 것이 시스템 관리자의 기본 수칙입니다

### 🔷 섹션 5 — Linux 스왑 파티션 생성 및 관리

**개념 요약**

- 스왑(Swap) 파티션은 물리적 RAM이 가득 찼을 때 시스템이 가상 메모리로 사용하는 특수한 공간입니다.
- `fdisk`에서 파티션을 생성할 때, 유형(`t` 명령)을 기본(Linux, 83)에서 'Linux swap (82)'으로 변경해야 합니다.
- `mkswap` 명령으로 스왑 전용 파일 시스템을 구성하고, `swapon`으로 활성화, `swapoff`로 비활성화합니다.
- `free -h` 명령을 통해 스왑 메모리의 전체 용량 및 현재 사용량을 모니터링할 수 있습니다.

**명령어 / 문법**

```bash
sudo fdisk /dev/sdb  # n -> p -> 2 -> +256M -> t -> 82 -> w
sudo partprobe

sudo mkswap /dev/sdb2
free -h
sudo swapon /dev/sdb2
free -h
swapon -s

sudo swapoff /dev/sdb2
free -h
```

**실습 메모**

- - - 스왑은 메모리 부족(OOM)으로 인해 중요한 프로세스(DB, 웹서버 등)가 강제 종료되는 것을 막아주는 최후의 보루입니다. 일반 데이터 저장소와 목적이 완전히 다르기 때문에 `mkfs.ext4`가 아닌 `mkswap`으로 포맷한다는 점, 그리고 `mount`가 아닌 `swapon`으로 메모리에 적재한다는 차이점을 명확히 인지해야 합니다.
