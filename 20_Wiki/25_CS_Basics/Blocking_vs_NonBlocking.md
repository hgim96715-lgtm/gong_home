## 개념 한 줄 요약

**"커피숍 진동벨"** 로 이해하면 가장 쉬워.

* **Blocking (블로킹):** 커피 주문하고 나올 때까지 카운터 앞에 서서 **하염없이 기다리는 것.** (뒷사람 주문 못 받음)
* **Non-Blocking (논블로킹):** 주문하고 **진동벨** 받은 뒤, 자리에 가서 **폰 보거나 딴짓하는 것.** (직원은 뒷사람 주문 계속 받음)

----
## 왜 필요한가 (Why)

컴퓨터 부품 간의 **"속도 차이"** 때문이야.

* **CPU (두뇌):** 빛의 속도로 일함. ⚡️
* **I/O (입출력 - 디스크/네트워크):** CPU에 비하면 거북이 수준. 🐢
* **문제점:** 파일을 읽어오라고 시켰는데, 파일이 다 읽힐 때까지 슈퍼컴퓨터(CPU)가 아무것도 안 하고 멍 때리고 있으면(Blocking) 돈 낭비잖아?
* **해결책:** "너 파일 읽는 동안 난 딴 거 하고 있을게!"(Non-Blocking)라고 해야 시스템 효율이 올라가.

---
## Core Concepts (블로킹 vs 논블로킹)

### A. Blocking (제어권 뺏김)

함수를 호출하면, 그 작업이 **완전히 끝날 때까지** 제어권(실행 권한)을 넘겨주지 않아.

* **Python 예시:** `requests.get()`, `time.sleep()`, 일반적인 파일 읽기.
* **특징:** 코드가 한 줄 한 줄 순서대로 정직하게 실행됨. 직관적이지만 느림.

### B. Non-Blocking (제어권 즉시 반환)

함수를 호출하면, "알았어, 시작했어!"라고 말하고 **즉시 제어권을 돌려줘.** 결과는 나중에 콜백(Callback)이나 폴링(Polling)으로 확인해.

* **Python 예시:** `asyncio`, `aiohttp`, Node.js의 기본 동작.
* **특징:** 기다리는 시간에 다른 일을 동시에 처리 가능 (동시성).

##  Deep Dive: `sleep`은 왜 I/O 작업일까? (심화) 

"파일을 읽거나 네트워크 통신을 하는 것도 아닌데, 왜 `time.sleep(10)`을 I/O 작업이라고 부를까?"

### 1) CPU의 관점

* **CPU Bound (계산):** 수학 문제를 푸는 것. 펜을 놓지 않고 계속 머리를 써야 해.
* **I/O Bound (대기):** 친구에게 카톡 답장을 기다리는 것. 답장이 올 때까지 나는 멍 때리거나 다른 짓을 할 수 있어.
* **Sleep (타이머):** "10분 뒤에 깨워줘"라고 알람을 맞추는 것. 자는 동안 나는(CPU는) 아무것도 안 해도 돼.

### 2) OS 스케줄러의 동작 원리

프로그램이 `sleep`을 호출하면 운영체제(OS)는 이렇게 행동해.
1.  **CPU 반납:** "어? 이 녀석 10초 동안 잘 거라네? 야, 비켜." (Running ➡ Waiting 상태로 전환)
2.  **문맥 교환 (Context Switch):** "노는 CPU 아까우니까 줄 서 있는 다른 애(다른 스레드/프로세스) 데려와서 일 시켜."
3.  **복귀:** 10초가 지나면 OS가 다시 깨워서 일을 시킴.

### 3) 결론

`sleep`은 데이터를 주고받는 진짜 I/O는 아니지만, **"CPU를 안 쓰고 시간만 보내며 대기한다"** 는 점에서 **I/O 작업과 행동 패턴이 100% 똑같아.**

> **Airflow 꿀팁:** 그래서 Airflow Queue를 나눌 때 `sleep`이 포함된 DAG는 **`io_intensive`** 큐로 보내는 게 정석이야!

## Practical Context (Airflow Sensor)

Airflow에서 **Sensor(감지기)** 를 쓸 때 이 개념이 결정적이야.

1.  **`mode='poke'` (Blocking 방식):**
    * 워커가 "파일 왔어?" 계속 물어보면서 **자리를 차지하고 있음.**
    * 파일이 1시간 뒤에 오면, 워커 1개가 1시간 동안 낭비됨. (비효율적)

2.  **`mode='reschedule'` (Non-Blocking 방식):**
    * "파일 왔어? 안 왔네? 그럼 나 10분 뒤에 다시 올게!" 하고 **자리를 비켜줌.**
    * 그 10분 동안 워커는 다른 DAG를 처리할 수 있음. (효율적)

##  Code Core Points

```python
import time

# --- 1. Blocking 방식 (일반적인 파이썬) ---
def blocking_way():
    print("A: 커피 주문")
    time.sleep(5)  # 5초 동안 아무것도 못 하고 멈춤 (Blocking)
    print("A: 커피 받음")
    
    print("B: 커피 주문") # A가 끝날 때까지 B는 주문도 못 함
    time.sleep(5)
    print("B: 커피 받음")
    # 총 소요 시간: 10초

# --- 2. Non-Blocking 개념 (Asyncio 흉내) ---
# 실제 코드는 복잡하지만, 개념적으로는 이렇습니다.
async def non_blocking_way():
    print("A: 커피 주문 (진동벨 받음)")
    print("B: 커피 주문 (진동벨 받음)") # A 기다리는 동안 B 주문 가능!
    
    await asyncio.sleep(5) # 기다리는 동안 CPU는 다른 일 함
    
    print("A, B: 커피 동시에 받음")
    # 총 소요 시간: 5초 + 알파
```

## 초보자가 자주 착각하는 포인트

1. "Non-Blocking이 무조건 빠른가요?"
	- 아니! 작업 하나만 놓고 보면 속도는 똑같아. (커피 나오는 시간은 똑같이 5분 걸림).
	- 하지만 **"전체 처리량(Throughput)"** 이 늘어나는 거야. 1시간에 100잔 팔 거 200잔 팔 수 있게 되는 거지.

2. "동기(Sync)랑 블로킹(Blocking)은 같은 말인가요?"
	- 거의 비슷하게 쓰이지만 엄밀히는 달라.
	- 하지만 초보 단계에서는 **"Blocking = Sync(순서대로)", "Non-Blocking = Async(동시에)"** 라고 퉁쳐서 이해해도 괜찮아.
