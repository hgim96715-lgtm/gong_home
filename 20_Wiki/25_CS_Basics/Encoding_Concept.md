## 개념 한 줄 요약 

**"컴퓨터(0과 1)와 인간(글자) 사이의 통역 번역기(Dictionary)."**

* **인코딩 (Encoding):** 글자('가') ➡️ 숫자(0101...)로 변환해서 저장. 사람의 데이터를 기계어로 포장하는 것
* **디코딩 (Decoding):** 숫자(0101...) ➡️ 글자('가')로 변환해서 화면에 표시.기계어를 다시 사람의 데이터로 해석
* **깨짐 현상:** 저장할 때 쓴 번역기(A)와 읽을 때 쓴 번역기(B)가 달라서 발생하는 대참사.

**Python:** `str.encode()` -> `bytes` (저장/전송용), `bytes.decode()` -> `str` (출력/분석용)

### 헷갈림 방지 팁 (Mnemonics)

**En**coding: **En**ter (컴퓨터 속으로 들어감)
**De**coding: **De**tect (사람이 다시 식별함)

##  인코딩 역사의 3단계 (이것만 알면 됨) 

### ① ASCII (아스키) - "미국 표준" 

* **특징:** 가장 기초적인 인코딩. **1바이트(7비트)** 만 사용.
* **표현 범위:** 영문 대소문자, 숫자, 특수기호 등 128개.
* **한글:** **전혀 표현 못 함.** (한글을 ASCII로 읽으면 `???`로 나옴)

### ② CP949 (EUC-KR) - "한국 윈도우 표준" 

* **특징:** ASCII로는 한글을 못 쓰니까, 한국 독자적으로 2바이트를 써서 확장한 규격.
* **EUC-KR:** 완성형 한글 표준 (2,350자만 표현 가능, '똠', '펲' 같은 글자 못 씀).
* **CP949:** 마이크로소프트(MS)가 EUC-KR을 더 확장해서 모든 한글을 쓸 수 있게 만든 것. (일명 'MS949')
* **문제점:** **윈도우(엑셀)** 에서만 표준이고, 맥/리눅스/서버 환경과는 호환이 꽝입니다.

### ③ UTF-8 (유니코드) - "전 세계 표준" 

* **특징:** **"전 세계 모든 문자를 하나의 규격으로 담자!"** (유니코드)
* **가변 길이:**
    * 영어/숫자: 1바이트 (ASCII와 호환됨, 용량 절약)
    * **한글: 3바이트** (CP949보다 용량은 더 차지함)
* **현황:** 웹(Web), 리눅스, 스파크, 파이썬3의 **기본값(Default)** 입니다.

## 왜 한글이 깨지나요? (Mojibake) 깩뛟쀍?!

가장 흔한 시나리오: **"윈도우 엑셀(CP949)로 만든 CSV를 파이썬/스파크(UTF-8)로 열었을 때"**

###  발생 원리

1.  **저장 (CP949):** '안'이라는 글자를 **2바이트(`B0 A1`)** 로 저장했습니다.
2.  **읽기 (UTF-8):** 파이썬은 기본적으로 **3바이트**를 읽어서 한 글자를 만들려고 합니다.
3.  **결과:** 파이썬이 `B0 A1` 뒤에 있는 엄한 바이트까지 끌어다가 해석하려다 보니, 엉뚱한 외계어(****, **깩**)가 튀어나오거나 에러가 납니다.

## 해결 방법 (Code) 

무조건 **"파일을 저장할 때 썼던 인코딩 방식"** 을 옵션으로 알려줘야 합니다.

### ① Pandas 해결법

```python
# 기본적으로 utf-8로 읽으려다가 에러남 (UnicodeDecodeError)
# df = pd.read_csv("korean_data.csv") 

# [해결] 인코딩을 cp949(또는 euc-kr)로 지정
df = pd.read_csv("korean_data.csv", encoding="cp949")
```

### ② Spark 해결법

```python
# 스파크도 기본이 utf-8임
df = spark.read \
    .option("encoding", "cp949") \
    .csv("korean_data.csv")
```
