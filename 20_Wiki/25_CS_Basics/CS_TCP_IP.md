데이터가 인터넷을 통해 어떻게 전달되는지

# ① TCP/IP 4계층

```
4층  응용 계층    (Application)  HTTP / FTP / DNS / Kafka
3층  전송 계층    (Transport)    TCP / UDP   ← 포트 번호
2층  인터넷 계층  (Internet)     IP 주소     ← 라우팅
1층  네트워크 계층(Network)      MAC 주소    ← 물리적 전달

OSI 7계층을 실용적으로 4개로 합친 모델
HTTP 는 TCP 위에서 동작
Kafka 도 TCP 기반
```

# ② 패킷 (Packet)

```
데이터를 잘게 쪼갠 단위
인터넷은 큰 데이터를 한번에 보내지 않음
→ 작은 조각(패킷)으로 나눠서 따로따로 전송
→ 수신측에서 다시 조립
```

## 왜 쪼개서 보내나?

```
1GB 파일을 통째로 보내면:
  전송 중 하나라도 오류 → 전부 다시 보내야 함
  네트워크 독점 → 다른 사람 못 씀

패킷으로 나눠 보내면:
  오류난 패킷만 재전송
  여러 경로로 동시에 전송 가능 (더 빠름)
  다른 데이터와 네트워크 공유 가능
```

## 패킷 구조

```
┌─────────────────────────────────────┐
│  헤더 (Header)                       │
│  출발지 IP / 목적지 IP               │
│  출발지 포트 / 목적지 포트           │
│  패킷 순서 번호 (Sequence Number)    │
│  프로토콜 종류 (TCP/UDP)             │
├─────────────────────────────────────┤
│  페이로드 (Payload)                  │
│  실제 데이터 내용                    │
│  (HTTP 요청 / 응답 / 파일 데이터 등) │
├─────────────────────────────────────┤
│  트레일러 (Trailer)                  │
│  오류 검출 코드 (Checksum)           │
└─────────────────────────────────────┘
```

## 패킷이 목적지까지 가는 과정

```
내 컴퓨터에서 API 서버로 데이터 전송 시:

1. 데이터 분할
   원본 데이터 → 패킷 1 / 패킷 2 / 패킷 3 ...
   각 패킷에 순서 번호 부여 (Sequence Number)

2. 전송
   패킷마다 다른 경로로 이동할 수 있음
   패킷 1 → 서울 → 부산 → 목적지
   패킷 2 → 서울 → 대전 → 목적지  (다른 경로!)
   패킷 3 → 서울 → 부산 → 목적지

3. 재조립
   수신측이 순서 번호 보고 올바른 순서로 조립
   누락된 패킷 있으면 → 재전송 요청 (TCP)
```

## TCP 에서 패킷 관리

```
Sequence Number:  이 패킷이 몇 번째 바이트인지
Acknowledgment:   "N번까지 잘 받았어, N+1번 줘"

패킷 유실 시:
  수신측: "패킷 3 안 왔어" → 재전송 요청
  송신측: 패킷 3 다시 전송
  → 신뢰성 보장

UDP:
  순서 번호 / 재전송 없음
  그냥 던지고 끝 → 빠르지만 유실 허용
```

## 실생활 비유

```
편지를 10장 보내는 경우:

❌ 일반 우편 (UDP 방식):
  10장 각자 다른 우체통에 투입
  순서 뒤바뀔 수 있음
  분실되어도 모름

✅ 등기 우편 (TCP 방식):
  각 편지에 번호 부여 (1/10, 2/10...)
  받는 사람이 확인증 보내줌
  분실 시 다시 보냄
  모두 받으면 순서대로 정렬
```

# ④ TCP — 신뢰성 보장

```
Transmission Control Protocol

특징:
  연결 지향 (Connection-oriented)  → 통신 전 연결 수립
  신뢰성                           → 데이터 손실 시 재전송
  순서 보장                         → 패킷 순서 맞춰서 재조립
  흐름 제어                         → 수신측 처리 속도에 맞춤

사용하는 곳:
  HTTP / HTTPS
  Kafka (메시지 유실 안 됨)
  PostgreSQL
  SSH
  → 데이터 정확성이 중요한 모든 곳
```

## 3-Way Handshake — TCP 연결 수립

```
클라이언트와 서버가 통신 시작 전에 연결을 확인하는 과정
3번의 신호로 연결 수립

1. SYN      클라이언트 → 서버  "나 연결하고 싶어"
2. SYN-ACK  서버 → 클라이언트  "OK 나도 준비됐어"
3. ACK      클라이언트 → 서버  "알겠어 시작하자"

→ 연결 수립 완료 → 데이터 전송 시작
```

```
클라이언트          서버
    |                |
    |──── SYN ──────>|
    |<── SYN-ACK ────|
    |──── ACK ──────>|
    |                |
    |  데이터 전송    |
```

```
4-Way Handshake — 연결 종료:
  FIN → ACK → FIN → ACK (4번으로 연결 종료)
```

# ⑤ UDP — 속도 우선

```
User Datagram Protocol

특징:
  비연결 (Connectionless)  → 연결 수립 없이 바로 전송
  신뢰성 없음              → 패킷 유실되어도 재전송 안 함
  순서 보장 없음
  빠름                     → 오버헤드 없음

사용하는 곳:
  DNS 조회     빠른 응답이 중요
  스트리밍     유튜브 / 화상통화 (약간 끊겨도 OK)
  게임         실시간 반응이 중요
  로그 수집    일부 유실 허용
```

>**DNS(Domain Name System)란?**  사람이 읽기 쉬운 도메인 이름을 컴퓨터가 이해하는 IP 주소로 변환(또는 그 반대)해주는 분산 네이밍 시스템이다.

# ⑥ TCP vs UDP

|항목|TCP|UDP|
|---|---|---|
|연결 방식|연결 지향 (3-Way Handshake)|비연결|
|신뢰성|✅ 재전송 보장|❌ 유실 허용|
|순서 보장|✅|❌|
|속도|상대적으로 느림|빠름|
|사용|HTTP / Kafka / DB|DNS / 스트리밍 / 게임|

# ⑦ DE 관점에서의 TCP/IP

```
Kafka:
  TCP 기반 → 메시지 유실 없이 전달 보장
  브로커 연결 시 3-Way Handshake 발생
  KAFKA_ADVERTISED_LISTENERS = kafka:9092
  → 컨테이너 내부 DNS + 포트로 TCP 연결

PostgreSQL:
  TCP 포트 5432 (프로젝트에서 5433/5434 로 포워딩)
  클라이언트 → TCP 연결 → 쿼리 전송

Docker 포트 포워딩:
  호스트:5434 → 컨테이너:5432
  외부에서 TCP 5434 로 접속하면
  Docker 가 컨테이너 5432 로 전달

방화벽/보안그룹:
  포트 단위로 TCP 접근 제어
  9092 열어야 Kafka 외부 접속 가능
```

# 소켓 (Socket)

```
두 프로그램이 네트워크로 통신할 때 사용하는 연결 창구
IP 주소 + 포트 번호 로 식별되는 통신 끝점(Endpoint)

비유:
  전화기 = 소켓
  전화번호 = IP:Port
  통화 = 소켓 연결 (TCP 연결)
```

## 소켓 = IP + Port

```
소켓 주소:
  192.168.0.1:5432

  클라이언트 소켓  192.168.0.10:54321  ← 클라이언트 IP + 임시 포트
  서버 소켓        192.168.0.1:5432    ← 서버 IP + 고정 포트

연결 = 클라이언트 소켓 ↔ 서버 소켓 쌍
  (192.168.0.10:54321, 192.168.0.1:5432)
  이 쌍이 하나의 TCP 연결
```

## 소켓 통신 흐름

```
서버:
  1. socket()   소켓 생성
  2. bind()     IP:Port 에 소켓 연결
  3. listen()   연결 대기 (LISTEN 상태)
  4. accept()   클라이언트 연결 수락

클라이언트:
  1. socket()   소켓 생성
  2. connect()  서버 IP:Port 로 연결 요청
               (3-Way Handshake 발생)
  3. 데이터 송수신

둘 다:
  send() / recv()  데이터 주고받기
  close()          소켓 닫기 (4-Way Handshake)
```

## lsof 에서 보는 소켓


```bash
$ lsof -i -P -n | grep LISTEN

com.docke  46521  gong  219u  IPv6  0xcf5bd0a91cfeae84  0t0  TCP  *:4040 (LISTEN)
#                               ↑
#                          소켓 식별자 (파일 디스크립터)
#                          리눅스에서 소켓도 파일로 관리

# *:4040 = 모든 IP 의 4040 포트에 소켓을 열어두고 대기
# LISTEN = 연결 대기 중인 서버 소켓
```

```
리눅스에서 "모든 것은 파일":
  일반 파일 / 디렉토리 / 네트워크 소켓 / 파이프 모두 파일
  → lsof (List Open Files) 로 소켓도 조회 가능
  → 파일 디스크립터 번호로 관리 (219u)
```

## Kafka / PostgreSQL 소켓

```
Kafka Producer → Broker:
  Producer 가 kafka:9092 로 TCP 소켓 연결
  3-Way Handshake → 연결 수립
  메시지 전송 (send)
  연결 유지 (keep-alive)

PostgreSQL 연결:
  DataGrip / Spark / Airflow → localhost:5434
  TCP 소켓 연결
  SQL 쿼리 전송 (send)
  결과 수신 (recv)

KAFKA_ADVERTISED_LISTENERS=kafka:9092:
  다른 컨테이너가 "kafka:9092" 로 소켓 연결할 수 있도록
  Kafka 가 이 주소로 소켓을 열어두는 것
```
