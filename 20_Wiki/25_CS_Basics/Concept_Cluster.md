## 개념 요약 

**"혼자서는 감당 못 할 거대한 데이터를 처리하기 위해, 여러 대의 컴퓨터를 네트워크로 묶어서 하나의 팀(Team)처럼 사용하는 시스템"** 입니다.

---
## 왜 필요한가?

 **문제 (Problem):** 
* 컴퓨터 한 대(Single Node)의 성능(CPU, RAM)에는 물리적인 한계가 있습니다. 
* 100TB짜리 데이터를 처리하려고 하면 메모리가 터지거나(OOM), 계산하는 데 10년이 걸립니다.

 **해결 (Solution):** 
 - 컴퓨터 100대를 연결해서 일을 1/100로 나누어 처리하면, 아무리 큰 데이터도 순식간에 처리할 수 있습니다.

---
## Practical Context (The Analogy)

> **"이사(Moving)를 간다고 상상해보세요."**

* **싱글 컴퓨터 (Single Node):**
    * 나 혼자서 냉장고, 침대, 책상을 다 옮겨야 합니다.
    * **결과:** 너무 무거워서 들지도 못하거나(Error), 허리가 나갑니다(System Crash).

* **클러스터 (Cluster):**
    * 친구 10명을 불러서 팀을 꾸립니다.
    * **반장 (Master):** "철수는 부엌 짐 싸고, 영희는 트럭 운전해." (지시만 함)
    * **친구들 (Worker):** 아무 생각 없이 시키는 짐만 나릅니다. (실행만 함)
    * **결과:** 엄청난 양의 짐도 금방 옮깁니다.

---
## Core Components (구성원)

분산 처리 시스템(Flink, Spark, Hadoop)은 모두 이 **Master-Worker 구조**를 따릅니다.

| 역할 (Role) | 비유 (이사 팀) | Flink/Spark 용어 | 하는 일 (Responsibility) |
| :--- | :--- | :--- | :--- |
| **Master Node** | **작업 반장** | JobManager / Driver | **"지시"**: 자원 관리, 스케줄링, 장애 감지. 직접 무거운 짐을 나르지는 않음. |
| **Worker Node** | **일꾼** | TaskManager / Executor | **"노동"**: 실제 데이터 연산 수행. 시키는 일만 묵묵히 처리. |
| **Node** | **사람 한 명** | Container / Server | 클러스터를 구성하는 컴퓨터(서버) 한 대 단위. |

## Detailed Analysis (현업 용어 해석)

개발자들이 회의 시간에 흔히 쓰는 말을 번역해드립니다.

1.  **"클러스터에 잡(Job)을 제출한다"**
    * = "작업 반장(Master)에게 작업지시서를 전달한다."
2.  **"클러스터를 띄운다 (Spin up)"**
    * = "작업 반장 1명과 일꾼 N명을 컴퓨터에 켜서, 일할 수 있는 팀을 조직한다."
3.  **"노드가 죽었다 (Node Failure)"**
    * = "일꾼 한 명이 배탈 나서 조퇴했다."
    * 이때 Master는 즉시 다른 일꾼에게 그 일을 다시 시킵니다. 이것이 **내결함성(Fault Tolerance)** 입니다.

## Common Beginner Misconceptions

1.  **"클러스터는 그냥 슈퍼컴퓨터 한 대 아닌가요?"**
    * 아닙니다. **'평범한 컴퓨터 여러 대'**를 랜선(네트워크)으로 연결한 것입니다. 
    * 그래서 컴퓨터 간의 통신 속도(네트워크 대역폭)가 느리면 전체가 느려질 수 있습니다.

2.  **"마스터 노드 성능이 제일 좋아야 하나요?"**
    * 보통은 아닙니다. 실제 무거운 계산은 워커들이 다 하므로, **워커 노드의 스펙(RAM, CPU 개수)** 이 훨씬 중요합니다. 
    * 마스터는 관리만 잘하면 됩니다.
