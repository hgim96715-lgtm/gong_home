## 개념 한 줄 요약

**"하둡(Hadoop)의 느린 디스크 I/O를 해결하기 위해, 데이터를 메모리(RAM)에 올려서 100배 빠르게 처리하는 분산 컴퓨팅 엔진."** 

---
##  왜 필요한가? (The Evolution)

### 🐢 과거: 하둡 맵리듀스 (Hadoop MapReduce)

* **방식:** 데이터를 처리할 때마다 결과를 **하드디스크(Disk)** 에 썼다가 다시 읽습니다.
* **문제:** 머신러닝처럼 데이터를 100번 반복해서 읽어야 하는 작업에서는 디스크 속도 때문에 너무 느렸습니다.

### 🐇 현재: 아파치 스파크 (Apache Spark)

* **방식:** **인메모리(In-Memory)** 컴퓨팅을 도입했습니다. 데이터를 **RAM**에 올려두고 거기서 계속 작업합니다. 
* **결과:** 하둡 맵리듀스보다 **최대 100배** 빠른 속도를 자랑합니다. 
* **특징:** 배치(Batch), 스트리밍(Real-time), 머신러닝(ML), 그래프(Graph) 등 모든 작업을 하나의 엔진에서 처리하는 **통합 플랫폼(Unified Platform)** 입니다. 

---
## 핵심 아키텍처 (Architecture)

스파크는 **"시키는 놈(Master)"** 과 **"일하는 놈(Worker)"** 으로 철저히 나뉩니다. 

### ① Driver Program (The Brain )

* **역할:** 사용자의 코드를 해석하고, **작업 계획(Task)**을 짜서 일꾼들에게 명령을 내리는 총사령관입니다. 
* **SparkContext:** 드라이버 안에 있는 **접속 창구**로, 클러스터와 통신하는 핵심 객체입니다. 

### ② Cluster Manager (The HR Manager )

* **역할:** 드라이버가 "나 CPU 100개랑 메모리 200GB 필요해!"라고 요청하면, 자원을 할당해 주는 중개자입니다.
* **종류:** Hadoop YARN, Kubernetes, Mesos, Standalone 등이 있습니다. 

### ③ Executor (The Muscle )

* **역할:** 실제 데이터를 씹고 뜯고 맛보고 즐기는 **일꾼 프로세스**입니다. 
* 워커 노드(Worker Node) 안에서 실행됩니다.
* **Task:** 드라이버가 내려보낸 쪼개진 작업 단위.
* **Cache:** 처리 중인 데이터를 메모리에 저장해두는 공간. (이게 스파크 속도의 비결!)

---

## 스파크의 5가지 무기 (Components)

스파크는 단순히 계산만 하는 게 아니라, 종합 선물 세트입니다.

1.  **Spark Core:** 메모리 관리, 작업 스케줄링 등 엔진의 심장.
2.  **Spark SQL & DataFrame:** 데이터를 SQL처럼 다루게 해줌. (데이터 엔지니어가 90% 시간을 쓰는 곳) 
3.  **Spark Streaming:** 실시간 데이터 처리 (Kafka 연동 등). 
4.  **MLlib:** 머신러닝 라이브러리.
5.  **GraphX:** 소셜 네트워크 같은 그래프 데이터 분석.

## 왜 하필 Python(PySpark)인가?

* **쉬운 학습:** 자바나 스칼라보다 문법이 훨씬 직관적입니다.
* **강력한 생태계:** Pandas, NumPy, Scikit-learn 같은 데이터 분석 라이브러리와 찰떡궁합입니다. 
* **업계 표준:** 실리콘밸리를 포함한 현업에서 가장 많이 찾는 스킬셋입니다.

## 초보자가 자주 하는 실수 (Misconceptions)

### ① "스파크는 하둡을 완전히 대체하나요?"

* **반은 맞고 반은 틀립니다.**
* **연산(Computing):** 맵리듀스(MapReduce) 대신 스파크를 씁니다. (대체 O)
* **저장(Storage):** 스파크는 저장소가 없습니다. 여전히 데이터는 하둡 파일시스템(HDFS)이나 S3에 저장합니다. (대체 X)

### ② "RAM이 부족하면 스파크 못 쓰나요?"

* 아닙니다. 
* 메모리가 꽉 차면 자동으로 디스크(Disk)를 잠시 빌려 씁니다(Spill). 
* 단지 속도가 조금 느려질 뿐, 작업이 불가능하진 않습니다.
