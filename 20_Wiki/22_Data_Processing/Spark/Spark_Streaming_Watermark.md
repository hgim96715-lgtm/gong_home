## 늦은 데이터 처리와 메모리 관리 

**Watermark**는 이벤트 시간(Event Time) 기반 처리에서 **늦게 도착한 데이터(Late Data)** 를 처리하는 메커니즘이자, **State Store(메모리)** 를 청소하는 기준입니다.

* **문제 상황:** 네트워크 지연 등으로 데이터가 순서대로 도착하지 않음 (Out of Order).
* **해결책:** "특정 시간(Threshold)까지만 기다리고, 그보다 더 늦으면 버린다"는 선을 긋습니다.
* **효과:** 워터마크를 지나간 오래된 윈도우의 상태(State)를 삭제하여 메모리를 확보합니다.

---
##  시간의 두 가지 기준 (Time Concepts) 

스파크 스트리밍은 시간을 두 가지로 구분합니다.

1.  **Event Time (이벤트 시간):** 데이터가 **실제로 발생한** 시간 (데이터 내부에 포함된 타임스탬프).
2.  **Processing Time (처리 시간):** 스파크가 해당 데이터를 **처리하는(읽어들인)** 시간.
    * *워터마크는 반드시 **Event Time**을 기준으로 동작합니다.*

---
## 문법 (Syntax)

`groupBy`나 `window`를 하기 **전에** `.withWatermark()`를 호출해야 합니다.

- 문법: `{python}.withWatermark(이벤트_시간_컬럼, 허용_지연_시간)`

```python
# 문법: .withWatermark(이벤트_시간_컬럼, 허용_지연_시간)
windowedCounts = words \
    .withWatermark("timestamp", "10 minutes") \
    .groupBy(
        window(col("timestamp"), "10 minutes", "5 minutes")
    ).count()
```

- **의미:** "최대 이벤트 시간보다 **10분** 전까지의 데이터는 받아주겠다. 하지만 10분보다 더 늦으면(Too Late) 버리고, 해당 윈도우 메모리도 지운다."

## Output Mode에 따른 동작 방식 (중요!) ⚠️

워터마크는 **Output Mode**에 따라 동작이 완전히 달라집니다

### ① Update Mode (권장)

- **동작:** 워터마크를 사용하면 **오래된 윈도우 상태를 제거**하여 메모리를 효율적으로 관리합니다.
- **특징:**
    - 집계 결과가 갱신될 때마다 출력됩니다.
    - **제약:** 파일 시스템(File Sink)에는 쓸 수 없습니다. Cassandra나 RDBMS처럼 **Upsert(수정)** 가 가능한 저장소여야 합니다.

### ② Append Mode (안전 제일)

- **동작:** 워터마크 경계를 완전히 지난(확정된) 윈도우의 결과만 **단 한 번** 출력합니다.
- **특징:**
    - **단점:** 결과가 나올 때까지 지연(Delay)이 발생합니다. (데이터는 왔지만 워터마크가 지날 때까지 출력을 안 함) .
    - **장점:** 결과가 수정되지 않으므로 파일 시스템(Parquet, CSV)에도 안전하게 저장할 수 있습니다.
    - 워터마크보다 늦게 온 데이터는 무시됩니다

### ③ Complete Mode (주의)

- **동작:** 워터마크를 설정해도 **무시(Ignored)** 됩니다.
- **문제:** 과거의 모든 윈도우 상태를 영원히 유지하므로, **메모리를 비우지 못합니다(Cannot clear up)**.
- 데이터가 적을 때는 괜찮지만, 운영 환경에서는 OOM 위험이 큽니다.

## 요약 비교 (Cheat Sheet)

|**모드**|**워터마크 적용**|**State 청소**|**지연 데이터 처리**|**용도**|
|---|---|---|---|---|
|**Update**|✅ 적용됨|✅ 삭제함|갱신해서 출력|실시간 대시보드, DB 저장|
|**Append**|✅ 적용됨|✅ 삭제함|**무시함(Drop)**|파일 저장, 확정된 결과만 필요할 때|
|**Complete**|❌ **무시됨**|❌ **유지함**|계속 누적|개발/테스트, 데이터가 아주 적을 때|

> 메모리 관리를 위해 **반드시** `Update`나 `Append` 모드와 함께 워터마크를 사용해야 합니다.

##  [실습] 워터마크 동작 시나리오 (Late Data Handling) 

실제 데이터를 통해 어떤 데이터가 살고, 어떤 데이터가 죽는지(Drop) 확인해 봅시다.

### ① 설정 (Setup)

* **윈도우:** 5분 단위 Tumbling (`windowDuration = "5 minutes"`)
* **워터마크:** 10분 지연 허용 (`.withWatermark("create_date", "10 minutes")`)
* **현재 상태:** 가장 최근에 들어온 데이터(Max Event Time)가 **`12:02:00`** 이라고 가정합니다.
    * 👉 **Watermark 기준선** = `12:02:00` - `10분` = **`11:52:00`**

### ② 입력 데이터 (Input Data)

```json
{"create_date":"2026-02-04 12:02:00","amount": 20}  -- (1) 현재 시간 (기준점 갱신)
{"create_date":"2026-02-04 11:53:00","amount": 30}  -- (2) 9분 지각 (아슬아슬 세이프)
{"create_date":"2026-02-04 11:40:00","amount": 100} -- (3) 22분 지각 (Too Late!)
```

### ③ 처리 로직 (Processing Logic)

1. **`12:02:00` (기준):** 정상 데이터. 워터마크 기준을 `11:52:00`으로 만듭니다.

2. **`11:53:00` (Late):**
    - 기준선(`11:52:00`)보다 **미래**입니다 (`11:53 > 11:52`).
    - **결과:** **생존 (Accepted)**. `11:50~11:55` 윈도우를 찾아가서 업데이트합니다.

3. **`11:40:00` (Too Late):**
    - 기준선(`11:52:00`)보다 **과거**입니다 (`11:40 < 11:52`).
    - **결과:** **삭제 (Dropped)**. 이미 메모리에서 청소된 윈도우라 갈 곳이 없습니다.

### ④ 결과 확인 (Output)

**생존한 데이터(`11:53:00`)의 결과:** 
`11:50` 윈도우가 정상적으로 업데이트되었습니다.

```text
+------------------------------------------+------------+
|window                                    |total_amount|
+------------------------------------------+------------+
|{2026-02-04 11:50:00, 2026-02-04 11:55:00}|30          |
+------------------------------------------+------------+
```

**버려진 데이터(`11:40:00`)의 결과:** 
너무 늦어서(`10분` 초과) 무시되었으므로 아무것도 출력되지 않습니다.

```text
+------+------------+
|window|total_amount|
+------+------------+
+------+------------+
```
