##  Fault Tolerance란? (목표)

스트리밍 애플리케이션은 24시간 365일 돌아가야 합니다. 하지만 서버 고장, 유지보수, 코드 배포 등으로 인해 언제든 멈출 수 있습니다

스파크 스트리밍의 궁극적인 목표는 **"End-to-End Exactly-Once Semantics"** (처음부터 끝까지 딱 한 번만 처리함)을 보장하는 것입니다

* **Do not miss:** 입력 데이터를 하나도 빠뜨리지 않음
* **Do not duplicate:** 결과 데이터가 중복으로 생성되지 않음

---
##  Exactly-once를 위한 3가지 조건 

스파크 혼자서 모든 걸 할 수는 없습니다. 소스(Source)와 싱크(Sink)가 함께 도와줘야 완벽한 무결성을 보장할 수 있습니다

### ① Replayable Source (다시 감을 수 있는 소스)

* **조건:** 스파크가 "나 아까 거기서 멈췄어, 다시 줘!"라고 했을 때, 지난 데이터를 다시 줄 수 있어야 합니다
* **가능:** Kafka, File Source (파일은 그대로 있으니까)
* **불가능:** Socket (지나가면 끝)

### ② Deterministic Computation (일관된 계산)

* **조건:** 같은 입력(`x`)을 넣으면 항상 같은 결과(`y`)가 나와야 합니다
* **의미:** 재시작해서 다시 계산하더라도 값이 달라지면 안 됩니다.

### ③ Idempotent Sink (멱등성 싱크)

* **조건:** 같은 데이터를 여러 번 써도 결과가 변하지 않아야 합니다
* **예시:**
    * **DB Update:** `ID=3`인 데이터를 `Value=100`으로 덮어쓰기 (여러 번 해도 결과는 100).
    * **File Overwrite:** 같은 이름의 파일을 다시 덮어쓰기.

## 핵심 기술: 읽기 장부와 쓰기 장부 

스파크는 장애 발생 시 복구를 위해 두 종류의 장부를 끊임없이 기록합니다.

### ⓵ 읽기 장부: Checkpoint (`chk/`)

"내가 소스(Kafka)에서 **어디까지 읽었나**"를 기록합니다.

1. **Read Position (Offset):** "Kafka의 A토픽 100번 오프셋까지 읽었음."
2. **State Information:** "현재까지 집계된 단어 카운트 맵은 이것임."

- **복구:** 다시 켜지면 체크포인트를 뒤져서 "101번부터 다시 가져와!" 하고 이어감.

### ⓶ 쓰기 장부: Metadata (`_spark_metadata/`)

"내가 목적지(File/S3)에 **어떤 파일을 최종 저장했나**"를 기록합니다.

- **내용:** 확인하신 `v1 {"path": "...", "action": "add"}` 로그입니다.
- **역할:** 파일은 생성됐지만 로그에 기록되지 않았다면 "가짜"로 판단해 버리고 중복 저장을 막습니다.

#### ① `v1` 파일 내부 분석

 로그 파일(`3`, `4` 등의 번호로 된 파일) 안에는 이런 내용이 들어있습니다.

```json
{ "path": "file:///workspace/data/transformed/part-00000...json", "size": 63, "action": "add" }
```

- **`path`**: 실제로 데이터가 저장된 물리적 위치입니다.
- **`action`: "add"**: 이 파일이 정상적으로 쓰기 완료되어 데이터 세트에 **추가**되었음을 확정(Commit)하는 표시입니다.

**⚠️ 주의사항 및 실무 팁**

- **폴더 삭제 금지**: `_spark_metadata` 폴더를 지우면 스파크는 어떤 파일이 진짜인지 알 수 없게 되어 전체 파이프라인이 꼬입니다.
- **체크포인트와 세트**: `chk/` 폴더와 `_spark_metadata`는 한 세트입니다. 만약 아예 처음부터 다시 시작하고 싶다면 두 폴더를 모두 지워야 합니다.

##  ⚠️ 협업 및 Git 커밋 주의사항 (중요!)

스파크 스트리밍 작업 중 생기는 `checkpoint`와 `_spark_metadata`는 **절대 Git에 올리지 않습니다.**

1. **이유:**
    - 로컬 경로(`file:///workspace/...`)가 포함되어 있어 다른 환경에서 동작하지 않음.
    - 실시간으로 수많은 파일이 생성되어 커밋 로그가 지저분해짐.
2. **해결책:** `.gitignore` 파일에 해당 폴더들을 등록하여 제외함.
3. **규칙:** 팀원들과 공유할 때는 **코드(`kafka_json.py`)** 만 공유하고, 각자의 환경에서 실행할 때 새로운 체크포인트 폴더가 생기도록 관리함.

## "Can I change anytime?" (코드 수정의 제약) 

PDF 마지막 장의 물음표(**?**)가 의미하는 아주 중요한 내용입니다

**"체크포인트가 있는 상태에서 코드를 내 마음대로 바꿔도 될까?"**
👉 **정답: 함부로 바꾸면 에러가 납니다!**

체크포인트에는 **"과거의 코드 논리로 만들어진 상태(State)"** 가 저장되어 있습니다. 그런데 코드를 수정해서 배포하면, 스파크가 "어? 저장된 상태랑 코드랑 안 맞는데?" 하고 복구를 실패할 수 있습니다.

* **가능한 변경:** 필터 조건 변경, 새로운 컬럼 추가(일부) 등.
* **위험한 변경:** 집계(Aggregation) 키 변경, 윈도우 사이즈 변경 등.
    * *해결책:* 구조가 크게 바뀌면 체크포인트를 지우고(Reset) 처음부터 다시 시작해야 할 수도 있습니다.
