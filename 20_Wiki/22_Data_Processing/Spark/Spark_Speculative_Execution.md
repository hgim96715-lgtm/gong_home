## 개념 한 줄 요약 

**"거북이(느린 태스크)가 보이면 토끼(복제 태스크)를 하나 더 출발시켜서, 먼저 도착한 놈만 인정하는 전략."**

* **Straggler(낙오자)란?** 남들 다 끝났는데 혼자 하드웨어 이슈나 네트워크 문제로 세월아 네월아 하고 있는 **엄청 느린 태스크**
* **해결책:** 스파크가 이를 감지하면 다른 노드에 **똑같은 태스크(Duplicate)** 를 하나 더 띄웁니다. 둘 중 **먼저 끝난 결과**를 채택하고, 늦은 녀석은 강제 종료(Kill) 시킵니다

---
##  작동 원리 (Diagram Concept) 

일반적인 작업 흐름과 어떻게 다른지 비교해봅시다.

1.  **정상 상황:** Task 1, 2, 3이 동시에 시작해서 비슷하게 끝납니다.
2.  **문제 상황:** Task 2가 장비 고장으로 **매우 느려짐(Straggler)**. 전체 Job이 Task 2 때문에 끝나질 않음.
3.  **Speculative Execution 발동:**
    * 스파크: "어? Task 2가 너무 늦네? 다른 노드에서 **'Speculative Task 2'** 출발시켜!"
    * 결과: 나중에 출발한 'Speculative Task 2'가 먼저 끝남.
    * 처리: 원조 Task 2는 죽이고(Kill), 복제본의 결과를 사용

##  언제 써야 하나요? (Use Cases)

무조건 켜는 게 아닙니다. 아래 상황에서 효과적입니다.

1.  **이기종 클러스터 (Heterogeneous Clusters):** 어떤 노드는 고성능이고, 어떤 노드는 똥컴(구형 장비)일 때
2.  **불안정한 네트워크:** 특정 노드의 네트워크가 일시적으로 느려질 때
3.  **데이터 스큐 (Data Skew):** 데이터가 불균형해서 특정 태스크만 오래 걸릴 때 (단, 심한 스큐는 이걸로 해결 안 됨)

## 핵심 설정 (Configuration) 

기본적으로 꺼져(`false`) 있는 경우가 많으므로, 필요하면 켜야 합니다.

```python
# 1. 투기적 실행 켜기 (Default: false)
spark.conf.set("spark.speculation", "true") 

# 2. 언제 끼어들까? (Quantile)
# "전체 태스크의 75%가 끝난 시점부터 감시를 시작해라"
# Default: 0.75
spark.conf.set("spark.speculation.quantile", "0.75")

# 3. 누구를 낙오자로 볼까? (Multiplier)
# "남들보다 1.5배(50%) 이상 느리면 낙오자로 간주해라"
# Default: 1.5
spark.conf.set("spark.speculation.multiplier", "1.5") 
```

## 주의사항 및 단점 (Disadvantages)

**"공짜 점심은 없다."** 이 기능을 켜면 자원을 더 씁니다.

1. **리소스 낭비:** 복제본을 실행해야 하므로 CPU와 메모리를 **이중으로 사용**합니다.
2. **모니터링 오버헤드:** 누가 늦는지 계속 감시해야 하므로 마스터 노드에 부하를 줍니다.
3. **데이터 스큐의 한계:** 만약 데이터 자체가 너무 많아서 느린 거라면(Data Skew), 복제본을 띄워봤자 **걔도 똑같이 느립니다.** 이럴 땐 파티셔닝(Repartitioning)을 다시 하는 게 정답입니다.

>"Job이 99%에서 멈춰서 안 끝나는 경우 있지? 범인은 보통 **'맛이 간 노드'** 하나 때문이야. 
>이때 `spark.speculation=true`를 켜주면 기가 막히게 그 노드를 버리고 다른 곳에서 빨리 끝내준다. 
>하지만 **평소에 리소스가 부족하다면 켜지 마.** 멀쩡한 애들도 복제하느라 클러스터 뻗을 수 있다!"
