## 개념 한 줄 요약 ️

**"모든 Executor가 함께 쓸 수 있는 '쓰기 전용(Write-Only)' 전역 변수."**

* 분산 처리를 하다 보면 **"전체 데이터 중 에러가 몇 건 났지?"**, **"지금까지 몇 줄 처리했지?"** 같은 **전역 통계**가 필요할 때가 있습니다.
* 이때 `count()` 같은 무거운 액션을 쓰지 않고, **Accumulator**를 심어두면 작업이 끝난 뒤 가볍게 통계를 확인할 수 있습니다.

---
## 작동 원리 (Driver vs Executor) 

Accumulator는 독특한 **비대칭 구조**를 가집니다.

1.  **Driver (읽기 가능):** 변수를 생성하고, 작업이 끝난 후 최종 결과(`value`)를 읽을 수 있습니다.
2.  **Executor (쓰기만 가능):** 작업을 처리하면서 값을 더할(`add`) 수는 있지만, 내부의 값을 **읽을 수는 없습니다.**

> **이유:** 각 Executor는 서로 다른 노드에서 돌기 때문에, 실시간으로 다른 노드의 값을 읽는 것은 성능상 불가능하기 때문입니다.

---
## 2.5. 주요 메서드 (API) 

Accumulator를 다룰 때 딱 3가지만 기억하면 됩니다.

| 메서드 | 누가 쓰나요? | 설명 |
| :--- | :--- | :--- |
| **`sc.accumulator(초기값)`** | **Driver** | Accumulator를 생성합니다. (보통 0으로 시작) |
| **`acc.add(값)`** | **Executor** | 값을 더합니다. (쓰기 전용) <br> *주의: 여기서 값을 읽을 순 없습니다!* |
| **`acc.value`** | **Driver** | 최종 합계(결과)를 확인합니다. (읽기 전용) |

> **💡 참고:** `add()` 안에는 1뿐만 아니라 `add(10)`, `add(5)` 처럼 원하는 숫자를 넣을 수 있습니다.

---
## 사용 예시 (Python: Classic Way) 

**상황:** 데이터 처리 중 발생하는 특정 상황(예: 홀수)을 카운트하고 싶을 때

```python
# 1. Driver에서 Accumulator 생성 (초기값 0)
acc = sc.accumulator(0)

# 2. 데이터 준비
rdd = sc.parallelize([1, 2, 3, 4, 5])

# 3. 함수 정의: 데이터를 처리하면서 Accumulator를 1씩 증가
def count_elements(x):
    global acc
    if x % 2 != 0:
        acc.add(1)  # Executor들이 실행함

# 4. 액션 실행 (반드시 Action이어야 함!)
rdd.foreach(count_elements)

# 5. Driver에서 결과 확인
print(f"Odd Numbers Count: {acc.value}")
# 결과: 3
```

## 비즈니스 로직엔 `filter().count()` 쓰세요 

**"Accumulator는 중요한 돈 계산에 쓰지 마세요."**

PySpark 환경이나 스파크의 재실행(Retry) 특성상 **값이 중복되거나 사라질 위험**이 있습니다. 
정확한 값이 생명인 비즈니스 로직에서는 안전한 정석 방법을 쓰세요.

```python
# [권장 코드] Accumulator 없이 똑같은 일 하기 (안전함)
# 1. 원하는 조건으로 필터링
errors = rdd.filter(lambda x: x % 2 != 0)

# 2. 개수 세기 (Action)
error_count = errors.count()
```

| **구분**     | **Accumulator**                  | **filter().count()**               |
| ---------- | -------------------------------- | ---------------------------------- |
| **장점**     | **빠름:** 데이터 한 번 훑을 때 덤으로 셈.      | **안전함:** 에러 확률 0%, 코드가 직관적임.       |
| **단점**     | **위험함:** 중복 집계 가능성 있음.           | **느림:** 데이터를 한 번 더 읽어야 함 (I/O 발생). |
| **용도**     | 단순 로그, 디버깅, 모니터링                 | 결제 금액 합산, 정산, 핵심 지표                |
| **코드 스타일** | `foreach` 안에서 `acc.add(1)` (절차적) | `filter(...)` -> `count()` (함수형)   |
| **실행 흐름**  | 데이터를 한 번 훑으면서 옆에다 기록함.           | 데이터를 걸러내는 **새로운 RDD**를 만들고 숫자를 셈.  |
| **안전성**    | 재수 없으면 중복 카운트 될 수 있음.            | **100% 정확함.** (스파크가 보장함)           |

## 그럼에도 Accumulator를 배워야 하는 3가지 이유 

"위험하다면서 왜 배워야 하나요?" -> **개발자에게는 최고의 무기**이기 때문입니다.

### ① "공짜"라서 씁니다 (성능 최적화) 

`filter().count()`는 데이터를 **한 번 더 읽어야(Scan)** 합니다. 
하지만 Accumulator는 메인 로직(`map`, `foreach`)이 돌아갈 때 **"곁다리로"** 슬쩍 카운트를 셉니다.

- **효과:** 추가적인 Job 실행 없이 **비용 "0"** 으로 통계를 얻을 수 있습니다.

### ② Spark UI에 그래프로 뜹니다 

Accumulator를 쓰면 **Spark UI (Web Dashboard)** 의 `Stages` 탭에 **사용자 정의 지표(Custom Metrics)** 로 예쁘게 표시됩니다.
>Stages 내용을 이해하려면  [[Spark_UI_Guide]]  같이 참고해서 보시오~!

- **효과:** 운영 중에 "지금 작업이 50% 진행됐네?", "에러가 갑자기 폭증하네?"를 실시간으로 감시할 수 있습니다.

### ③ 디버깅의 구세주 (억울함을 풀 때) 

"분명 로직은 맞는데 왜 데이터가 비지?" 이럴 때 코드 사이사이에 Accumulator를 심어보세요.

- **효과:** "이 `if` 문을 몇 번 탔는지" 확인하면, **"아, 데이터가 애초에 여기로 안 들어왔구나!"** 하고 바로 원인을 찾을 수 있습니다.

---
## 핵심 주의사항 (절대 규칙!) 

Accumulator는 반드시 **Action(`foreach`, `count` 등) 안에서만** 사용해야 합니다.

- **Transformation(`map`, `filter`)에서 사용 시:** 스파크가 최적화를 위해 태스크를 여러 번 시도하거나 재실행할 경우, 카운트가 **뻥튀기(중복)** 될 수 있습니다.
- **Action에서 사용 시:** 스파크가 태스크 재실행이 발생해도, Accumulator 업데이트는 **"딱 한 번만(Idempotent)"** 반영되도록 보장해줍니다.

---
>**'정확한 계산'은 `count()`에게 맡기고, 너는 '빠른 모니터링'이 필요할 때만 Accumulator를 꺼내 써라!**"
