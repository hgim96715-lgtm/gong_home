## 개념 한 줄 요약 (Concept Summary)

Flink 대시보드는 **"내 공장(Cluster)의 CCTV"** 다. 
데이터가 어디서 들어와서 어디로 나가는지, 중간에 막히거나(Bottleneck) 터진 곳은 없는지 실시간으로 보여준다.

---
##  왜 필요한가 (Why)

* **블랙박스 탈출:** 코드를 실행(`run`)하고 나면 터미널은 조용해진다. 이때 대시보드가 없으면 "지금 돌고 있는 거야? 멈춘 거야?" 하고 답답해 미친다.
* **성능 튜닝:** "왜 이렇게 느리지?" 할 때, 어느 단계(박스)에서 데이터가 쌓이는지 눈으로 확인하고 그 부분만 고칠 수 있다.

## Practical Context (실무 포인트)

성공한 화면(세 번째 스크린샷)에서 확인해야 할 **핵심 3요소**는 다음과 같다.

### ① 상태 (Status) : "파란색인가?"

* **위치:** 화면 상단의 `Job State: FINISHED` 파란색 배지.
* **의미:**
    * 🔵 **FINISHED:** 성공적으로 완료됨. (Batch 작업일 때)
    * 🟢 **RUNNING:** 지금 열심히 도는 중. (Streaming 작업일 때)
    * 🔴 **FAILED:** 에러 나서 뻗음. (로그 확인 필요)
    * 🟡 **CANCELED:** 사용자가 강제로 멈춤.

### ② DAG (Job Graph) : "박스가 왜 합쳐졌지?"

* **위치:** 화면 중앙의 파란색 네모 박스들.

* **의미:** 우리가 짠 코드의 흐름도.
    * **왼쪽 박스:** `Source` (읽기) + `FlatMap` + `Map` ...
    * **오른쪽 박스:** `Reduce` (합치기) + `Sink` (쓰기)

* **핵심 질문:** "코드는 5단계로 짰는데 왜 박스는 2개뿐이야?"
    * **Operator Chaining (체인 연결):** Flink는 똑똑해서, "굳이 옆 사람한테 넘기지 말고 니가 다 해!"라며 합칠 수 있는 작업은 **한 덩어리(Task)** 로 묶어버린다.
    * 이게 묶여 있어야 데이터 이동 비용이 줄어서 **성능이 빨라진다.**

### ③ 수치 (Metrics) : "데이터가 진짜 갔어?"

* **위치:** 하단 테이블의 `Records Sent` / `Records Received`

* **분석 (님 화면 기준):**
    * **Records Sent: 31** → "총 31번 데이터를 옆 박스로 던졌다."
    * **Records Received: 31** → "옆 박스에서 31개를 잘 받았다."
    * **Bytes Sent: 1.21 KB** → "데이터 크기는 아주 작다."

* **활용:** 만약 코드는 정상인데 결과 파일이 비어있다면? 여기 숫자가 **0**인지 확인하면 된다. 0이면 로직이 틀린 것!

## 상세 분석 (Detailed Analysis)

### 

1.  **Job Name:** `word_count_20260206`
    * 아까 `env.execute("...")`에 적은 이름이 여기 뜬다. 

2.  **Duration:** `4s 553ms`
    * 전체 작업에 4.5초 걸렸다.

3.  **Parallelism:** `1`
    * 박스 안에 `Parallelism: 1`이라고 써있다. 일꾼 1명이 처리했다는 뜻.

## 초보자의 오해 (Common Misconceptions)

* **Q: FAILED(빨간색)가 뜨면 망한 건가요?**
    * 아니다. `Exceptions` 탭(상단 메뉴)을 누르면 **"왜 죽었는지"** 에러 로그가 아주 친절하게 나온다. 터미널보다 여기서 보는 게 훨씬 편하다.

* **Q: Timeline은 뭔가요?**
    * 각 작업이 언제 시작해서 언제 끝났는지 보여주는 막대그래프다. 나중에 "어떤 작업이 제일 오래 걸렸나?" 범인 찾을 때 쓴다.
