## 개념 한 줄 요약

**"원본 파일을 가리키는 두 가지 방법: '별명' vs '바로가기'"**

* **Inode (주민등록번호):** 파일의 실제 정체(메타데이터)를 담고 있는 고유 번호.
* **Hard Link (별명):** 하나의 파일(Inode)에 이름표를 하나 더 붙이는 것. (원본과 운명 공동체)
* **Soft Link (바로가기):** 원본 파일의 **위치(Path)** 만 가리키는 이정표. (윈도우의 바로가기 아이콘)

---
## 왜 필요한가? (Why)

**문제점:**
- "로그 파일이 너무 깊은 곳(`/var/www/html/project/logs/error.log`)에 있어서 열 때마다 타자 치기 힘들어요."
- "중요한 데이터라 실수로 지워질까 봐 백업본을 만들고 싶은데, 복사(`cp`)하면 용량을 2배로 차지하잖아요."

**해결책:**
- **Soft Link:** 홈 디렉토리에 `log_link`라는 바로가기를 만들어서 클릭 한 번으로 접근한다.
- **Hard Link:** 용량 차지 없이 파일에 접근할 수 있는 '비상구'를 하나 더 뚫어둔다. (원본 이름을 지워도 하드 링크가 남아있으면 데이터는 살아있음)

---
## 핵심 개념: Inode (Index Node) 

리눅스 파일 시스템의 심장이다. 파일의 **"실제 데이터가 디스크 어디에 있는지"** 알려주는 지도 역할을 한다.

* **포함하는 정보:** 파일 크기, 소유자, 권한, 수정 시간, 데이터 블록 위치 등.
* **포함하지 않는 정보:** **파일 이름** (이름은 그냥 껍데기일 뿐!)
* **특징:** 모든 파일은 고유한 Inode 번호를 가진다.

---
## 상세 분석: Hard Link vs Soft Link

| 특징 | Hard Link (하드 링크) | Soft Link (심볼릭 링크) |
| :--- | :--- | :--- |
| **비유** | **'별명' (Nick Name)** | **'바로가기' (Shortcut)** |
| **원리** | 원본과 **똑같은 Inode**를 가리킴 | 원본의 **파일 경로(Path)**를 가리킴 |
| **원본 삭제 시** | **살아있음** (데이터 유지됨) | **깨짐** (Dangling Link, 빨간색 뜸) |
| **파일 시스템** | 같은 디스크(파티션) 안에서만 가능 | 다른 디스크로 연결 가능 |
| **디렉토리** | 폴더는 링크 불가 (무한루프 방지) | 폴더도 링크 가능 (자주 씀) |

## 링크 만들고 사용하기 (How to Use)

### 링크 생성 

**`ln` (Link)** 명령어를 사용한다. 옵션 `-s`가 있냐 없냐가 핵심이다.

```bash
# 1. Soft Link 만들기 (추천 ⭐️)
# ln -s [원본] [바로가기이름]
# "target.txt를 가리키는 soft.lnk를 만들어라"
ln -s target.txt soft.lnk

# 2. Hard Link 만들기
# ln [원본] [하드링크이름] (옵션 없음)
ln target.txt hard.lnk
```

### B. 링크 사용 (Use)

링크 파일도 **"파일"** 이다. 원본이랑 똑같이 다루면 된다.

```bash
# 내용 읽기 (cat)
cat soft.lnk   # 원본(target.txt)의 내용이 보인다.

# 내용 수정 (vim / nano)
nano hard.lnk  # 여기서 수정하고 저장하면, 원본(target.txt)도 같이 바뀐다!

# 디렉토리 이동 (cd) - 만약 링크 대상이 폴더라면
cd my_link_folder  # 원본 폴더로 이동된다.
```

### C. 결과 확인 (ls -l)

```bash
ls -l
# 결과 예시:
# -rw-r--r--  2 user  user   85 ... hard.lnk  (맨 앞 숫자가 2가 됨 -> 하드링크)
# lrwxrwxrwx  1 user  user   10 ... soft.lnk -> target.txt (화살표가 보임 -> 소프트링크)
```

## 실전 분석 

### ① 파일의 주민등록증 확인 (`stat`)

```bash
stat file.txt

# [해석]
# File: file.txt
# Size: 85 Blocks: 8 IO Block: 4096 regular file
# Device: 3eh/62d Inode: 690626 Links: 1
# ...
```

- **Inode: 690626** 👉 이 파일의 **진짜 고유 번호**다. (이름이 바뀌어도 이 번호는 안 바뀜)
- **Links: 1** 👉 현재 이 Inode(690626)를 가리키는 파일 이름이 **1개**(`file.txt`)라는 뜻.
- _만약 하드 링크를 하나 만들면 이 숫자가 **2**로 변한다!_

### ② 파일 시스템의 남은 자리 확인 (`df -i`)

디스크 용량(GB)이 남아도, **Inode(번호표)** 가 다 떨어지면 파일을 못 만든다!

```bash
$ df -i

# [해석]
# Filesystem       Inodes    IUsed     IFree IUse% Mounted on
# overlay       121782272  1015863 120766409    1% 
```

- **Inodes:** 발급 가능한 전체 번호표 개수 (약 1.2억 개)
- **IUsed:** 현재 사용 중인 번호표 (약 100만 개)
- **결론:** 아주 넉넉하다. (가끔 파일이 수억 개 생성되는 서버에서는 이 수치를 꼭 확인해야 함)

## 초보자가 자주 하는 실수 (Misconceptions)

### ① "하드 링크를 지우면 원본도 지워지나요?"

- **아니요.** 하드 링크는 원본과 동등한 권리를 가집니다.
- `Links: 2`인 상태에서 하나를 지워도 `Links: 1`이 되어 데이터는 살아있습니다.
- `Links: 0`이 되어야 비로소 데이터가 삭제됩니다.

### ② "소프트 링크는 용량 안 차지하나요?"

- 아주 조금 차지합니다. "원본의 주소(글자)"를 저장할 만큼의 아주 작은 공간만 씁니다.

### ③ "실무에선 뭘 더 많이 써요?"

- 95% 이상 **Soft Link (`ln -s`)** 를 씁니다.
- 복잡한 설정 파일 경로를 단축하거나, 프로그램 버전 관리(`python3` -> `python3.9`)할 때 필수입니다
