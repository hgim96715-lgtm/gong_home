## 개념 한 줄 요약

"쉘(Shell)은 사용자가 입력한 명령을 해석해 리눅스의 심장인 커널(Kernel)에게 전달하고, 그 결과를 다시 사용자에게 보여주는 번역기이다."
껍데기라는 이름처럼 알맹이인 커널을 감싸고 있는 인터페이스 층을 의미한다.

---
## CLI vs GUI: 인터페이스의 두 얼굴

리눅스와 소통하는 방식은 크게 두 가지로 나뉜다.

### ① GUI (Graphical User Interface)

- **방식:** 마우스로 아이콘을 클릭하고 창을 옮기는 그래픽 환경.
- **비유:** 식당에 가서 메뉴판의 **사진**을 보고 손가락으로 가리켜 주문하는 것.
- **장점:** 배우기 쉽고 직관적이다.
- **단점:** 세밀한 설정이 어렵고, 서버 자원을 많이 소모한다


### ② CLI (Command-Line Interface) ⭐️ 데이터 엔지니어의 주력

- **방식:** 키보드로 직접 텍스트 명령어를 타이핑하는 환경.
- **비유:** 요리사에게 "양파 3mm 두께로 썰어서 약불에 5분간 볶아주세요"라고 **글**로 상세하게 주문하는 것.
- **장점:** 반복 작업을 자동화하기 좋고, 매우 정교한 제어가 가능하다.
- **단점:** 명령어를 외워야 하며 진입 장벽이 있다.

----
## 왜 데이터 엔지니어는 CLI(쉘)를 써야 할까?

1. **자동화 (Automation):** 수백 장의 파일을 하나씩 마우스로 옮길 수 없지만, 쉘에서는 명령어 한 줄(`mv *.log backup/`)로 끝난다.
2. **원격 접속:** 실제 데이터 서버는 화면이 없는 경우가 많다. 텍스트만 주고받는 쉘은 네트워크가 느려도 원격 조종이 가능하다.
3. **자원 효율:** GUI를 돌릴 메모리(RAM)를 아껴서 실제 데이터 연산(`Spark` 등)에 더 쏟아부을 수 있다.

## 쉘의 종류 (Shell Flavors)

우리가 쓰는 쉘도 여러 가지 브랜드가 있다.

- **Bash (Bourne Again Shell):** 대부분의 리눅스 배포판(Ubuntu, CentOS 등)의 기본 쉘. 가장 표준이다
- **Zsh (Z Shell):** 맥북(macOS)의 기본 쉘. `Bash`를 기반으로 자동 완성, 테마 등 편리한 기능이 추가된 현대적인 쉘이다.
- **Sh (Bourne Shell):** 가장 오래되고 가벼운 조상님 쉘. 시스템 복구 등 최소한의 환경에서 쓴다.

## Practical Context: 쉘과 터미널은 다른 건가요?

입문자가 가장 헷갈리는 부분이다.

- **터미널 (Terminal / 콘솔):** 껍데기를 보여주는 **화면(프로그램)** 그 자체. (예: Mac의 Terminal, iTerm2, VS Code의 터미널 창)
- **쉘 (Shell):** 그 화면 뒤에서 명령어를 기다리고 해석하는 **프로그램(두뇌)**. (예: `zsh`, `bash`)

> **상황:** 사용자님은 Mac의 **터미널** 프로그램 안에서 **Zsh(쉘)** 을 통해 `docker exec`라는 명령어를 날려 **리눅스 컨테이너**와 대화하고 있었던 것이다

>`docker exec -it ... bash`라고 쳤던 거 기억나? 그건 '도커야, 저 컨테이너 안에 들어가서 **bash라는 쉘**을 하나 띄워줘!'라는 뜻이었어.
