## 개념 한 줄 요약

**"리눅스판 '틀린 그림 찾기' 형제들"**

* **`diff` (Difference):** 텍스트 파일의 **내용(줄 단위)** 이 무엇이 바뀌었는지 상세하게 알려준다. (개발자용), 텍스트 전용
* **`cmp` (Compare):** 두 파일이 **완전히 똑같은지(바이트 단위)** 기계적으로 확인한다. (컴퓨터용), 이미지나 바이너리 파일

---

##  왜 필요한가? (Why)

**문제점:**
- "서버 설정 파일(`nginx.conf`)을 고쳤는데 에러가 나요. 원본이랑 뭘 다르게 썼을까요?"
- "10GB짜리 데이터 백업했는데, 원본이랑 1바이트도 안 틀리고 똑같이 복사된 거 맞나요?"

**해결책:**
- **`diff`** 로 "3번째 줄에 오타 났네"라고 바로 찾아낸다.
- **`cmp`** 로 "파일 크기는 같은데 내용물이 깨졌네"라고 무결성을 검증한다.

---

## 실무 적용 사례 (Practical Context)

1.  **코드 리뷰 & 설정 비교 (`diff`):**
    - 개발 서버 설정(`dev.conf`)과 운영 서버 설정(`prod.conf`)이 어디가 다른지 비교할 때.
    - Git에서 `git diff`가 내부적으로 사용하는 원리.

2.  **파일 무결성 검증 (`cmp`):**
    - 이미지 파일(.jpg)이나 실행 파일(.exe) 같은 **바이너리 파일**이 깨졌는지 확인할 때.
    - 대용량 데이터 전송 후 검증용.

>실무에서는 **설정 파일 비교할 땐 `diff -u`**,  
>**데이터 파일 깨졌나 확인할 땐 `cmp`**

## Code Core Points: ① `diff` (텍스트 비교)

줄 단위로 비교해서 **"어떻게 고쳐야 두 파일이 같아지는지"** 를 보여준다.

### A. 기본 사용법

```bash
# 사용법: diff [파일1] [파일2]
diff old.txt new.txt

# 결과 해석:
# < (왼쪽 파일 내용)
# > (오른쪽 파일 내용)
# 1c1 : 첫 번째 줄(1)이 변경(c) 되었다.
```

### B. 실무 필수 옵션

그냥 `diff`만 치면 보기가 힘들어서, 보통 옵션을 붙여서 쓴다.

```bash
# 1. 보기 편하게 출력하기 (-u: Unified Mode) ⭐️
# Git diff 처럼 -, + 기호로 변경 사항을 한눈에 보여줌. (가장 많이 씀)
diff -u old.txt new.txt

# 2. 공백 무시하기 (-w: Ignore Whitespace) ⭐️
# 띄어쓰기나 탭 차이는 무시하고 "내용"만 비교. (코드 비교할 때 필수)
diff -w code_v1.py code_v2.py

# 3. 폴더째로 비교하기 (-r: Recursive)
# 두 폴더 안에 있는 모든 파일을 싹 다 비교함.
diff -r folder_A/ folder_B/

# 4. 대소문자 무시 (-i: Ignore Case)
diff -i file1.txt file2.txt
```

### [보충] diff 결과 해석법 (암호 해독기)

출력되는 `1c1`, `2d3` 같은 코드는 **"[왼쪽파일 줄번호] [동작] [오른쪽파일 줄번호]"** 형식이다.

**1. 동작(Action)의 종류**
- **`c` (Change):** 내용이 **바뀜**. (수정됨)
    - `2c2` -> 왼쪽 2번째 줄이 오른쪽 2번째 줄로 변함.

- **`a` (Add):** 내용이 **추가됨**.
    - `5a6` -> 왼쪽 5번째 줄 **다음에**, 오른쪽 6번째 줄 내용이 추가됨.
        
- **`d` (Delete):** 내용이 **삭제됨**.
    - `3d2` -> 왼쪽 3번째 줄이 삭제되어서, 오른쪽엔 없음(2번째 줄에서 멈춤).

**2. 화살표의 의미**

- **`<` (왼쪽 꺽쇠):** **왼쪽 파일(원본)** 에만 있는 내용 (즉, 지워지거나 바뀐 옛날 내용)
- **`>` (오른쪽 꺽쇠):** **오른쪽 파일(새파일)** 에만 있는 내용 (즉, 새로 추가되거나 바뀐 새 내용)

```bash
diff file1.txt file2.txt

# 출력 결과
2c2           <-- "2번째 줄이 바뀌었어!"
< banana      <-- (왼쪽 파일 내용)
---
> BANANA      <-- (오른쪽 파일 내용)

4d3           <-- "왼쪽 4번째 줄이 삭제됐어!"
< melon       <-- (왼쪽 파일에만 있고 오른쪽엔 없음)
```

## Code Core Points: ② `cmp` (정밀 비교)

파일을 **바이트(Byte)** 단위로 뜯어본다. 
텍스트보다는 **이진 파일(Binary)** 비교에 강력하다.

### A. 기본 사용법 및 결과값 (Exit Status)

화면에 아무것도 안 나오면 **"완벽히 똑같다"** 는 뜻이다.

```bash
cmp file1.jpg file2.jpg

# 결과(Exit Code) 확인: echo $?
# 0 : 두 파일이 완벽하게 동일함.
# 1 : 다름. (Differ)
# 2 : 에러 (파일이 없거나 권한 없음)
```

### B. 주요 옵션

```bash
# 1. 조용히 결과만 줘 (-s: Silent) ⭐️
# 화면 출력 없이 성공/실패 여부(Exit Code)만 필요할 때 (스크립트용)
cmp -s data.bin backup.bin && echo "복사 성공!" || echo "파일 깨짐!"

# 2. 어디가 다른지 자세히 (-l, -b)
# 틀린 바이트 위치와 값을 8진수/문자로 보여줌
cmp -lb file1.txt file2.txt
```

### C. 결과 해석 가이드 (암호 해독) 🕵️‍♀️

`cmp` 결과에 나오는 숫자는 10진수가 아니라 **8진수(Octal)** 다.
사람이 보려면 **`-b` (print bytes)** 옵션을 써야 정신 건강에 좋다.

**상황 예시:**

```bash
cmp -l f1.txt f2.txt
# 결과: 24  61  62
# 해석: "24번째 글자가 다르다. (왼쪽은 8진수 61, 오른쪽은 8진수 62)"

cmp -b f1.txt f2.txt
# 결과: ... byte 24, line 2 is  61 1  62 2
# 해석: "24번째 글자가 왼쪽은 '1'(값 61), 오른쪽은 '2'(값 62)다."
```

>**💡 왜 8진수(Octal)를 쓰나요?** 
>리눅스/유닉스의 조상님들이 3비트씩 끊어 읽는 걸 좋아했기 때문입니다. (전통)
> 그냥 "아, 이건 컴퓨터용 숫자구나" 하고 **`-b` 옵션으로 실제 글자를 확인**하는 게 낫습니다.

## 초보자가 자주 하는 실수 (Mistakes)

### ① "`diff`를 썼는데 화면이 깨져요! 😱"

- **이유:** 이미지나 실행 파일(Binary)을 `diff`로 열어서 그렇다. `diff`는 텍스트 전용이다.
- **해결:** 바이너리 파일은 **`cmp`** 를 써야 한다.


### ② "내용은 똑같은데 `diff`가 다르대요!"

- **이유:** 눈에 안 보이는 **공백(Space)** 이나 **엔터키(Windows vs Linux 줄바꿈 차이)** 때문일 확률 99%.
- **해결:** **`diff -w`** (공백 무시) 옵션을 써보면 "아, 띄어쓰기 때문이었구나" 하고 알 수 있다.
