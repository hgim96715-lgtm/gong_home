## 개념 한 줄 요약 

**"명령어의 실행 결과를 화면에 뿌리지 않고 파일로 저장하거나(`>`, `>>`), 다른 명령어에게 일감을 토스(`|`)해주는 연결 고리이다."**

---
## 왜 필요한가? (Why)

**문제:**
- 리눅스 명령어(ls, cat 등)는 기본적으로 결과를 **터미널 화면(Standard Output)** 에만 보여주고 휘발된다.
- 데이터를 가공하고 싶은데, 화면에 뜬 글자를 복사해서 엑셀에 붙여넣기 할 수는 없지 않은가?

**해결:**
- **리다이렉션(`>`):** "화면에 뿌리지 말고 파일에 적어놔!"라고 방향을 튼다.
- **파이프(`|`):** "네가 처리한 결과를 다음 명령어한테 바로 넘겨줘!"라고 연결한다. 이것이 리눅스의 철학인 **'작은 도구들의 조합'** 을 완성한다.

---
## 실무 적용 사례 (Practical Context)

데이터 엔지니어는 이 기호들을 숨 쉬듯이 쓴다.

1.  **로그 저장:** 파이썬 스크립트 실행 결과를 화면에 띄우면 퇴근할 때 사라지니까 파일로 저장한다.
    - `python dag_run.py >> daily_log.txt`
2.  **데이터 파이프라인:** CSV를 읽어서(`cat`) -> 특정 단어를 찾고(`grep`) -> 개수를 센다(`wc`).
    - `cat data.csv | grep "Error" | wc -l`
3.  **환경 설정:** 필요한 패키지 목록을 파일로 내보낸다.
    - `pip freeze > requirements.txt`

##  코드 핵심 포인트 (Code Core Points)

|    기호    | 이름            | 기능                           | 비유             |
| :------: | :------------ | :--------------------------- | :------------- |
| **`>`**  | **Overwrite** | 결과를 파일에 저장 (기존 내용 **삭제**됨)   | 새 종이에 받아적기     |
| **`>>`** | **Append**    | 결과를 파일 끝에 추가 (기존 내용 **보존**됨) | 쓰던 노트 다음 줄에 적기 |
|    \|    | **Pipe**      | 앞 명령어의 결과를 뒤 명령어의 입력으로 전달    | 공장의 컨베이어 벨트    |

## 상세 코드 분석 (Detailed Analysis)

### ① 리다이렉션: 화면 대신 파일로 (`>`, `>>`)

```bash
# 1. 덮어쓰기 (>) : 기존 파일이 있으면 내용을 다 지우고 새로 씀 ⚠️ 주의!
# "Hello"라는 글자를 welcome.txt 파일에 저장해라.
echo "Hello Linux" > welcome.txt

# 2. 이어쓰기 (>>) : 기존 파일 내용을 유지하고 맨 뒤에 추가함.
# 로그 쌓을 때 무조건 이걸 써야 함.
echo "Logs for 2026-01-25" >> welcome.txt

# 확인해볼까?
cat welcome.txt
# 출력:
# Hello Linux
# Logs for 2026-01-25
```

### ② 파이프: 명령어끼리 이어달리기 (`|`)

```bash
# 시나리오: 현재 폴더에 파일이 몇 개나 있는지 세고 싶다.

# 1단계: 목록 출력 (ls)
ls -l
# 결과가 화면에 쫘르륵 나온다. (사람 눈엔 보이지만 컴퓨터는 계산 불가)

# 2단계: 파이프 연결 (|)
# ls의 결과를 화면에 보여주는 대신, wc(Word Count) 명령어에게 넘겨준다.
ls -l | wc -l
# 해석: 리스트를 뽑아서(ls) -> 라인 수를 세어라(wc -l)
# 결과: 15 (파일 개수만 딱 나옴)
```

### ③ 심화: 조합해서 쓰기 (Pipe + Redirection)

```bash
# "프로세스 목록(ps)을 뽑아서 -> 'python'을 포함한 줄만 찾아서(grep) -> 파일로 저장해(>)"
ps -ef | grep "python" > python_processes.txt
```

## 5. 심화: 명령어 이어달리기 (Chaining Operators)

명령어 여러 개를 한 줄에 쓰고 싶을 때 사용하는 기호들입니다. 
**성공 여부**에 따라 실행 여부가 갈립니다.

### ① `&&` (AND 연산자: 성공하면 실행) 

* **의미:** "앞의 명령어가 **성공(에러 없음)** 해야만 뒤의 명령어를 실행해라."
* **사용 예시:**

  ```bash
  # 업데이트 목록 갱신에 성공하면 -> 실제 업그레이드 진행
  sudo apt update && sudo apt upgrade
  
  # 폴더 생성에 성공하면 -> 그 폴더로 이동
  mkdir my_folder && cd my_folder
  ```

### ② `||` (OR 연산자: 실패하면 실행) 

- **의미:** "앞의 명령어가 **실패(에러 발생)** 했을 때만 뒤의 명령어를 실행해라." (주로 에러 처리용)
- **사용 예시:**

```bash
# 파일 복사에 실패하면 -> 에러 메시지 출력
cp file1.txt backup/ || echo "파일 복사 실패!"
```

### ③ `;` (세미콜론: 무조건 실행) 

- **의미:** "성공하든 실패하든 상관없이 그냥 순서대로 다 실행해."
- **사용 예시:**

```bash
# date 실행하고 나서 (실패하든 말든) echo 실행
date ; echo "시간 출력 끝"
```

## 초보자가 자주 착각하는 포인트

- **`>` 썼다가 망했어요:**
    - 중요한 설정 파일에 `echo "test" > config.conf`를 하는 순간, 기존 수백 줄의 설정이 사라지고 "test" 한 줄만 남는다.
    - **항상 `>>`인지 `>`인지 확인하는 습관**을 들여야 한다.

- **에러 메시지는 왜 저장이 안 되나요?**
    - `>`는 **표준 출력(성공한 결과)** 만 가져간다. 
    - 에러 메시지(표준 에러)는 여전히 화면에 뜬다.
    - 에러까지 파일로 넣으려면 `command > log.txt 2>&1` 같은 고급 문법이 필요하다. (
    - 일단은 성공 결과만 저장된다고 알면 충분하다.)
   
- **파이프 앞뒤 순서:**
    - 데이터는 무조건 **왼쪽에서 오른쪽(`->`)** 으로 흐른다. 
    - `grep "error" | cat log.txt` (X)는 틀린 문법이다. 
    - 파일을 먼저 읽고(`cat`) 필터링(`grep`)해야 한
