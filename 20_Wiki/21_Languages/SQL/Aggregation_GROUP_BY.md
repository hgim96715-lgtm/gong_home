## 개념 한 줄 요약

**GROUP BY**는 흩어져 있는 데이터들을 특정 기준(컬럼)으로 묶어서, **그룹별 통계(개수, 합계, 평균 등)** 를 내는 명령어다.
(엑셀의 **피벗 테이블** 기능을 SQL로 짠다고 생각하면 딱이야! 📊

---
## 왜 필요한가 (Why)

**Raw Data의 한계:**
- "주문 내역 100만 줄"을 그냥 보여주면 아무런 인사이트가 없어.

**요약의 힘:**
- "그래서 **일별(Daily)** 매출이 얼만데?"
- "**카테고리별(Category)** 로 가장 많이 팔린 게 뭐야?"
- "**팀별(Team)** 평균 연봉이 얼마야?"
- 👉 이런 질문에 답하려면 데이터를 묶어서 계산해야 해

---
## Practical Context (실무 활용)

보통 리포트나 대시보드를 만들 때 무조건 쓰여.

* **e커머스:** 상품 카테고리별 총판매액 (`SUM`)
* **로그 분석:** 시간대별 접속자 수 (`COUNT`)
* **HR:** 부서별 평균 근속 연수 (`AVG`)

## Code Core Points (문법 공식)

```sql
SELECT
    기준_컬럼,          -- 1. 묶을 기준 (예: 부서명)
    COUNT(*),         -- 2. 집계 함수 (예: 인원수)
    SUM(매출액)        -- 2. 집계 함수 (예: 총매출)
FROM 테이블명
WHERE 조건             -- 3. 묶기 전 필터링 (원본 데이터 거르기)
GROUP BY 기준_컬럼     -- 4. ⭐️ SELECT에 있는 기준 컬럼은 무조건 여기 써야 함!
HAVING 집계_조건       -- 5. 묶은 후 필터링 (통계치 거르기)
ORDER BY 정렬_기준;
```

##  기준을 여러 개로 쪼개고 싶다면? (다중 그룹핑)

**"부서별 인원수"** 만 궁금한 게 아니라, **"부서별 + 성별 인원수"** 까지 쪼개서 보고 싶다면?
👉 고민할 필요 없이 **콤마(`,`)** 로 구분해서 계속 적어주면 된다.

### 개념 시각화 (폴더 정리와 같아)

`GROUP BY 부서, 성별` 이라고 쓰면 데이터가 이렇게 묶인다고 상상해 봐.

- 📂 **영업팀** (1차 그룹)
    - 📁 남자 (2차 그룹) 👉 3명
    - 📁 여자 (2차 그룹) 👉 2명
- 📂 **인사팀** (1차 그룹)
    - 📁 남자 (2차 그룹) 👉 1명
    - 📁 여자 (2차 그룹) 👉 4명

###  코드 예시

```sql
SELECT
    city,          -- 1차 기준 (도시)
    gender,        -- 2차 기준 (성별)
    COUNT(*) as cnt
FROM customers
GROUP BY city, gender; -- ⭐️ 그냥 순서대로 콤마 찍고 나열하면 끝!
```

###  결과 데이터 형태

결과는 **"가능한 모든 조합(Combination)"** 이 한 줄씩 나온다.

|**city**|**gender**|**cnt**|
|---|---|---|
|서울|남|150|
|서울|여|180|
|부산|남|90|
|부산|여|110|

> **주의할 점 (Select 절의 규칙)** 
> `GROUP BY`에 컬럼을 2개 썼으면, **`SELECT` 절에도 그 2개를 똑같이 써주는 게 국룰**이야.

##  Deep Dive: 집계함수와 NULL의 비밀  (매우 중요!)

단순히 "개수 세고 합계 구한다"라고 알면 큰일 나. 
**"NULL(빈 값)을 포함하냐 마냐"** 가 천지차이거든.

### ① COUNT 삼형제 (가장 많이 헷갈림)

가상의 테이블 `orders`에 데이터가 5줄 있는데, `user_id` 하나가 **NULL(비회원 주문)** 이라고 치자.

| row_id | user_id | 비고 |
| :--- | :--- | :--- |
| 1 | A | |
| 2 | A | (중복) |
| 3 | B | |
| 4 | C | |
| 5 | **NULL** | 🚨 비회원 |

| 함수 | 결과 | 설명 | **NULL 처리** |
| :--- | :--- | :--- | :--- |
| **`COUNT(*)`** | **5** | 테이블의 **"줄(Row) 수"** 를 셈. | **포함 O** (NULL이든 뭐든 줄만 있으면 셈) |
| **`COUNT(컬럼)`** | **4** | 해당 컬럼에 **"값이 있는"** 칸만 셈. | **무시 X** (NULL은 카운트 안 함) |
| **`COUNT(DISTINCT 컬럼)`** | **3** | 중복 떼고, 빈 값 떼고 **"순수 유저 수"** 만 셈. (A, B, C) | **무시 X** (NULL 제외 + 중복 제거) |

> **💡 실무 Tip:**
> "전체 주문 건수 가져와" 👉 `COUNT(*)` (비회원 주문도 주문이니까!)
> "활동 중인 회원 수 가져와" 👉 `COUNT(user_id)` (비회원은 회원 수에 넣으면 안 되니까!)

### ② 기타 집계함수 (SUM, AVG, MIN, MAX)

얘네들은 기본적으로 **NULL을 "없는 셈" 쳐.** (0으로 치는 게 아님! 아예 투명인간 취급)

* 데이터: `[100, 200, NULL]` (총 3건)

| 함수 | 계산 과정 | 결과 | 주의사항 🚨 |
| :--- | :--- | :--- | :--- |
| **`SUM(컬럼)`** | 100 + 200 | **300** | NULL을 0으로 보고 더한 것과 결과는 같음. |
| **`AVG(컬럼)`** | (100 + 200) / **2** | **150** | **나누기 3이 아니라 나누기 2야!** (여기서 사고 많이 터짐) |
| **`MIN(컬럼)`** | 100, 200 중 최소 | **100** | NULL은 비교 대상에서 제외 |
| **`MAX(컬럼)`** | 100, 200 중 최대 | **200** | NULL은 비교 대상에서 제외 |

> **💡 AVG의 함정:**
> 만약 `NULL`을 `0`으로 쳐서 평균을 내고 싶다면? (예: 매출 없는 날도 포함해서 일평균 구할 때)
> 👉 **`AVG(COALESCE(amount, 0))`** 처럼 NULL을 0으로 바꿔주는 함수를 씌워야 해!

## Detailed Analysis (상황별 함수 선택 가이드)

가상의 **`orders` (주문)** 테이블이 있다고 치자.

### 상황 A. "카테고리별로 **몇 개나(횟수)** 팔렸어?" (COUNT)

주문이 발생한 **'사건의 빈도'** 를 묻고 있다.

- **목적:** 건수 세기
- **함수:** `COUNT(*)` 또는 `COUNT(컬럼명)`

```sql
SELECT
    category,            -- 기준: 카테고리
    COUNT(order_id) as cnt      -- 집계: 주문 번호가 몇 개인지(주문 건수) 세어줘
FROM orders
GROUP BY category;       -- 카테고리가 같은 놈들끼리 뭉쳐!
```

> "아이디(ID)는 **세는 것(COUNT)** 이지, **더하는 것(SUM)** 이 아니다."

### 상황 B. "카테고리별로 **얼마치(총액)** 팔렸어?" (SUM)

주문된 물건들의 **'값의 합'** 을 묻고 있다.

- **목적:** 합계 구하기
- **함수:** `SUM(컬럼명)`

```sql
SELECT
    user_id,            -- 기준: 고객 ID
    SUM(amount) as total_spent  -- 집계: 금액(amount) 다 더해
FROM orders
GROUP BY user_id;
```

### 상황 C. "평균 10만 원 이상 쓴 VIP 고객만 뽑아줘." (AVG + HAVING) 중요!

- **목적:** 평균 구하기 + **통계 낸 결과로 필터링하기**
- **함수:** `AVG(컬럼명)`

```sql
SELECT
    user_id,
    AVG(amount) as avg_spent
FROM orders
GROUP BY user_id
HAVING AVG(amount) >= 100000;  -- ⭐️ WHERE가 아니라 HAVING을 써야 해!
```

## 초보자가 자주 착각하는 포인트 (면접 단골 질문)

### Q1. WHERE vs HAVING 차이가 뭐예요?

가장 많이 헷갈리고 가장 중요한 개념이야.

|**구분**|**WHERE**|**HAVING**|
|---|---|---|
|**시점**|그룹핑 **하기 전** (Before)|그룹핑 **한 후** (After)|
|**대상**|원본 데이터 (Raw Data)|요약된 데이터 (Aggregated Data)|
|**예시**|"서울 사는 사람만 뽑아서..."|"평균 매출이 100만 원 이상인 팀만..."|
|**불가**|집계함수 못 씀 (`WHERE SUM(x) > 10` ❌)|집계함수 조건 전용|

>**💡 요약:** **"원본을 거를 땐 WHERE, 계산된 결과를 거를 땐 HAVING"**
>자세한 내용은 [[HAVING_vs_WHERE]] 참조 ! 

### Q2. "에러 났어요! `not a GROUP BY expression`"

- **원인:** `SELECT`에는 있는데 `GROUP BY`에는 없는 컬럼을 썼을 때 발생해.
- **법칙:** `SELECT` 절에 있는 모든 컬럼은, **집계함수(`SUM`, `COUNT` 등)에 감싸져 있거나**, 아니면 **반드시 `GROUP BY` 절에 적혀 있어야 해.**

```sql
-- ❌ 틀린 예시
SELECT department, name, COUNT(*)  -- name은 기준으로 안 넣었는데?
FROM employees
GROUP BY department; -- 에러 발생! (같은 부서에 이름 다른 사람이 많잖아)

-- ⭕️ 맞는 예시
SELECT department, COUNT(*)
FROM employees
GROUP BY department;
```

### Q3. "평균의 합계(`SUM(AVG)`)나 합계의 평균(`AVG(SUM)`) 되나요?" 

처음에 가장 많이 하는 시도입니다.
`SELECT AVG(SUM(amount)) FROM orders GROUP BY date`... 될 것 같죠?

- **정답:** **절대 안 됩니다! (SQL 문법 에러)** 
- **이유:** SQL은 한 번의 단계(Level)에서 **오직 한 번의 집계**만 할 수 있습니다. "묶어서 합계 구하기(1차)"와 "그것들의 평균 내기(2차)"를 동시에 못 합니다.
- **해결책:** **서브쿼리(Subquery)** 를 써서 단계를 나눠야 합니다.

```sql
-- ❌ 틀린 코드 (에러 발생: Group function is nested too deeply)
SELECT AVG(SUM(total_bill))
FROM tips
GROUP BY day;

-- ⭕️ 맞는 코드 (2단계로 분리)
SELECT AVG(daily_total)
FROM (
    -- [1단계] 안쪽에서 먼저 '일별 합계'를 구해서 가상 테이블로 만듦
    SELECT day, SUM(total_bill) AS daily_total
    FROM tips
    GROUP BY day
) t;
```

## 💡 초보자 실수 패턴 분석: ID를 더하지 마라!

**상황:** "고객이 **몇 번** 대여했는지 구해라"라는 질문을 받고, 습관적으로 `SUM(customer_id)`를 작성함.

**문제점:**

1. **논리적 오류:** `customer_id`는 사람을 구별하는 **이름표(식별자)**일 뿐, 계산 가능한 **수량(Quantity)**이 아니다.
    - _비유:_ 전교생이 몇 명인지 세려고 **학번을 다 더하는 짓**을 한 셈이다. (학번 1번 + 학번 2번 = 3명? ❌)
2. **결과값 폭발:** ID가 100번인 고객이 3번 빌렸다면, 대여 횟수는 `3`이어야 하는데 `SUM`을 쓰면 `300`이 나와버린다.

**해결 공식:**

- **"~한 횟수", "~한 건수", "몇 번?"** ➡️ 무조건 **`COUNT`** (행의 개수를 센다)
- **"총 매출", "총합", "전체 금액"** ➡️ 무조건 **`SUM`** (값의 크기를 더한다)
