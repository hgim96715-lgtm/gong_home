# 관계형 데이터베이스(RDBMS)의 정체

>"수많은 데이터베이스 종류 중 전 세계에서 가장 많이 쓰이는 표준입니다. 
>데이터를 '엑셀 같은 표(Table)'로 만들고, 그 표들을 '연결(Relation)'해서 관리하는 시스템입니다."

---
## RDBMS란 무엇인가? (Relational Database Management System)

앞서 **[[Concept_Database]]** 에서 배운 DBMS(데이터베이스 관리 소프트웨어) 중에서, **관계형 모델(Relational Model)** 을 기반으로 만들어진 시스템을 RDBMS라고 부릅니다.
우리가 흔히 아는 Oracle, MySQL, PostgreSQL, SQL Server가 모두 여기에 속합니다.

### 2차원 표(Table) 형태로 저장한다

RDBMS는 데이터를 엑셀의 시트처럼 **행(Row)과 열(Column)** 로 이루어진 2차원 표 형태로 저장합니다.
수학적/논리적 용어로는 이 표를 **'릴레이션(Relation)'** 이라고 부릅니다. (SQLD 시험에서는 '테이블 = 릴레이션'이라고 생각하셔도 무방합니다.)

### 표와 표를 '연결(Relation)'한다

이름이 하필 '관계형(Relational)'인 이유가 바로 여기에 있습니다
엑셀처럼 한 시트에 모든 정보를 다 때려 넣지 않고, **주제별로 표를 쪼갠 뒤 서로 연결**합니다.

---
## 왜 굳이 표를 쪼개고 연결할까? (관계형의 위력)

그냥 한 표에 다 적으면 보기 편하지 않나요?" 데이터가 적을 때는 맞습니다. 하지만 수백만 건의 데이터를 다루는 실무에서는 끔찍한 재앙이 일어납니다.

❌ 쪼개지 않은 엑셀식 데이터 (비효율의 끝)

| **주문번호** | **고객이름** | **고객전화번호** | **고객주소** | **주문상품** |
| -------- | -------- | ---------- | -------- | -------- |
| 001      | 아무개      | 010-1111   | 서울       | 맥북       |
| 002      | 아무개      | 010-1111   | 서울       | 마우스      |
| 003      | 아무개      | 010-1111   | 서울       | 키보드      |

**문제점 (이상 현상):** 아무개 고객이 이사를 가서 주소가 '부산'으로 바뀌었습니다. 그럼 기존 주문 내역 3줄을 일일이 찾아서 다 수정해야 합니다. 10만 번 주문했다면 10만 번 수정해야 하죠. (이걸 데이터 **중복**과 **수정 이상**이라고 합니다.)


✅ RDBMS의 해결책: 쪼개고 연결하라!

고객 테이블 (회원 정보만 관리)

| **고객ID (PK)** | 고객이름 | 고객전화번호   | 고객주소 |
| :------------ | :--- | :------- | :--- |
| **USER_01**   | 아무개  | 010-1111 | 서울   |

주문 테이블 (주문 내역만 관리)

| 주문번호 | **주문한_고객ID (FK)** | 주문상품 |
| :--- | :---------------- | :--- |
| 001  | **USER_01**       | 맥북   |
| 002  | **USER_01**       | 마우스  |
| 003  | **USER_01**       | 키보드  |

**해결:** 이제 김한경 고객이 이사를 가도 **고객 테이블에서 주소를 딱 1번만 '부산'으로 바꾸면 끝**입니다!
이렇게 공통된 기준점(`고객ID`)으로 두 테이블을 묶어주는 것을 **관계(Relationship)** 라고 부르며, 이때 사용하는 문법이 바로 우리가 아는 **JOIN**입니다.

## SQL과 RDBMS의 관계

- **RDBMS:** 표를 만들고 관계를 맺어주는 거대한 '데이터 창고 시스템'
- **SQL (Structured Query Language):** 그 창고 관리인(RDBMS)에게 "고객 테이블이랑 주문 테이블 연결해서 엑셀처럼 보여줘!"라고 명령을 내리는 **'전용 대화 언어'**
