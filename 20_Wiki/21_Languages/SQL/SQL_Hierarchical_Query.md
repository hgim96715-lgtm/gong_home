# Hierarchical_Query : 꼬리에 꼬리를 무는 트리 구조 탐색(계층 쿼리)

## Concept Summary

부모와 자식 관계(예: 매니저와 사원, 대분류와 소분류)로 이루어진 트리(Tree) 구조의 데이터를, 뿌리(Root)부터 잎(Leaf)까지 꼬리에 꼬리를 물고 반복해서 탐색하는 쿼리 기법이다.

---
## Why is it needed

`Self Join`은 "내 바로 위 매니저"처럼 딱 1단계만 찾아볼 수 있다. 
하지만 "회장님 ➔ 본부장 ➔ 팀장 ➔ 나"로 이어지는 전체 조직도를 한 번에 쫙 뽑아내거나, 끝을 알 수 없는 10단계 깊이의 카테고리 트리를 훑어야 할 때는, 더 이상 데이터가 안 나올 때까지 쿼리 스스로 **반복(재귀)** 해서 파고들어 가는 계층형 쿼리가 반드시 필요하다.

---
## Practical Context(실무 맥락 )

게시판의 '답글의 답글의 답글', 쇼핑몰의 '의류 > 남성복 > 아우터 > 패딩' 같은 카테고리 네비게이션, 제조업의 부품 명세서(BOM)를 만들 때 필수로 쓴다. 
**SQLD 시험**에서는 Oracle 전용 문법인 `CONNECT BY`가 무조건 출제되지만, 우리가 실무에서 쓰는 **PostgreSQL이나 MySQL**은 표준 문법인 `WITH RECURSIVE`를 쓰기 때문에 두 가지 형태를 모두 비교해서 알아두는 것이 핵심이다!

---
## Code Core Points (1. SQLD 시험용 / Oracle 문법 완벽 해부)

오라클은 계층 쿼리를 위한 전용 키워드와, 진행 상황을 알려주는 마법 같은 **가상 컬럼(Pseudo-column)** 들을 제공해. 
시험에 무조건 나오는 핵심어들을 먼저 뜯어보자.

### 뼈대를 잡는 3대 키워드

- **`START WITH`**: 나무의 뿌리(시작점)를 지정한다. (예: "매니저가 없는 회장님부터 시작해라")
- **`CONNECT BY`**: 떤 가지를 타고 전개할 것인지 '조인 조건'을 적어준다. 
	- 바로 여기서 `PRIOR`의 위치에 따라 **순방향 전개(Top-Down)** 인지 **역방향 전개(Bottom-Up)** 인지가 결정된다!
- **`PRIOR`**: 방향키 역할. `PRIOR`가 붙은 쪽이 '이전 단계(부모)'가 된다.
	- **순방향 전개 (위 ➔ 아래):** `CONNECT BY PRIOR 사원번호 = 매니저번호`
	- (해석: 직전에 찾은 사람의 사번이 다음 사람의 매니저인 경우. 즉, 부모에서 자식으로 내려감)
	- **역방향 전개 (아래 ➔ 위):** `CONNECT BY PRIOR 매니저번호 = 사원번호`
	- (해석: 직전에 찾은 사람의 매니저가 다음 사람의 사번인 경우. 즉, 자식에서 부모로 거슬러 올라감)

### 결과를 풍성하게 만드는 4대 가상 컬럼 (SQLD 단골)

- **`LEVEL`**: 현재 트리의 깊이. 뿌리가 1, 그다음 자식이 2, 손자가 3으로 점점 늘어난다.
- **`SYS_CONNECT_BY_PATH(컬럼명, '구분자')`**: 뿌리부터 현재 행까지 걸어온 경로를 문자열로 쭉 이어 붙여준다. (예: `/회장/부장/사원`)
- **`CONNECT_BY_ROOT 컬럼명`**: 현재 내가 어느 나뭇가지에 있든, 나를 파생시킨 **최상위 뿌리(보스)** 의 데이터를 바로 가져온다.
- **`ONNECT_BY_ISLEAF`**: 내가 이 트리의 마지막 잎사귀(내 밑으로 부하직원이 없는 막내)이면 **1**, 밑에 자식이 더 있으면 **0**을 반환한다.

```sql
SELECT 
    LEVEL AS 계층_깊이, 
    -- 💡 실무 꿀팁: LEVEL을 이용해 앞에 공백을 주면 진짜 조직도처럼 들여쓰기가 된다!
    LPAD(' ', 2 * (LEVEL - 1)) || 사원명 AS 조직도_시각화, 
    
    사원명, 
    매니저번호,
    
    -- 가상 컬럼 활용
    CONNECT_BY_ROOT 사원명 AS 최상위_보스,          -- 누굴 조회하든 '회장님' 이름이 찍힘
    SYS_CONNECT_BY_PATH(사원명, ' ➔ ') AS 결재라인, -- 예:  ➔ 회장 ➔ 부장 ➔ 사원
    CONNECT_BY_ISLEAF AS 막내여부                   -- 부하직원 없으면 1, 있으면 0
FROM 사원
START WITH 매니저번호 IS NULL         -- 1. [시작점] 매니저가 없는 사람(뿌리)부터 탐색 시작!
CONNECT BY PRIOR 사원번호 = 매니저번호;   -- 2. [방향/조건] 직전에 찾은 사람(PRIOR)의 사번이 다음 사람의 매니저번호인 사람을 찾아라!
```

### 계층 쿼리의 정렬: 트리 구조를 지키는 `ORDER SIBLINGS BY`

공들여 만든 트리 구조(조직도) 안에서, 부장님들끼리 또는 사원들끼리 '급여순'이나 '이름순'으로 정렬을 하고 싶을 때가 있다. 
이때 우리가 평소 쓰던 일반적인 `ORDER BY`를 쓰면 절대 안 된다!

**❌ 일반 `ORDER BY` (조직도 파괴):**
계층(부모-자식 관계)이고 뭐고 다 무시하고, 그냥 전체 데이터를 기준으로 일렬로 다시 줄을 세워버린다. 기껏 연결해 둔 회장 ➔ 부장 ➔ 사원 트리가 산산조각 난다.

**⭕️ `ORDER SIBLINGS BY` (조직도 유지):**

영어 단어 뜻 그대로 **'형제(Siblings)들끼리만'** 줄을 세운다. 
즉, 부모-자식 간의 상하 트리 구조는 완벽하게 유지한 채, **같은 부모를 둔 같은 레벨의 데이터끼리만** 내부적으로 정렬하는 마법의 키워드다.

```sql
SELECT 
    LEVEL AS 계층_깊이,
    LPAD(' ', 2 * (LEVEL - 1)) || 사원명 AS 조직도,
    급여
FROM 사원
START WITH 매니저번호 IS NULL
CONNECT BY PRIOR 사원번호 = 매니저번호
-- 💡 [핵심] 일반 ORDER BY 급여 DESC를 쓰면 트리가 깨진다!
-- 반드시 SIBLINGS를 붙여서 "같은 라인(동일 매니저 소속)에 있는 사람끼리만 급여순으로 세워라"라고 지시해야 한다.
ORDER SIBLINGS BY 급여 DESC;
```


**정렬 방식에 따른 결과 비교 (왜 SIBLINGS를 써야 할까?)**

| 구분                              | 정렬 결과                                                                                        |
| ------------------------------- | -------------------------------------------------------------------------------------------- |
| **원래 조직도 (정렬 전)**               | 회장 (9000)➔ 김부장 (5000)  ➔ 박과장 (4000)  ➔ 이대리 (3000)➔ 최부장 (6000)  ➔ 정대리 (3500)                  |
| **❌ ORDER BY 급여 DESC**          | 회장 (9000)최부장 (6000)김부장 (5000)박과장 (4000)정대리 (3500)이대리 (3000)❗ **트리 구조 붕괴**→ 전체가 단순 급여순        |
| **⭕ ORDER SIBLINGS BY 급여 DESC** | 회장 (9000)➔ 최부장 (6000)  ➔ 정대리 (3500)➔ 김부장 (5000)  ➔ 박과장 (4000)  ➔ 이대리 (3000)✅ **형제 노드끼리만 정렬** |

### 표로 보는 계층형 쿼리 작동 원리(헷갈려!)

```sql
SELECT EMPLOYEE_ID, NAME, MANAGER_ID
FROM EMPLOYEES
START WITH MANAGER_ID IS NULL            -- 1. [시작점] 매니저가 없는 Steven(100)부터 시작
CONNECT BY PRIOR EMPLOYEE_ID = MANAGER_ID  -- 2. [전개] 위에서 아래로(순방향) 파고들기
ORDER SIBLINGS BY EMPLOYEE_ID DESC;      -- 3. [정렬] 같은 형제끼리는 사번이 '큰' 사람부터 먼저 출력해라!
```

| 108             | Nancy       | 101            |

## Code Core Points (2. 실무용 / PostgreSQL CTE 재귀)

PostgreSQL은 오라클 같은 전용 키워드가 없는 대신, `WITH` 절을 이용해 쿼리 스스로가 자신을 계속 호출하는 '재귀(Recursive)' 방식을 쓴다. (오라클의 가상 컬럼 기능들은 쿼리 안에서 직접 연산으로 구현해야 해!)

```sql
WITH RECURSIVE 조직도 AS (
    -- 1. [시작점: Anchor] 닻을 내린다. (오라클의 START WITH 역할)
    SELECT 
        사원번호, 
        사원명, 
        매니저번호, 
        1 AS 계층_깊이,                     -- 오라클의 LEVEL 역할 (1부터 시작)
        사원명::text AS 결재라인              -- 오라클의 SYS_CONNECT_BY_PATH 역할 (시작점 세팅)
    FROM 사원
    WHERE 매니저번호 IS NULL
    
    UNION ALL
    
    -- 2. [반복부: Recursive] 꼬리를 물고 계속 찾는다. (오라클의 CONNECT BY 역할)
    SELECT 
        c.사원번호, 
        c.사원명, 
        c.매니저번호, 
        p.계층_깊이 + 1,                    -- 부모 깊이에 1씩 더해줌 (LEVEL 구현)
        p.결재라인 || ' ➔ ' || c.사원명       -- 부모 경로에 내 이름을 계속 이어 붙임 (PATH 구현)
    FROM 사원 c -- c는 자식(Child) 사원
    JOIN 조직도 p  -- p는 방금 찾은 부모(Parent) 사원 (스스로를 JOIN!)
      ON c.매니저번호 = p.사원번호 -- 내 매니저번호가 방금 찾은 윗사람(p)의 사번인 사람 데려오기!
)
SELECT * FROM 조직도;
```

### (PostgreSQL `WITH RECURSIVE` 작동 원리)

1. **시작 (Anchor):** `WHERE 매니저번호 IS NULL`을 실행해서 '회장님' 행을 딱 하나 찾아서 가상의 `조직도` 테이블에 쏙 넣는다. (계층 깊이: 1)
2. **첫 번째 반복:** 이제 `UNION ALL` 밑의 쿼리가 실행된다. 방금 찾은 '회장님(p)'을 매니저로 두고 있는 '부장님들(c)'을 쫙 찾아와서 `조직도` 테이블에 밑으로 착착 이어 붙인다. (계층 깊이: 1 + 1 = 2)
3. **두 번째 반복:** 쿼리가 또 돌아간다! 이번엔 방금 추가된 '부장님들(p)'을 매니저로 두는 '과장님, 대리님들(c)'을 찾아와서 또 밑에 붙인다. (계층 깊이: 2 + 1 = 3)
4. **종료:** 말단 사원까지 다 찾아서, 더 이상 밑에 딸린 직원이 안 나오면(조인 결과가 0건이면) 반복을 멈추고 최종 결과를 우리에게 딱 보여준다.

## 초보자 실수 

**"Oracle `CONNECT BY`에서 `PRIOR` 위치가 맨날 헷갈려요! 어디에 붙여야 하죠?"**
- `PRIOR`는 영어 단어 뜻 그대로 **"직전의, 이전의"** 라는 뜻
- `PRIOR 사원번호 = 매니저번호`: "이전에 찾은 사람의 사번" ➔ "새로 찾을 사람의 매니저"라는 뜻이니, 회장님부터 사원까지 위에서 아래로 파고드는 **순방향(Top-Down)** 탐색이 돼.
- 만약 위치를 바꿔서 `PRIOR 매니저번호 = 사원번호`로 쓰면? 나부터 시작해서 회장님까지 거꾸로 올라가는 **역방향(Bottom-Up)** 탐색
