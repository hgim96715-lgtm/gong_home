## 개념 한 줄 요약 (Concept Summary)

**"공통된 열(Key)을 기준으로 두 개의 테이블을 하나로 합치는 연산이다."**
마치 엑셀의 `VLOOKUP`을 더 강력하게 사용하는 것과 같으며, 집합 이론(벤다이어그램)으로 이해하면 가장 정확하다.

---
## 왜 필요한가? (Why)

**문제점:**
- 데이터베이스는 중복을 피하기 위해 정보를 쪼개서 저장한다. (정규화)
- 예: `고객 정보`는 A 테이블에, `주문 내역`은 B 테이블에 따로 있다.

**해결책:**
- "김철수가 무엇을 주문했는지 보고 싶어"라는 요청이 오면, 따로 떨어진 두 테이블을 **고객 ID(Key)** 를 기준으로 붙여서 하나의 표로 만들어야 한다. 
- 이때 JOIN이 필요하다.

---
##  실전 맥락 (Practical Context)

**데이터 엔지니어의 고민:**
1.  **INNER JOIN:** "주문을 한 적이 있는 **'진성 고객'** 만 뽑아줘." (교집합)
2.  **LEFT JOIN:** "주문을 안 했더라도 **'전체 고객'** 명단을 뽑고, 주문 내역이 있으면 옆에 붙여줘." (전체 집합 + 부분)
3.  **OUTER JOIN:** "매칭되든 안 되든 양쪽 데이터를 싹 다 긁어모아줘." (합집합 - 잘 안 씀)

---
## 상세 분석: 벤다이어그램으로 이해하기 (Detailed Analysis)

두 개의 테이블, **Table A (왼쪽)** 와 **Table B (오른쪽)** 가 있다고 가정하자.

### ① INNER JOIN (교집합) ⭐️ 가장 많이 씀

> **"양쪽 모두에 존재하는 데이터만 남긴다."**

- **벤다이어그램:** 두 원이 겹치는 **가운데 부분**만 색칠.

- **특징:**
    - 짝이 없는 데이터는 가차 없이 버려진다.
    - 결과 행(Row)의 수가 줄어들 수 있다.

- **예시:** `회원` 테이블과 `주문` 테이블을 Inner Join 하면, **"주문 내역이 있는 회원"** 만 남는다. 
- (가입만 하고 주문 안 한 유령 회원은 사라짐)

```sql
SELECT *
FROM Customer A
INNER JOIN Orders B ON A.id = B.customer_id;
```


### ② LEFT (OUTER) JOIN (왼쪽 기준) 

> **"왼쪽 테이블은 무조건 다 살리고, 오른쪽은 있으면 붙이고 없으면 비워둔다."**

- **벤다이어그램:** **왼쪽 원 전체**를 색칠. (겹치는 부분 포함)
    
- **특징:**
    - 기준이 되는 왼쪽 테이블(`FROM`절)의 데이터는 절대 사라지지 않는다.
    - 오른쪽 테이블에 매칭되는 정보가 없으면 그 자리는 **`NULL`** 로 채워진다.

- **예시:** "모든 회원을 보여주되, 주문 내역이 있으면 보여주고 없으면 '없음(NULL)'으로 표시해."

```sql
SELECT *
FROM Customer A  -- A가 주인공(Left)
LEFT JOIN Orders B ON A.id = B.customer_id;
```

### ③ FULL (OUTER) JOIN (합집합)

> **"너랑 나랑 가진 거 다 합치자. 없는 건 빈칸으로 두고."**

- **벤다이어그램:** **두 원 전체**를 색칠.

- **특징:**
    - 양쪽 어디든 데이터가 있으면 결과에 나온다.
    - `MySQL`에서는 지원하지 않아서 `LEFT JOIN`과 `RIGHT JOIN`을 `UNION`해서 쓴다. 
    - (실무 사용 빈도 낮음)

## Code Core Points (문법 공식)

```sql
SELECT
    A.col1,
    B.col2
FROM Table_A as A          -- 1. 왼쪽 테이블 (기준)
<JOIN_TYPE> JOIN Table_B as B  -- 2. 조인 방식 (INNER, LEFT)
    ON A.key = B.key       -- 3. 연결 고리 (가장 중요!)
WHERE
    A.active = true;       -- 4. 필터링은 조인 후에
```

## [심화] JOIN의 연결고리: ID가 없어도 된다?

많은 초보자가 **"JOIN은 무조건 PK(Primary Key)와 FK(Foreign Key)로만 해야 한다"** 고 오해한다. 
하지만 컴퓨터는 그저 **"값이 같니?"** 만 확인한다.

### 연결의 본질: "같은 의미"를 가진 컬럼

- ID가 없어도 **`요일(day)`, `도시(city)`, `카테고리(category)`** 처럼 **양쪽 테이블이 공유하는 "공통된 의미의 값"** 이 있다면 무엇이든 JOIN의 조건(`ON`)이 될 수 있다.
- **예시:**
    - A테이블: `서울` 날씨 정보
    - B테이블: `서울` 매출 정보
    - ➡ 두 테이블을 `지역명(city)`으로 JOIN하면 "날씨에 따른 매출 추이"를 볼 수 있다.

### [패턴] 집계 후 조인 (Aggregate then Join) ⭐️⭐️⭐️

**"원본 데이터도 보고 싶고, 집계된 필터링 조건도 쓰고 싶을 때"** 사용하는 필살기 패턴이다.

- **상황:** "요일별 매출이 1500불 이상인 날(조건)의 **상세 결제 내역(원본)**을 다 보여줘."
- **딜레마:**
    - `GROUP BY`를 하면 상세 내역(결제자 이름 등)이 사라진다.
    - 그냥 조회하면 요일별 합계를 알 수 없다.
- **해결책:**
    1. **CTE(WITH 절)** 로 요일별 통계표(`sum_bill`)를 먼저 만든다. (여기서 1500불 이상만 남김)
    2. **원본 테이블(`tips`)** 과 **통계표(`sum_bill`)** 를 **`요일(day)`** 이라는 **공통 값**으로 JOIN 한다.
    3. 이렇게 하면 살아남은 요일(1500불 이상)에 해당하는 원본 데이터만 출력된다.

```sql
/* 1단계: 필터용 요약표 만들기 (집계) */
WITH sum_bill AS (
    SELECT day, SUM(total_bill) as total
    FROM tips
    GROUP BY day
    HAVING SUM(total_bill) >= 1500
)

/* 2단계: 원본과 요약표를 '요일(day)'로 붙이기 */
SELECT t.* FROM tips t
INNER JOIN sum_bill sb ON t.day = sb.day; -- ID가 아니라 '요일'이 연결고리!
```

## 초보자가 자주 하는 실수 (Misconceptions)

### ① "LEFT JOIN을 했는데 왜 행 개수가 늘어나죠?" 

- **착각:** "왼쪽 테이블이 기준이니까, 왼쪽 행 개수만큼만 나오겠지?"
- **현실:** 오른쪽 테이블에 매칭되는 키가 여러 개라면(예: 한 명이 주문을 10번 함), 왼쪽 데이터가 **복제되어 늘어난다.** (1:N 관계의 뻥튀기 현상)
    
### ② "Inner Join을 썼더니 데이터가 다 사라졌어요."

- **이유:** 연결 고리(`ON A.key = B.key`)가 일치하는 데이터가 하나도 없기 때문이다.
- **해결:** 데이터가 날아가면 안 되는 상황(마스터 데이터 조회 등)이라면 **LEFT JOIN**을 써야 한다.
    

### ③ "NULL 처리를 안 해요."

- **위험:** LEFT JOIN 결과에서 매칭 안 된 `price` 컬럼은 `NULL`이 된다. 이걸 모르고 `SUM(price)`를 하거나 연산을 하면 결과가 이상해질 수 있다. (항상 `COALESCE`나 `IFNULL`을 고려해야 한다.)


>**"교집합만 필요하면 `INNER`, 기준 테이블을 지켜야 하면 `LEFT`."** 
>이 두 가지만 확실히 구분해도 SQL의 90%는 해결된다.


### ④ "PK(기본키)로만 조인해야 하는 줄 알았어요."

- **현실:** JOIN `ON` 절은 **"이 컬럼의 값과 저 컬럼의 값이 같으면 붙여라"** 라는 명령일 뿐이다.
- **활용:** `ON A.date = B.date` 처럼 시간으로 묶을 수도 있고, 심지어 `ON A.score > B.average` 처럼 부등호를 쓸 수도 있다. (Non-Equi Join)
- **핵심:** **"데이터의 맥락(Context)이 같은가?"** 가 중요하다.
