## 개념 한 줄 요약

**"쪼갤 수 없는 업무 처리의 최소 단위. (All or Nothing)"**
(데이터베이스 세상에서는 "하다 말았어"는 통하지 않는다.
**전부 성공하거나(Commit), 아예 없던 일로 하거나(Rollback)** 둘 중 하나뿐이다.)

---
## 왜 필요한가? (Why)

**문제점 (은행 이체 시나리오):**

1. A 계좌에서 100만원을 뺀다. (UPDATE)
2. **⚡️ 전원 차단 / 서버 에러 발생!**
3. B 계좌에 100만원을 넣는 코드는 실행되지 못함.
4. **결과:** A의 돈은 사라지고, B는 돈을 못 받음. (돈이 증발함)

**해결책:**

- 이 두 과정을 **하나의 트랜잭션**으로 묶는다.
- 중간에 에러가 나면 1번 과정(돈 뺀 것)도 **취소(Rollback)** 해서 원상 복구한다.

---
## Code Core Points: TCL 명령어

**TCL (Transaction Control Language)** 은 DML(Insert, Update, Delete)을 확정하거나 취소하는 명령어입니다.

### ① COMMIT (저장)

메모리(Buffer)에만 있던 변경 사항을 **DB(디스크)에 영구적으로 반영**합니다.

- Commit 이후에는 누구도 되돌릴(Rollback) 수 없습니다.
- 다른 사용자들도 변경된 데이터를 조회(SELECT)할 수 있게 됩니다.

### ② ROLLBACK (취소)

마지막 Commit 시점(또는 트랜잭션 시작 시점)으로 **모든 변경 사항을 되돌립니다.**

- 데이터 무결성이 깨졌거나 에러가 났을 때 사용합니다.


### ③ SAVEPOINT (중간 저장 & 부분 취소)

트랜잭션이 너무 길 때, 특정 지점으로 돌아가서 **다시 시작**할 수 있습니다.
핵심은 **"되돌린(Rollback) 후에도 트랜잭션은 끝나지 않고 계속 이어진다"** 는 점입니다.

- 롤백할 때 처음으로 안 돌아가고, 특정 지점까지만 돌아갈 수 있습니다.

```sql
BEGIN TRANSACTION; -- 트랜잭션 시작

-- 1. A 데이터 넣기 (성공)
INSERT INTO table_a VALUES ('A');
SAVEPOINT sp1; -- 🚩 깃발 꽂기 (여기까지 저장점)

-- 2. B 데이터 넣기 (실수!)
INSERT INTO table_a VALUES ('B');

-- 3. 실수 깨닫고 sp1 지점으로 되감기 (B는 취소됨)
-- ⚠️ 주의: 트랜잭션이 끝난 게 아님! A를 넣은 상태로 돌아온 것뿐임.
ROLLBACK TO sp1;

-- 4. C 데이터 넣기 (정상 진행)
-- 되감기 후, 여기서부터 다시 작업을 이어감.
INSERT INTO table_a VALUES ('C');

COMMIT; -- 최종 저장 (쾅!)
```

**[최종 결과 확인]**

| **값** | **저장 여부**   | **이유**                                  |
| ----- | ----------- | --------------------------------------- |
| **A** | **O (저장됨)** | `SAVEPOINT` 이전에 넣었고, 취소되지 않음.           |
| **B** | **X (사라짐)** | `sp1` 이후에 넣었다가 `ROLLBACK TO sp1`으로 취소됨. |
| **C** | **O (저장됨)** | 롤백한 뒤에 새로 넣었고, 마지막에 `COMMIT` 됨.         |
|       |             |                                         |

>**핵심 요약:**
>`ROLLBACK TO`는 **"실행 취소(Ctrl+Z)"** 와 같습니다
>실수를 지우고 나서(`B` 삭제), 그 다음 작업(`C` 입력)을 계속해서 마무리(`COMMIT`)할 수 있습니다.

## 상세 분석: 트랜잭션의 4대 특징 (ACID) ⭐️

|**특징**|**영어(Full)**|**한글**|**설명**|**비유**|
|---|---|---|---|---|
|**A**|**Atomicity**|**원자성**|**All or Nothing.** 더 이상 쪼갤 수 없음. 모두 성공하거나, 모두 실패해야 함.|반반 치킨은 있어도 반반 송금은 없다.|
|**C**|**Consistency**|**일관성**|트랜잭션 전후에 **DB의 규칙(제약조건)이 유지**되어야 함. (예: 잔액은 마이너스가 될 수 없다)|법을 어기면서 송금할 순 없다.|
|**I**|**Isolation**|**고립성**|실행 중인 트랜잭션에 **다른 트랜잭션이 끼어들 수 없음.**|내가 비밀번호 치는 동안 남이 내 키보드를 못 누른다.|
|**D**|**Durability**|**지속성**|성공(Commit)한 데이터는 **서버가 꺼져도 영원히 보존**되어야 함.|입금 확인증 받으면 은행 불나도 내 돈은 있다.|

## 동시성 제어와 격리 수준 (Isolation Level)

여러 사람이 동시에 데이터를 건드리면 생기는 부작용(Side Effect)들입니다. 
SQLD에서는 **Dirty Read**가 가장 자주 나옵니다.

**Dirty Read:**

- A가 데이터를 수정하고 **아직 커밋 안 했는데**, B가 그 값을 읽어버림.
- (나중에 A가 롤백하면 B는 가짜 데이터를 읽은 셈이 됨)

**Non-Repeatable Read:**

- A가 데이터를 읽고 있는데, B가 데이터를 **수정(Update)** 하고 커밋해버림.
- A가 다시 읽으니 값이 달라짐. (반복 읽기 불가능)

1. **Phantom Read:**
    
    - A가 데이터를 읽고 있는데, B가 데이터를 **삽입(Insert)** 해버림.
    - A가 다시 읽으니 아까 없던 유령(Phantom) 데이터가 튀어나옴.

---
### ① "DELETE 했는데 롤백이 안 돼요!" (Auto Commit)

- **Oracle:** 기본적으로 `AUTO COMMIT`이 꺼져 있어서 `DELETE` 후 `ROLLBACK` 하면 살아납니다.
- **SQL Server / MySQL:** 기본적으로 **`AUTO COMMIT`이 켜져 있습니다.** `DELETE` 치자마자 바로 저장됩니다. (설정을 끄거나 `BEGIN TRAN`을 명시해야 함)

### ② "TRUNCATE vs DELETE 차이점 (롤백 유무)" ⭐️

|**구분**|**DELETE**|**TRUNCATE**|**DROP**|
|---|---|---|---|
|**명령어 종류**|**DML** (데이터 조작)|**DDL** (구조 정의)|**DDL** (구조 정의)|
|**Rollback**|**가능 (O)**|**불가능 (X)** ⚡️|**불가능 (X)** ⚡️|
|**속도**|느림 (로그 남김)|빠름 (로그 안 남김)|아주 빠름|
|**내용**|데이터만 지움|데이터 + 저장공간 초기화|테이블 자체를 날림|


>**암기 팁:** "DDL(Create, Drop, Truncate)은 강력해서 실행 즉시 자동 커밋(Auto Commit) 된다. 그래서 롤백이 안 된다."
