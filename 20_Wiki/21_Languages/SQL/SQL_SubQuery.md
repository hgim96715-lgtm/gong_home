# SQL 서브쿼리

## 개념 요약 (Concept Summary)

**"쿼리 안의 쿼리."** 메인 쿼리(Main Query)의 결과를 도출하기 위해, 괄호 `()` 안에 숨어서 먼저 실행되거나 보조 역할을 하는 하위 쿼리를 뜻합니다.

---
## 왜 필요한가? (Why is it needed)

**문제점:** 
"우리 회사 평균 급여보다 돈을 많이 받는 사람을 찾아줘!" 라고 했을 때, 1) 평균 급여를 먼저 구하고, 2) 그 값을 메모해 둔 다음, 3) 다시 쿼리를 짜서 필터링해야 하는 번거로움이 생깁니다.

**해결책:** 
서브쿼리를 쓰면 괄호 안에 `(SELECT AVG(급여) FROM 직원)`을 넣어두어, **DB가 알아서 평균을 먼저 구하고 그 결과를 바탕으로 메인 쿼리를 한 방에 실행**하게 만들어 줍니다.

---
## Code Core Points: 위치에 따른 3가지 분류

서브쿼리는 **"어느 절(Clause)에 위치하느냐"** 에 따라 부르는 이름과 규칙이 완전히 다릅니다.

### 스칼라 서브쿼리 (Scalar Subquery) 👉 `SELECT` 절

- **특징:** 마치 하나의 '컬럼'처럼 동작합니다. **반드시 1개의 행(Row), 1개의 컬럼(Column)** 만 반환해야 합니다.
- **실무 맥락:** 주로 다른 테이블의 특정 값을 엑셀의 `VLOOKUP` 처럼 딱 하나만 땡겨와서 보여줄 때 씁니다.
- **💡 확장의 핵심 (어디든 들어간다!):** 보통 `SELECT` 절에 쓰는 것만 스칼라 서브쿼리라고 오해하기 쉽지만, 사실 **'하나의 값'이 들어갈 수 있는 자리(컬럼명이나 상수 값이 올 수 있는 위치)라면 SQL 문법 어디에든** 들어갈 수 있습니다.

	- **`WHERE` / `HAVING` 절:** 비교할 기준값이 필요할 때 (예: `WHERE 급여 > (SELECT AVG(급여) FROM 직원)`)
	- **`ORDER BY` 절:** 동적인 기준으로 정렬하고 싶을 때 (예: 다른 테이블의 코드명 가나다순 정렬)
	- **`UPDATE` 문의 `SET` 절:** 특정 값을 계산된 결과로 일괄 업데이트할 때

```sql
-- ① 가장 흔한 SELECT 절에서의 활용 (VLOOKUP 스타일)
SELECT 
    직원명,
    급여,
    -- 내 급여 옆에 회사 전체 평균 급여를 나란히 보여주고 싶을 때
    (SELECT AVG(급여) FROM 직원) AS 전체평균급여 
FROM 직원;

-- ② UPDATE 문의 SET 절에서의 활용 (실무 데이터 패치 로직) 
-- 특정 부서의 '최고급여' 컬럼 값을, 실제 직원 테이블을 뒤져서 나온 최댓값으로 업데이트
UPDATE 부서
SET 최고급여 = (
    SELECT MAX(급여) 
    FROM 직원 
    WHERE 직원.부서코드 = 부서.부서코드
)
WHERE 부서코드 = 100;
```

### 인라인 뷰 (Inline View) 👉 `FROM` 절

- **특징:** 마치 하나의 '가상 테이블'처럼 동작합니다. 메모리에 임시로 올려두고 쓰기 때문에 **반드시 별칭(Alias)** 을 지어줘야 합니다.
- **실무 맥락:** 무거운 원본 테이블을 그대로 쓰기 부담스러울 때, 필요한 데이터만 먼저 걸러내서 가벼운 임시 테이블로 만든 뒤 메인 쿼리와 JOIN 할 때 씁니다.

```sql
SELECT 
    e.직원명, 
    t.부서_최대급여
FROM 직원 e
JOIN (
    -- FROM 절 안에서 가상의 테이블을 미리 조립함 (인라인 뷰)
    SELECT 부서코드, MAX(급여) AS 부서_최대급여 
    FROM 직원 
    GROUP BY 부서코드
) t  -- 🚨 인라인 뷰는 무조건 별칭(t)이 필수!
  ON e.부서코드 = t.부서코드;
```

### 중첩 서브쿼리 (Nested Subquery) 👉 `WHERE` / `HAVING` 절

- **특징:** 주로 메인 쿼리의 데이터를 **"필터링(조건 걸기)"** 할 때 씁니다.
- **실무 맥락:** "특정 조건을 만족하는 집단" 안에 속하는지 안 속하는지 찾아낼 때 씁니다. 

```sql
SELECT 직원명, 급여
FROM 직원
WHERE 급여 > (SELECT AVG(급여) FROM 직원); -- 평균 급여보다 큰 사람만 필터링!
```


---
## Detailed Analysis: 중첩 서브쿼리(`WHERE`절) 완벽 해부

`WHERE` 절에 들어가는 중첩 서브쿼리는 다시 **결과값의 형태(반환 행/열 수)** 와 **실행 순서(연관성)** 에 따라 나뉩니다. 
코딩 테스트와 실무 면접에서 가장 헷갈려하는 부분입니다.

### A. 반환 형태에 따른 분류 (단일행 vs 다중행 vs 다중컬럼)

서브쿼리가 뱉어내는 결과물 크기에 따라 쓸 수 있는 연산자가 다릅니다.

####  **단일행 서브쿼리 (Single-Row)**

- 결과가 **무조건 1건**만 나옵니다.
- 사용할 수 있는 연산자: `{text}=`, `>`, `<`, `>=`, `<=`, `<>`

```sql
-- '개발부'의 부서코드(가령 100)는 딱 1개만 나오므로 '=' 연산자 사용 가능
SELECT 직원명 FROM 직원 
WHERE 부서코드 = (SELECT 부서코드 FROM 부서 WHERE 부서명 = '개발부');
```

#### **다중행 서브쿼리 (Multi-Row)**

- 결과가 **여러 건** 나옵니다.(2건 이상의 행을 반환 )
- 사용할 수 있는 연산자: `IN`, `ANY`, `ALL`, `EXISTS` (절대 `{text}=` 를 쓰면 안 됨!)

```sql
-- 연봉 1억 이상인 사람들의 부서코드는 여러 개(100, 200, 300...)일 수 있음.
-- 따라서 '=' 대신 'IN'을 사용해야 에러가 안 남!
SELECT 부서명 FROM 부서 
WHERE 부서코드 IN (SELECT 부서코드 FROM 직원 WHERE 급여 >= 100000000);
```

#### **다중 컬럼 서브쿼리 (Multi-Column)**

- 결과가 **여러 개의 컬럼(세트)** 으로 나옵니다. (주로 Oracle이나 PostgreSQL에서 자주 씀)

```sql
-- (부서코드, 급여) 두 개의 쌍이 완벽하게 일치하는 사람만 필터링
SELECT 직원명, 부서코드, 급여 
FROM 직원 
WHERE (부서코드, 급여) IN (
    SELECT 부서코드, MAX(급여) FROM 직원 GROUP BY 부서코드
);
```

### B. 연관성에 따른 분류 (비연관 vs 연관 서브쿼리)

메인 쿼리와 서브쿼리가 **데이터를 주고받는지**에 대한 차이입니다. (파이썬의 2중 for문 구조를 떠올리면 쉽습니다!)

#### **비연관 서브쿼리 (Uncorrelated)**

- 서브쿼리가 메인 쿼리의 컬럼을 전혀 사용하지 않고, **혼자서 완벽하게 독립적으로 실행**됩니다.
- **실행 순서:** 서브쿼리 실행 (딱 1번) ➡️ 메인 쿼리 실행. 속도가 빠릅니다.

```sql
-- (SELECT AVG(급여) FROM 직원)은 바깥 메인 쿼리와 상관없이 자기 혼자 계산 가능함
SELECT 직원명 FROM 직원 
WHERE 급여 > (SELECT AVG(급여) FROM 직원);
```

#### **연관 서브쿼리 (Correlated)**

- 서브쿼리가 **메인 쿼리의 컬럼 값을 끌고 들어와야만** 실행될 수 있습니다.
- **실행 순서:** 메인 쿼리에서 1줄 읽음 ➡️ 그 값을 서브쿼리에 밀어 넣고 계산 ➡️ 다시 메인 쿼리 1줄 읽음... (마치 파이썬의 중첩 `for`문과 같아서 데이터가 많으면 **속도가 몹시 느려집니다.**)

```sql
SELECT e1.직원명, e1.부서코드, e1.급여
FROM 직원 e1
-- 내(e1) 급여가, 내가 속한 부서(e2)의 평균 급여보다 큰가?
-- 메인 쿼리의 e1.부서코드가 서브쿼리 안으로 침투함! (연관됨)
WHERE e1.급여 > (
    SELECT AVG(e2.급여) 
    FROM 직원 e2 
    WHERE e2.부서코드 = e1.부서코드 
);
```

## 초보자가 자주 하는 실수 (Misconceptions)

**"인라인 뷰(`FROM`절)를 썼는데 자꾸 문법 에러가 나요!"**
- PostgreSQL, MySQL, MSSQL 등 대부분의 모던 DB에서는 `FROM` 절에 서브쿼리를 쓸 때 **반드시 뒤에 `AS t1` 처럼 별칭(Alias)을 강제로 붙여야 합니다.** (Oracle은 안 붙여도 되지만, 표준을 위해 무조건 붙이는 습관을 들이세요!)

**"`ORA-01427: single-row subquery returns more than one row` 에러가 터졌어요!"**
- 초보자들이 가장 많이 겪는 에러 1순위입니다. `WHERE 부서코드 = (SELECT ...)` 처럼 단일행 연산자(`{text}=`)를 썼는데, 괄호 안의 쿼리 결과가 2건 이상 튀어나왔기 때문입니다. 이럴 때는 `{text}=` 를 **`IN`** 으로 슬쩍 바꿔주면 바로 해결됩니다.
