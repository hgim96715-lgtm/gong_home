## 반정규화(De-normalization) 개요

**정의:** 시스템의 성능 향상, 개발 및 운영의 단순화를 위해 **정규화된 데이터 모델을 고의적으로 변형(중복, 통합, 분리)** 하는 기법.
**목표:** **조회(SELECT) 성능 향상** (단, 입력/수정/삭제 성능은 저하될 수 있음)

---
## 테이블 반정규화 (Table De-normalization)

테이블 자체를 합치거나, 쪼개거나, 새로 만드는 3가지 방법이 있습니다.

### ① 테이블 병합 (Merging)

가장 많이 쓰이지만, **1:M 병합**은 주의해야 합니다.

**1:1 관계 병합:**
- 두 테이블을 하나로 합침.
- **효과:** 조인이 사라져서 성능이 좋아짐. 부작용이 거의 없음.

**1:M 관계 병합 (⭐️ 시험 출제 포인트)**
- 부서(1)와 사원(M)을 합치는 경우.
- **문제점:** **데이터 중복 발생!** (부서 정보가 사원 수만큼 계속 반복됨). 저장 공간 낭비 및 데이터 불일치 위험 증가.

**슈퍼/서브 타입 병합:**
- 공통 속성 테이블(슈퍼)과 개별 속성 테이블(서브)을 하나로 합침.

### ② 테이블 분할 (Partitioning)

테이블이 너무 클 때 쪼개는 기술입니다.

**수직 분할 (Vertical Partitioning):**
- **컬럼(Column)** 단위로 쪼갬.
- **결과:** **1:1 관계**의 테이블 2개가 생성됨.⭐️
- **상황:** 특정 컬럼만 자주 조회되고, 사진(BLOB) 같은 무거운 컬럼은 거의 안 쓸 때 별도 테이블로 분리.
- **효과:** I/O 성능 향상 (한 번에 읽는 블록 수 감소).

**수평 분할 (Horizontal Partitioning):**
- **행(Row)** 단위로 쪼갬.
- **결과:** 스키마(컬럼 구조)는 똑같고, 데이터만 나뉜 여러 테이블이 생성됨.
- **상황:** 데이터가 10억 건이라 너무 느림. 연도별(2023, 2024...) 혹은 지역별(서울, 부산...)로 테이블을 나눔.
- **분할 기준 (Partitioning Key):
	- 주로 **기간(Date, 월별/연도별)** 이 가장 많이 쓰이지만,**지역(Region)** 이나 **ID 범위(Range)**, **해시(Hash)** 등 데이터 특성에 따라 **다양한 속성**이 기준이 될 수 있음.
- **다른 말:** 파티셔닝(Partitioning), 샤딩(Sharding).

### ③ 테이블 추가 (Table Addition)

원본은 그대로 두고, **보조용 테이블**을 하나 더 만드는 것입니다.

- **중복 테이블 추가:** 다른 서버/사이트에 배포하기 위해 똑같은 테이블을 하나 더 만듦, 성능상 필요하다고 판단되는 경우 ,원격지 조인(Remote Join)을 피하기 위해 사용함.
- **통계(집계) 테이블 추가:** `SUM`, `AVG` 등을 미리 계산해 둔 테이블. (배치 작업으로 갱신)
- **이력 테이블 추가:** 마스터 테이블의 변경 기록(History)을 관리하는 전용 테이블.(트리거 등으로 자동화)
- **부분 테이블 추가:** 자주 조회되는 컬럼들만 모아서 별도 테이블 생성(디스크 I/O를 줄이는 효과, 인덱스만으로 부족할 때 사용)

>**⚠️ 주의사항 (Critical Point):**
>단순히 성능을 위해 **같은 데이터를 여러 테이블에 중복 저장**하는 것은, 데이터 변경(Update/Delete) 시 모든 테이블을 동시에 갱신하지 않으면 **데이터 정합성(Consistency)에 위배되는 상황**을 발생시킬 수 있습니다.
>(즉, A테이블은 수정됐는데 B테이블은 수정 안 돼서 서로 값이 달라지는 '데이터 불일치' 위험이 큼)

## 컬럼 반정규화 (Column De-normalization)

테이블 구조는 안 바꾸고, 컬럼만 살짝 추가하는 방식입니다. (가장 흔함)

|**유형**|**설명**|**예시**|
|---|---|---|
|**중복 컬럼 추가**|**조인(JOIN)을 줄이기 위해** 관계된 테이블의 컬럼을 복사해옴.|게시판 테이블에 회원 테이블의 `이름` 컬럼 추가|
|**파생 컬럼 추가**|**계산 부하를 줄이기 위해** 미리 계산한 값을 저장.|주문 테이블에 `주문총액`(단가*수량) 컬럼 추가|
|**이력 테이블 컬럼 추가**|**최신 값을 조회하기 위해** 이력 테이블의 최신 날짜/상태를 마스터 테이블에 저장.|회원 테이블에 `최근_로그인_일시` 추가|
|**PK에 의한 컬럼 추가**|복합 PK(예: 부서+사번)인 경우, 단일 속성으로 조회하기 편하게 일반 컬럼으로 추가.|잘 안 쓰임 (개념만 파악)|
|**오류 복구용 컬럼**|사용자의 실수로 인한 데이터 복구를 위해 이전 데이터를 임시 저장.|`이전_주소` 컬럼|

## 관계 반정규화 (Relationship De-normalization) ⭐️

**"데이터 무결성을 깨지 않으면서 성능을 올리는 유일한 방법"** 

- **개념:** 테이블과 컬럼은 건드리지 않고, **관계(FK)** 만 추가하는 것.
- 중복 관계 추가:
	- **상황:** A -> B -> C 순서로 조인해야만 C의 데이터를 볼 수 있음. (경로가 너무 멂)
	- **해결:** A 테이블에서 C 테이블로 바로 갈 수 있는 **지름길(FK)을 추가**함.
	- **특징:** 데이터 중복은 없지만, 관계(FK)가 중복됨. 조인 경로 단축 효과.

## SQLD 핵심 요약

- **반정규화 대상 조사 순서:**
	- 자주 쓰나? -> 대량의 범위인가? -> 통계 정보인가? -> 조인이 너무 많은가?

- **1:M 병합의 치명적 단점:**
	- 데이터 중복 발생 (공간 낭비)
	- 오히려 성능 저하 가능성 (Scan할 데이터 양이 많아져서)

- **관계 반정규화:**
	- 데이터 무결성을 깨지 않는다! (유일하게)
