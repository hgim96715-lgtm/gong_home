# SQL View

## 개념 한 줄 요약 (Concept Summary)

실제 데이터는 하나도 가지고 있지 않으면서, 복잡한 `SELECT` 쿼리 문장을 저장해 두고 마치 **'진짜 테이블'** 인 것처럼 쓸 수 있게 해주는 **"가상 테이블(Virtual Table)"** 입니다.

---
## ⭐️ 뷰(VIEW)의 3대 핵심 특징

**보안성 (Security):**
- 원본 테이블에 있는 주민번호, 연봉 같은 민감한 컬럼을 빼고, 보여주고 싶은 컬럼만 골라서 뷰를 만들 수 있습니다. 
- 사용자에게 원본 테이블 대신 뷰에 대한 접근 권한만 주면 완벽한 보안 통제가 가능합니다.

**독립성 (Logical Data Independence):**
- 원본 테이블의 구조가 바뀌더라도(예: 테이블이 쪼개지거나 컬럼이 추가됨), 뷰를 수정해서 기존과 똑같은 형태의 출력을 유지할 수 있습니다. 
- 즉, 뷰를 바라보고 있는 애플리케이션이나 대시보드는 원본 DB 구조 변화에 영향을 받지 않고 **독립적**으로 돌아갑니다.

**편리성 (Convenience):**
- `JOIN`이 5개씩 걸리고 `GROUP BY`가 들어간 수십 줄짜리 복잡한 쿼리를 뷰로 예쁘게 포장해 두면, 다음부터는 `SELECT * FROM 뷰이름` 단 한 줄로 똑같은 결과를 뽑아낼 수 있습니다.

---
## 왜 필요한가 (Why is it needed)

**문제점:** 매번 조인(JOIN)이 잔뜩 걸린 복잡한 쿼리를 칠 때마다 오타가 나고, 외부인에게 DB를 열어줄 때 데이터 유출 사고의 위험이 큽니다.
**해결책:** 복잡한 로직과 필터링이 완료된 안전한 형태의 쿼리를 `VIEW`라는 이름표를 붙여 DB에 영구 저장해 두고, 필요할 때마다 꺼내 씁니다.

---
## 실전 맥락 (Practical Context) 

데이터 파이프라인을 구축하거나 Streamlit, Tableau 같은 시각화 대시보드를 연결할 때, DB의 원본 테이블을 직접 찌르는 경우는 거의 없습니다. 
대신 분석하기 좋게 잘 정제된(JOIN, 렌더링 완료) 형태의 `VIEW`를 만들어 두고, 대시보드는 이 가상 테이블만 쳐다보게 만드는 것이 현업 데이터 엔지니어들의 **국룰 아키텍처**입니다.

---
## Code Core Points

###  뷰(VIEW) 생성 및 사용하기

- **문법:** `CREATE VIEW 뷰이름 AS (SELECT 쿼리...);`
- **동작 원리:** 데이터(행)를 복사해서 저장하는 것이 아닙니다! `AS` 뒤에 적힌 **'SQL 텍스트(로직)' 자체를 딕셔너리에 저장**하는 것입니다.

```sql
-- 1. 뷰 생성하기 (민감한 연봉, 주민번호는 빼고 조회용 뷰 만들기)
CREATE VIEW v_emp_public_info AS
SELECT 
    e.사원번호, 
    e.직원명, 
    d.부서명 
FROM 직원 e
JOIN 부서 d ON e.부서코드 = d.부서코드;
-- 🚨 주의: 이 순간 데이터가 복사되는 게 아니라, 위 5줄의 텍스트만 저장됨!

-- 2. 뷰 사용하기 (마치 진짜 테이블인 것처럼 쓴다!)
SELECT * FROM v_emp_public_info 
WHERE 부서명 = '데이터엔지니어링팀';
```

### 뷰(VIEW) 수정 및 삭제

- 뷰는 본질이 '쿼리 텍스트'이므로 데이터를 조작하듯 `UPDATE` 하는 것이 아니라, 구조 자체를 다시 덮어씌우거나 날려버립니다.

```sql
-- 1. 뷰 덮어씌우기 (OR REPLACE를 쓰면 기존 뷰를 안전하게 갈아끼울 수 있음)
CREATE OR REPLACE VIEW v_emp_public_info AS
SELECT 사원번호, 직원명, 부서명, 입사일 -- (입사일 컬럼 추가)
FROM ... ;

-- 2. 뷰 삭제하기 (원본 테이블의 데이터는 절대 지워지지 않음! 껍데기만 삭제됨)
DROP VIEW v_emp_public_info;
```

## 가상 테이블 3대장 비교 (VIEW vs 인라인 뷰 vs WITH CTE)

목적은 같지만 **"얼마나 오래 보존되는가(생명 주기)"** 에 따라 완벽히 구분됩니다.

|**구분**|**뷰 (VIEW)**|**WITH 구문 (CTE)**|**인라인 뷰 (Inline View)**|
|---|---|---|---|
|**정의**|DB에 영구적으로 저장된 쿼리 객체|쿼리 맨 위에서 임시로 선언하는 변수 같은 블록|`FROM` 절 괄호 안에서 즉석으로 만드는 가상 테이블|
|**생성 키워드**|`CREATE VIEW 뷰이름 AS`|`WITH 임시테이블명 AS`|`FROM (SELECT ...) AS 별칭`|
|**수명 (Lifespan)**|**내가 `DROP` 하기 전까지 영구적**|**해당 `SELECT` 문이 끝나는 순간 소멸**|**해당 `SELECT` 문이 끝나는 순간 소멸**|
|**재사용성**|내일도, 다른 팀원도 재사용 가능|오직 그 쿼리 스크립트 안에서만 재사용 가능|재사용 불가 (딱 그 자리에서 1회용)|
|**비유**|**법원 호적에 정식 등록된 이름**|**이번 회의 때만 잠깐 부를 별명 (포스트잇)**|**지나가던 행인 1 (이름 없음)**|

## 상세 분석 (Detailed Analysis)

뷰를 호출(`SELECT * FROM 뷰`)하면 DB 내부에서는 어떤 일이 일어날까요?

1. 사용자가 뷰를 조회합니다.
2.  DB는 딕셔너리에 저장해 둔 "뷰의 원본 SQL 텍스트"를 꺼내옵니다.
3.  사용자가 쓴 `WHERE` 조건 등을 원본 SQL과 하나로 짬뽕(Merge)합니다.
4. 그제야 실제 원본 테이블로 달려가 데이터를 긁어옵니다.

>👉 즉, 원본 테이블의 데이터가 바뀌면 **뷰의 결과도 실시간으로 똑같이 바뀝니다.** 거울(Mirror)과 똑같습니다!

---
## 초보자가 자주 하는 실수 (Misconceptions)

 **"인라인 뷰(Inline View)랑 그냥 뷰(VIEW)는 뭐가 다른가요?"**
- **인라인 뷰:** 쿼리의 `FROM` 절 안에 괄호 `()` 로 임시로 만든 가상 테이블. 쿼리가 끝나면 메모리에서 펑 사라집니다. (1회용 종이컵)
- **뷰(VIEW):** `CREATE VIEW` 명령어로 DB에 이름을 붙여 저장해 둔 가상 테이블. 내가 `DROP` 하기 전까지 계속 쓸 수 있습니다. (머그컵)

 **"뷰를 통해서 원본 데이터를 수정(`INSERT`, `UPDATE`)할 수 있나요?"**
- 문법적으로 **가능은 합니다.** 하지만 실무에서는 거의 절대 하지 않습니다.
- 뷰에 `GROUP BY`, `JOIN`, `DISTINCT` 등이 섞여 있으면 어떤 원본 테이블의 어느 행을 고쳐야 할지 DB가 판단할 수 없어서 십중팔구 에러가 납니다. 뷰는 **'읽기 전용(Read-Only)'** 으로 생각하는 것이 마음 편합니다.

 **"뷰는 매번 쿼리를 새로 실행하니까 속도가 느린가요?" (실무 심화 - Materialized View)**
-  맞습니다. 뷰는 매번 쿼리를 다시 돌리기 때문에, 연산이 무거우면 성능이 떨어집니다.
- 그래서 데이터 엔지니어들은 진짜로 결과 데이터를 스냅샷처럼 디스크에 콱 찍어서 저장해 두는 **MVIEW (구체화된 뷰, Materialized View)** 라는 기술을 씁니다. 배치(Batch) 작업으로 매일 새벽에 MVIEW를 갱신해 두고, 낮에는 그 스냅샷을 빠르게 읽기만 하는 패턴이 아주 유명합니다.
