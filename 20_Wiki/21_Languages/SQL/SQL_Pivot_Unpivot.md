# 행을 열로, 열을 행으로 (데이터 형태 변환)

## Concept Summary

`PIVOT`은 길게 세로로 쌓인 행(Row) 데이터를 가로 열(Column)로 넓게 펼쳐서 요약하는 기능이고, 
`UNPIVOT`은 반대로 가로로 퍼진 열 데이터를 다시 세로 행으로 해체하는 기능이다.

---
## Why is it needed(왜 필요한가)?

데이터베이스에는 데이터가 발생한 순서대로 세로로 길게(Long format) 쌓이지만, 사람이 보고서를 읽거나 엑셀로 분석할 때는 "가로축은 월별, 세로축은 부서별" 같은 넓은 표(Wide format)가 훨씬 직관적이기 때문에 형태를 자유자재로 변환해야 한다.

---
## Practical Context(실무맥락)

**SQLD 시험**에서는 오라클의 `PIVOT`과 `UNPIVOT` 문법의 각 구성요소(집계함수, FOR, IN)가 정확히 어떤 역할을 하는지 묻는 문제가 단골로 출제된다. 하지만 실무에서 많이 쓰는 **PostgreSQL**에는 해당 연산자가 없기 때문에, `FILTER` 절이나 `UNION ALL`을 사용해 수동으로 피벗/언피벗을 구현하는 방법을 반드시 알아야 한다.

---
## Code Core Points 1: PIVOT (행 ➔ 열로 눕히기)

세로로 긴 원본 테이블을 가로로 요약할 때 쓴다.

### **1. [SQLD 시험용 / Oracle] `PIVOT` 연산자 구성요소 및 별칭(Alias) 규칙 해부**

무엇을 계산할지, 어떤 값을 눕힐지가 핵심이다. 빈칸 뚫기 문제로 자주 나온다.

- **`집계함수`**: 피벗은 데이터를 압축하는 과정이므로 `SUM`, `COUNT`, `MAX` 같은 집계함수가 반드시 필요하다.
	- **[별칭 규칙 1]** 집계함수에 부여한 별칭은 최종 헤더명의 **맨 뒤(접미사)** 에 붙는다.
- **`FOR`**: "어떤 컬럼을 쪼갤 건데?" (예: '월' 컬럼을 쪼개겠다)
- **`IN`**: "그 컬럼 안에서 구체적으로 어떤 값들을 가로로 눕힐 건데?" (예: '1월', '2월' 데이터만 뽑아서 가로로 눕히겠다)
	- **[별칭 규칙 2]** IN 절 안에 부여한 별칭은 최종 헤더명의 **맨 앞(접두사)** 에 붙는다.
	- **[별칭 규칙 3]** 두 별칭은 컴퓨터가 알아서 **언더바(`_`)** 로 연결해 준다.

```sql
SELECT * FROM 부서별매출
PIVOT (
    SUM(매출) AS SAL          -- [1. 집계함수 별칭] ➔ 헤더명 맨 '뒤'에 붙음
    FOR 월                    -- [2. FOR 절] 기준 컬럼
    IN ('1월' AS JAN,         -- [3. IN 절 별칭] ➔ 헤더명 맨 '앞'에 붙음
        '2월' AS FEB)
);
```


| **부서명** | **JAN_SAL** | **FEB_SAL** | **💡 규칙 해석**                       |
| ------- | ----------- | ----------- | ---------------------------------- |
| 영업부     | 100         | 200         | `JAN`(IN절 별칭) + `_` + `SAL`(함수 별칭) |
| 마케팅부    | 300         | 400         | `FEB`(IN절 별칭) + `_` + `SAL`(함수 별칭) |


### **[실무용 / PostgreSQL] 집계함수 + `FILTER` 절 (최신 문법)**

PostgreSQL에는 `PIVOT` 명령어와 저런 복잡한 별칭 자동 합성 규칙이 없다. 내가 `AS` 뒤에 적어준 이름을 100% 그대로 직관적으로 사용한다.
대신 최신 SQL 표준인 `FILTER`를 쓰면 오라클보다 훨씬 직관적이고 깔끔하게 피벗을 구현할 수 있다.

🔗 **더 자세한 내용은 [[SQL_CASE_WHEN#PostgreSQL 전용 `FILTER` 구문|PostgreSQL-FILTER]] 참조**

```sql
-- "부서별로 묶어서(GROUP BY), 1월과 2월 데이터만 필터링해 가로로 펼쳐라!"
SELECT 
    부서명, 
    SUM(매출) FILTER (WHERE 월 = '1월') AS "1월_매출",
    SUM(매출) FILTER (WHERE 월 = '2월') AS "2월_매출"
FROM 부서별매출
GROUP BY 부서명;  -- 💡 PostgreSQL에서는 집계의 기준이 될 컬럼을 반드시 명시해야 함!
```

## Code Core Points 2: UNPIVOT (열 ➔ 행으로 세우기)

가로로 넓은 테이블을 다시 세로로 해체할 때 쓴다.

### ** [SQLD 시험용 / Oracle] `UNPIVOT` 연산자 구성요소 해부** 

집계를 하는 게 아니라 그냥 해체만 하므로 **집계함수가 없다**는 것이 PIVOT과의 가장 큰 차이다!

- **`INCLUDE NULLS` (선택 옵션)**: 가로 표에 빈칸(NULL)이 있을 때, 기본적으로 UNPIVOT은 그 데이터를 아예 결과에서 날려버린다(`EXCLUDE NULLS`가 디폴트). 빈칸이 있는 행도 살려서 세로로 출력하고 싶다면 이 키워드를 반드시 적어주어야 한다.
- **`가공된_값_컬럼명`**: 흩어져 있던 숫자들을 모을 바구니 이름. (집계함수 아님!)
- **`FOR`**: 열 이름표였던 애들을 데이터로 강등시켜서 담을 바구니 이름.
- **`IN`**: "어떤 열 이름표들을 해체할 건데?"(세로로 해체할 기존 열(Column) 이름들.)
	- `IN (기존컬럼명 AS '새로운값')` 형태로 별칭을 쓰면, `FOR` 절 바구니에 담길 **'데이터의 실제 텍스트 값'을 예쁘게 바꿀 수 있다!** (PIVOT이 컬럼명을 바꾸는 거였다면, UNPIVOT은 데이터 값을 바꾼다!)

```sql
SELECT * FROM 피벗된_테이블
UNPIVOT INCLUDE NULLS (    -- 💡 [디테일 1] 원래 표에 값이 NULL이어도 행을 날리지 말고 살려라!
    매출액                 -- [1. 가공된_값_컬럼명] 숫자들이 모일 새 컬럼명
    FOR 해당월             -- [2. FOR 절] 기존 헤더명이 데이터로 담길 새 컬럼명
    
    -- 💡 [디테일 2] IN 절에서 AS를 쓰면, '해당월' 컬럼 안에 들어갈 글자를 내 맘대로 바꿀 수 있다!
    IN (
        "1월_매출" AS '1월', -- "1월_매출" 컬럼을 찢어서, 값은 '매출액'에 넣고, 글자는 '1월'이라고 적어라!
        "2월_매출" AS '2월'
    ) 
);
```

**UNPIVOT 별칭(`AS`) 작동 원리 표**

|**부서명**|**해당월**|**매출액**|**💡 원리 해석**|
|---|---|---|---|
|영업부|**1월**|100|원본 컬럼명(`"1월_매출"`) 대신 **`AS '1월'`**로 지정한 값이 들어감!|
|영업부|**2월**|NULL|원래 빈칸(NULL)이었지만, **`INCLUDE NULLS`** 옵션 덕분에 행이 삭제되지 않고 살아남음!|

### [실무용 / PostgreSQL] `UNION ALL`을 이용한 수동 해체

PostgreSQL에서 가로 데이터를 세로로 돌릴 때는, 각 컬럼을 분리해서 조회한 뒤 `UNION ALL`로 밑으로 차곡차곡 이어 붙이는 방식을 가장 많이 쓴다.

```sql
-- 1. [첫 번째 표: 1월 데이터] 
-- 💡 하드코딩 마법: 오라클이 하던 것처럼, 우리가 직접 '1월'이라는 글자를 만들어서 '해당월'이라는 바구니에 담아준다!
SELECT 
    부서명, 
    '1월' AS 해당월,       -- 텍스트 값을 직접 만들고 별칭 부여
    "1월_매출" AS 매출액   -- 찢어낸 컬럼의 값에 새 이름표 부여
FROM 피벗된_테이블

UNION ALL 

-- 2. [두 번째 표: 2월 데이터 밑으로 이어 붙이기]
SELECT 
    부서명, 
    '2월' AS 해당월, 
    "2월_매출" AS 매출액 
FROM 피벗된_테이블;
```

>**`UNION ALL`의 헤더명(컬럼명) 절대 규칙** `UNION ALL`로 여러 개의 쿼리를 위아래로 합칠 때, 최종 결과물의 **헤더명(컬럼명)은 무조건 '가장 위에 있는 첫 번째 쿼리'의 별칭(Alias)만을 따라간다!** 
> 즉, 두 번째, 세 번째 쿼리에서 `AS 엉뚱한이름`이라고 적어도 컴퓨터는 쿨하게 무시해 버린다. 따라서 수동 언피벗을 할 때는 **첫 번째 쿼리에 `AS 해당월`, `AS 매출액`을 정확하게 달아주는 것**이 생명이다!
>
>`UNION ALL`은 중복이 있든 말든 있는 그대로 100% 싹 다 이어 붙여주기 때문에, 실무에서는 특별한 이유가 없으면 속도와 정확성을 위해 무조건 `UNION ALL`을 쓰는 게 정배야!  
>🔗 **더 자세한 내용은 [[SQL_UNION#1. `UNION ALL` 묻지도 따지지도 않고 다 붙이기 (가장 빠름)|UNION ALL]] 참조**

## Detailed Analysis (PIVOT vs UNPIVOT 문법 완벽 비교)

| **구분**                | **PIVOT (행 ➔ 열 눕히기)**                                       | **UNPIVOT (열 ➔ 행 세우기)**                                                             |
| --------------------- | ----------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| **핵심 목적**             | 데이터를 요약하여 엑셀처럼 가로로 펼침                                       | 가로 데이터를 다시 DB에 넣기 좋게 세로로 해체함                                                        |
| **`집계함수`의 유무**        | ⭕️ **필수** (`SUM`, `MAX` 등)<br><br>데이터를 압축해야 하므로 꼭 필요함.      | ❌ **없음** (새로운 컬럼명 명시)<br><br>해체만 하므로 집계가 필요 없음.                                     |
| **오라클 별칭(`AS`) 규칙**   | **`IN별칭_함수별칭`** 형태로 **새로운 헤더명(컬럼명)**을 강제 합성함 (예: `JAN_SAL`) | `IN (원래컬럼 AS '새값')` 형태로 **실제 데이터(텍스트) 값**을 내 입맛대로 변경함                               |
| **`FOR` 절의 의미**       | 가로 **열 이름**의 기준이 될 **기존 컬럼**                                | 기존 열 이름들을 데이터로 담을 **새로운 컬럼명**                                                       |
| **`IN` 절의 의미**        | 가로로 눕힐 **실제 데이터 값들**                                        | 세로로 해체할 **기존 열(Column) 이름들**                                                        |
| **`NULL` 처리 방식** (중요) | 매칭되는 데이터가 없으면 빈칸을 자동으로 `NULL`로 채워서 보여줌                      | 🚨 기본적으로 `NULL`인 데이터(행)는 **결과에서 삭제(누락)시켜 버림.**<br><br>(살리려면 `INCLUDE NULLS` 옵션 필수!) |

## 초보자 실수

**"PIVOT 했던 테이블을 그대로 UNPIVOT 하면, 완벽하게 옛날 원본 데이터로 돌아가나요?"**
- **절대 아니다! (매우 중요)**
- 1월 매출이 원래 (50원, 50원) 2건이었는데 피벗하면서 `SUM`을 때려 100원이 되었다고 치자. 이걸 나중에 `UNPIVOT` 한다고 해서 다시 50원짜리 2건으로 인공지능처럼 분리해 주지는 않는다. 그냥 100원짜리 1건으로 세로로 내려올 뿐이다. 즉, **PIVOT 과정에서 정보의 압축(손실)이 일어났기 때문에 완벽한 복구는 불가능하다!**
