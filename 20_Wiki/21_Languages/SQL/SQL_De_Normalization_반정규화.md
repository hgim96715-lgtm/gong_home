# De-Normalization: 성능을 위해 다시 합치는 기술

> **핵심 요약** **"조회(SELECT) 속도를 극대화하기 위해, 정규화 원칙을 깨고 고의적으로 데이터 중복을 허용하는 기법"** 
> 정규화가 '정리정돈'이라면, 반정규화는 '자주 쓰는 물건을 책상 위에 꺼내놓는 것'과 같다.

> ⚠️ **선 정규화, 후 반정규화** — 이 순서는 절대 바꾸지 않는다.

---
## 정규화 vs 반정규화 — 목적이 다르다

> 💡 정규화와 반정규화는 서로 반대되는 개념이 아니라, **상황에 따라 전략적으로 선택**하는 것입니다.

|구분|**정규화 수행 이유**|**반정규화 수행 이유**|
|:--|:--|:--|
|**핵심 목적**|데이터 **무결성** 확보|조회 **성능** 향상|
|**언제 하나?**|설계 초기, 중복/이상 현상이 있을 때|정규화 이후, 조회가 너무 느릴 때|
|**해결하는 문제**|삽입/갱신/삭제 이상 현상|과도한 JOIN으로 인한 SELECT 성능 저하|
|**데이터 중복**|중복 **제거**|중복을 **허용** (의도적으로)|
|**저장 공간**|절약됨|증가함|
|**쓰기 성능**|✅ 향상 (한 곳만 수정)|❌ 저하 (여러 곳 동시 수정 필요)|
|**읽기 성능**|❌ 저하 가능 (JOIN 증가)|✅ 향상 (JOIN 감소)|
|**위험 요소**|없음 (정석 설계)|데이터 불일치(갱신 이상) 재발 위험|
|**적용 기준**|기본 원칙 (항상 먼저 수행)|성능 테스트 후 **필요한 경우에만** 수행|

>💡 정규화와 반정규화는 서로 반대되는 개념이 아니라, **상황에 따라 전략적으로 선택**하는 것입니다. 
>📎 정규화(1NF/2NF/3NF) 개념이 궁금하다면 → **[[SQL_Normalization_정규화]]** 참고

## 반정규화 적용 순서 (실무 가이드)

무조건 반정규화를 하는 게 아닙니다. 다음 순서를 따르세요.

```
1단계: 정규화를 충실히 수행한다 (기본 원칙)
         ↓
2단계: JOIN이 너무 많거나 대용량 데이터라서 조회가 느리다
         ↓
3단계: 인덱스(Index) 튜닝으로 먼저 해결을 시도한다
         ↓
4단계: 그래도 느리면 → 최후의 수단으로 반정규화 적용
```

##  테이블 반정규화 (Table De-normalization)

테이블 자체를 합치거나, 쪼개거나, 새로 만드는 3가지 방법입니다.

---
## ① 테이블 병합 (Merging)

| 기법              | 설명                            | 효과 / 주의                                         |
| :-------------- | :---------------------------- | :---------------------------------------------- |
| **1:1 관계 병합**   | 항상 같이 조회되는 두 테이블을 하나로 합침      | 조인 제거로 성능 향상. 부작용 거의 없음                         |
| **1:N 관계 병합** ⭐ | 부모(1)와 자식(N)을 합치는 경우          | ❌ **데이터 중복 발생!** 부모 정보가 자식 수만큼 반복. 오히려 성능 저하 가능 |
| **슈퍼/서브 타입 병합** | 공통 속성(슈퍼) + 개별 속성(서브)을 하나로 합침 | 조회 단순화. 단, NULL 컬럼 증가                           |

>💡 **슈퍼/서브 타입 설계가 적절한 이유** **공통 속성(슈퍼)을 조회하는 빈도**가 **개별 속성(서브)을 조회하는 빈도**보다 높기 때문입니다.
>예를 들어 `직원 전체 목록`은 자주 조회하지만, `정규직만의 특수 정보`나 `계약직만의 특수 정보`는 상대적으로 드물게 조회됩니다. 
>따라서 공통 속성은 슈퍼 타입에, 개별 속성은 서브 타입에 분리해두는 구조가 **조회 효율과 설계 명확성** 측면에서 적절합니다.

```text
[슈퍼] 직원 (사번, 이름, 입사일)     ← 자주 조회 ✅
   ├── [서브] 정규직 (연봉, 부서)     ← 가끔 조회
   └── [서브] 계약직 (계약기간, 단가) ← 가끔 조회
```

>⭐ **시험 출제 포인트 — 1:N 병합의 함정** 부서(1) - 사원(N)을 합치면 부서 정보(이름, 위치 등)가 사원 수만큼 중복됩니다. 
>저장 공간 낭비 + 데이터 불일치 위험 증가 → **대부분 하면 안 되는 케이스!**


---
## ② 테이블 분할 (Partitioning)

테이블이 너무 클 때 쪼개는 기술입니다.

| 기법                     | 기준                    | 결과                      | 상황                                                                                                 |
| ---------------------- | --------------------- | ----------------------- | -------------------------------------------------------------------------------------------------- |
| **수직 분할** (Vertical)   | **컬럼(Column)** 단위로 쪼갬 | **1:1 관계** 테이블 2개 생성 ⭐  | ① 사진(BLOB) 같은 무거운 컬럼이 거의 안 쓰일 때 별도 분리  <br>② **한 테이블의 속성 개수가 과도하게 많을 때** → 자주 쓰는 컬럼만 분리하여 조회 성능 향상 |
| **수평 분할** (Horizontal) | **행(Row)** 단위로 쪼갬     | 스키마는 동일, 데이터만 나뉜 여러 테이블 | 데이터가 10억 건이라 너무 느릴 때 연도별/지역별로 분리                                                                   |


**수평 분할 기준 (Partitioning Key):** 주로 **기간(월별/연도별)** 이 가장 많이 쓰이지만, 지역(Region), ID 범위(Range), 해시(Hash) 등 데이터 특성에 따라 다양한 속성이 기준이 될 수 있습니다.

> 수평 분할 = **파티셔닝(Partitioning)**, **샤딩(Sharding)** 이라고도 부릅니다.

---

## ③ 테이블 추가 (Table Addition)

원본은 그대로 두고, **보조용 테이블**을 하나 더 만드는 것입니다.

|기법|설명|예시|
|:--|:--|:--|
|**중복 테이블 추가**|다른 서버/원격지 조인(Remote Join)을 피하기 위해 동일한 테이블을 복사|서울/부산 서버에 각각 동일한 코드 테이블 배포|
|**통계(집계) 테이블 추가**|`SUM`, `AVG` 등을 미리 계산해 둔 테이블 (배치로 갱신)|일별 주문 통계 테이블|
|**이력 테이블 추가**|마스터 테이블의 변경 기록(History) 전용 테이블|가격 변경 이력, 상태 변경 이력|
|**부분 테이블 추가**|자주 조회되는 컬럼들만 모아서 별도 테이블 생성|상품 목록용 요약 테이블 (디스크 I/O 감소)|

## 컬럼 반정규화 (Column De-normalization)

테이블 구조는 안 바꾸고, **컬럼만 추가**하는 방식입니다. 가장 흔하게 쓰입니다.

|유형|설명|예시|
|:--|:--|:--|
|**중복 컬럼 추가**|JOIN을 줄이기 위해 관계된 테이블의 컬럼을 복사해옴|게시판 테이블에 회원 테이블의 `이름` 컬럼 추가|
|**파생 컬럼 추가**|계산 부하를 줄이기 위해 미리 계산한 값을 저장|주문 테이블에 `주문총액`(단가×수량) 컬럼 추가|
|**이력 컬럼 추가**|최신 값을 빠르게 조회하기 위해 최신 날짜/상태를 마스터에 저장|회원 테이블에 `최근_로그인_일시` 추가|
|**PK에 의한 컬럼 추가**|복합 PK인 경우, 단일 속성으로 조회하기 편하게 일반 컬럼으로 추가|잘 안 쓰임 (개념만 파악)|
|**오류 복구용 컬럼**|사용자 실수로 인한 데이터 복구를 위해 이전 데이터를 임시 저장|`이전_주소` 컬럼|

> ⚠️ **위험성 예시:** 유저가 닉네임을 바꿨다.
> 
> - **정규화:** `User` 테이블 1곳만 수정하면 끝.
> - **반정규화:** `Board`, `Order`, `Review` 등 닉네임을 복사해둔 **모든 테이블**을 다 찾아서 수정해야 함. 하나라도 빠뜨리면 → **데이터 불일치 버그 발생!**

##  관계 반정규화 (Relationship De-normalization) ⭐

> **"데이터 무결성을 깨지 않으면서 성능을 올리는 유일한 방법"**

- **개념:** 테이블과 컬럼은 건드리지 않고, **관계(FK)만 추가**하는 것.
- **중복 관계 추가:**
    - **상황:** A → B → C 순서로 조인해야만 C의 데이터를 볼 수 있음. (경로가 너무 멂)
    - **해결:** A 테이블에서 C 테이블로 바로 갈 수 있는 **지름길(FK)을 추가**함.
    - **특징:** 데이터 중복은 없고, 관계(FK)만 중복됨. 조인 경로 단축 효과.

```
[기존]  A → B → C  (2번 조인 필요)
[추가]  A --------→ C  (지름길 FK 추가, 1번 조인으로 해결)
```

> ✅ **관계 반정규화는 데이터 중복이 없어서 데이터 무결성을 깨지 않습니다!** 반정규화 기법 중 유일하게 부작용이 거의 없는 방법입니다.

##  Trade-off 요약

|작업|정규화 후|반정규화 후|
|:--|:-:|:-:|
|**INSERT / UPDATE / DELETE**|✅ 빠름 (한 곳만 수정)|❌ 느림 (여러 곳 수정)|
|**SELECT (단순 조회)**|보통|✅ 빠름|
|**SELECT (복잡한 JOIN)**|❌ 느림|✅ 빠름|
|**저장 공간**|✅ 절약|❌ 증가|
|**데이터 일치성**|✅ 항상 일치|⚠️ 관리 필요|

> 대부분의 서비스는 **읽기(조회): 쓰기 = 8:2 비율**로 조회가 압도적으로 많습니다. 그래서 정규화를 원칙(3NF)대로 하되, **조회 성능이 실제로 문제가 될 때만 전략적으로 반정규화**를 수행합니다.

## SQLD 핵심 요약 

**반정규화 대상 조사 순서:** 자주 쓰나? → 대량의 범위인가? → 통계 정보인가? → 조인이 너무 많은가?

**1:M 병합의 치명적 단점:**

- 데이터 중복 발생 (공간 낭비)
- Scan할 데이터 양이 많아져서 오히려 성능 저하 가능성

**관계 반정규화의 핵심:**

- 유일하게 **데이터 무결성을 깨지 않는** 반정규화 기법

**초보자 실수 3가지:**

|오해|진실|
|:--|:--|
|"반정규화는 나쁜 건가요?"|아닙니다. **성능 최적화의 핵심 기술**입니다. 쇼핑몰 상품 목록, 뉴스 피드 등에서 필수입니다.|
|"처음부터 반정규화해서 설계하면 안 되나요?"|**안 됩니다.** 처음부터 중복을 허용하면 나중에 어디가 원본인지 모르게 됩니다.|
|"JSON 컬럼을 쓰는 것도 반정규화인가요?"|**맞습니다.** 별도 테이블로 쪼개야 할 속성들을 하나의 JSON 컬럼에 몰아넣는 것도 반정규화입니다.|
