# Self Join : 하나의 테이블을 두 개인 것처럼 엮기

## Concept Summary

하나의 테이블을 마치 완전히 다른 두 개의 테이블인 것처럼 별칭(Alias)을 부여하여, 자기 자신과 결합(JOIN)하는 문법 기법

---
## 왜 필요한가 Why is it needed)?

**문제점:** 
일반적으로 JOIN은 '사원 테이블'과 '부서 테이블'처럼 서로 다른 두 테이블의 정보를 합칠 때 쓴다. 하지만 "이 사원의 담당 매니저 이름은 무엇인가?"처럼, **사원도 매니저도 모두 '같은 테이블(사원 테이블)'에 들어있는 경우**에는 단순히 다른 테이블을 가져올 수가 없다. 

**해결책:** 
사원 테이블을 복사해서 '사원용 테이블'과 '매니저용 테이블' 2개로 나눈다고 상상한 뒤, 이 둘을 JOIN 시키면 내 정보 옆에 매니저의 정보를 나란히 붙일 수 있다.)

---
## Practical Context(실무맥락)

실무에서는 사원-매니저 관계, 카테고리 대분류-소분류 관계 등 **'계층형 구조(조직도)'** 를 파악할 때 가장 많이 사용한다.  
또한 윈도우 함수(`LAG`, `LEAD`)가 익숙하지 않거나 지원되지 않는 과거 환경에서 "어제 매출과 오늘 매출 비교", "나보다 월급 많이 받는 직장 동료 찾기" 같은 상대적 비교 연산을 할 때도 요긴하게 쓰인다. SQLD 시험에서는 개념과 결과 행 수를 묻는 문제로 자주 출제된다.

```sql
SELECT 
    e1.사원번호 AS 내_사번,
    e1.사원명 AS 내_이름,
    e1.매니저번호 AS 내_담당_매니저사번,
    e2.사원명 AS 매니저_이름  -- 매니저 테이블(e2)에서 이름을 뽑아온다!
FROM 사원 e1            -- 1. [사원용 기준 테이블] 별칭을 e1으로 준다.
LEFT JOIN 사원 e2       -- 2. [매니저용 참조 테이블] 똑같은 테이블이지만 별칭을 e2로 준다.
    ON e1.매니저번호 = e2.사원번호; -- 3. [연결 고리] 내(e1) 매니저 번호와 일치하는 사번(e2)을 찾아라!
```

### Detailed Analysis (컴퓨터의 해석 흐름)

`FROM 사원 e1`: 일단 원본 '사원' 테이블을 복사해서 `e1`이라는 이름표를 붙인다. 이건 **'나(사원 본인)'** 를 기준으로 볼 테이블

`LEFT JOIN 사원 e2`: 원본 '사원' 테이블을 한 번 더 복사해서 이번엔 `e2`라는 이름표를 붙인다. 이건 내 **'매니저 정보'** 를 찾기 위한 카탈로그용 테이블

`ON e1.매니저번호 = e2.사원번호`: `e1` 테이블에 있는 '나보통' 사원의 매니저 번호가 `100`번이라고 치자. 그럼 컴퓨터는 `e2` 테이블을 쫙 뒤져서 사번이 `100`번인 사람(예: '김부장')의 행을 통째로 가져와서 '나보통' 옆에 착 붙여준다.

`SELECT ... e2.사원명`: 둘을 찰싹 붙여놨으니, 이제 안심하고 `e2` 쪽에 있는 '김부장'이라는 이름을 꺼내올 수 있게 되는 거지!

## 자주하는 실수

**"테이블 별칭(Alias) 안 쓰고 그냥 `사원.사원명` 이렇게 쓰면 안 되나요?"**
- **절대 불가능하다 (에러 발생).** 
- 테이블 이름이 똑같기 때문에, `SELECT 사원.사원명`이라고 쓰면 컴퓨터는 "어느 쪽 사원 테이블에서 가져오라는 거야?" 하고 멘붕에 빠진다.
- 셀프 조인에서 테이블 별칭(`e1`, `e2` 등)은 선택이 아니라 **필수 생존 규칙**이다.

**"왜 `INNER JOIN` 안 쓰고 굳이 `LEFT JOIN`을 썼나요?"**
- 회사 최고참인 **회장님(CEO)은 담당 매니저가 없으므로 '매니저번호' 컬럼이 `NULL`**이다.
- 만약 `INNER JOIN`을 해버리면, 짝꿍을 찾지 못한 회장님 데이터는 결과 화면에서 아예 날아가 버린다! (회장님 명단 누락 ➔ 시말서 작성)
- 따라서 매니저가 없는 사람도 내 이름은 일단 출력되도록 반드시 **`LEFT JOIN`**을 사용해서 원본(`e1`) 데이터를 보존해 주어야 한다.
