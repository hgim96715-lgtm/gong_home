# SQL Self Join (셀프 조인)

## 개념 한 줄 요약

> **"하나의 테이블에 별칭(Alias)을 두 개 붙여, 마치 두 테이블인 것처럼 자기 자신과 JOIN 하는 기법"**

일반 JOIN 은 **서로 다른 두 테이블** 을 합칠 때 쓴다. 하지만 **같은 테이블 안에서** 데이터끼리 비교해야 할 때는 테이블을 가상으로 두 개로 나눠야 한다.

> JOIN 기본 개념 → [[SQL_Standard_JOIN]] 참고

---

## 언제 쓰나?

|사용 사례|예시|
|---|---|
|계층 구조 탐색|사원-매니저 관계, 카테고리 대분류-소분류|
|누적 집계|특정 일자 이전의 모든 누적 매출|
|상대적 비교|어제 매출 vs 오늘 매출 비교|
|조건부 매칭|같은 테이블 내에서 조건이 맞는 쌍 찾기|

> 윈도우 함수(`LAG`, `LEAD`, `SUM OVER`) 가 지원되지 않는 환경에서 특히 유용하다.

---

## 별칭(Alias) 은 필수, 선택이 아니다 ⚠️

```sql
-- ❌ 에러: 어느 테이블의 일자인지 DB가 구분 불가
SELECT 일자별매출.일자 FROM 일자별매출 JOIN 일자별매출 ...

-- ✅ 별칭으로 구분
SELECT A.일자 FROM 일자별매출 A JOIN 일자별매출 B ...
```

테이블 이름이 같아서 DB 엔진이 어느 쪽을 참조할지 알 수 없다. 셀프 조인에서 별칭은 **필수 생존 규칙** 이다.

# 패턴 ① — 범위 조건 조인 (누적합)

`>=` 같은 부등호로 "과거 데이터를 전부 끌어오는" 패턴이다.

## 원본 데이터

|일자|매출액|
|---|---|
|2015.11.01|1000|
|2015.11.02|1000|
|2015.11.03|1000|

## 쿼리

```sql
SELECT
    A.일자,
    SUM(B.매출액) AS 누적매출액
FROM 일자별매출 A
JOIN 일자별매출 B
    ON A.일자 >= B.일자   -- ★ A 날짜 기준, 같거나 과거인 B를 전부 매칭
GROUP BY A.일자
ORDER BY A.일자;
```

## 결과

|일자|누적매출액|
|---|---|
|2015.11.01|1000|
|2015.11.02|2000|
|2015.11.03|3000|

---

## 내부 동작 단계별 추적

`A.일자 = 11.03` 일 때 DB 엔진이 어떻게 처리하는지 따라가보자.

**[1단계] 테이블 복제** 같은 테이블에 별칭 A, B를 붙여 두 개처럼 취급한다.

|A.일자|A.매출액|B.일자|B.매출액|
|---|---|---|---|
|11.01|1000|11.01|1000|
|11.02|1000|11.02|1000|
|11.03|1000|11.03|1000|

**[2단계] ON 조건으로 매칭 (`A.일자 >= B.일자`)** A 가 `11.03` 이면 조건을 만족하는 B 는 `11.01, 11.02, 11.03` 3건이 달라붙는다.

|A.일자|B.일자|B.매출액|
|---|---|---|
|11.03|11.01|1000|
|11.03|11.02|1000|
|11.03|11.03|1000|

**[3단계] GROUP BY + SUM** `A.일자` 로 묶고 `B.매출액` 을 모두 더한다 → `1000 + 1000 + 1000 = 3000`

---

## 부등호 방향이 바뀌면 결과가 뒤집힌다 ⭐️

|조건|동작|결과 방향|
|---|---|---|
|`ON A.일자 >= B.일자`|A 기준 **과거** B 를 합산|날짜 갈수록 누적합 **증가**|
|`ON A.일자 <= B.일자`|A 기준 **미래** B 를 합산|날짜 갈수록 누적합 **감소**|

`<=` 조건으로 바꾸면 `11.01` 에 전체 총합이, `11.03` 에 1000이 찍히는 반대 결과가 나온다.

# 패턴 ② — 등가 조인 (계층 구조)

범위 조건(`>=`) 이 아닌 **등호(`{text}=`) 조인** 으로 계층 관계를 표현하는 패턴이다. 사원 테이블에서 각 사원의 매니저 이름을 찾는 경우.

```sql
SELECT
    E.사원명  AS 사원,
    M.사원명  AS 매니저
FROM 사원 E
LEFT JOIN 사원 M               -- 같은 테이블을 M 별칭으로 한 번 더 사용
    ON E.매니저번호 = M.사원번호; -- E의 매니저번호 = M의 사원번호
```

|사원|매니저|
|---|---|
|홍길동|NULL (최상위)|
|이순신|홍길동|
|이민정|홍길동|

> **LEFT JOIN 을 쓰는 이유:** 최상위 사원(매니저 없음)도 결과에 포함시키기 위해서다. INNER JOIN 을 쓰면 매니저가 없는 행은 제외된다.

# 패턴 ③ — 조건부 매칭 (멘토링 문제)

> **"같은 테이블에서 조건이 다른 두 그룹을 뽑아 매칭하라."** → Self JOIN + LEFT JOIN 의 결합

## 문제 조건

- **멘티:** 2021-12-31 기준 3개월 이내 입사 (`hire_date > 2021-09-30`)
- **멘토:** 2021-12-31 기준 재직 2년 이상 (`hire_date <= 2019-12-31`)
- 서로 **다른 부서** 끼리만 매칭
- 멘토가 없어도 **멘티는 전부 출력** (NULL 허용)
- 정렬: 멘티 ID → 멘토 ID 오름차순

## 풀이 사고 흐름

```
1. 같은 테이블(employees)에서 두 역할을 뽑아야 함  →  Self JOIN
2. 기준이 되는 쪽은?  →  멘티 (멘토 없어도 출력해야 함)
3. 어떤 JOIN?         →  LEFT JOIN (멘티 보존, 멘토 없으면 NULL)
4. 멘티 필터는?       →  WHERE 절 (기준 테이블 e1 자체를 거름)
5. 멘토 필터는?       →  ON 절  (조인 대상 e2를 거르되 e1은 보존)
6. 다른 부서 조건은?  →  ON 절  (조인 조건이므로)
```

> `ON` 절 vs `WHERE` 절 구분법 → [[SQL_Standard_JOIN# 핵심 개념 — ON 절 vs WHERE 절|ON절 vs WHERE절]] 참고

## 쿼리

```sql
SELECT
    e1.employee_id  AS mentee_id,
    e1.name         AS mentee_name,
    e2.employee_id  AS mentor_id,
    e2.name         AS mentor_name
FROM employees e1                          -- e1 = 멘티 (기준 테이블)
LEFT JOIN employees e2                     -- e2 = 멘토 (같은 테이블 Self JOIN)
    ON  e2.hire_date <= '2019-12-31'       -- ✅ ON: 멘토 조건 (조인 대상 필터)
    AND e1.department_id <> e2.department_id  -- ✅ ON: 다른 부서 조건
WHERE e1.hire_date > '2021-09-30'          -- ✅ WHERE: 멘티 조건 (기준 테이블 필터)
ORDER BY mentee_id ASC, mentor_id ASC;
```

## 왜 이 구조인가?

```
WHERE e1.hire_date > '2021-09-30'
→ 멘티 자격 없는 사람은 아예 불필요 → e1 행 자체를 삭제해도 됨 → WHERE 절 ✅

ON e2.hire_date <= '2019-12-31'
→ 멘토를 필터링하되 e1(멘티)은 반드시 보존 → ON 절 ✅
→ 이 조건을 WHERE 에 두면: 멘토 없는 멘티까지 제거 → 문제 요구사항 위반! ❌
```

> ✍️ **SQLD·면접용 한 줄 회고** `LEFT JOIN` 에서 **기준 테이블 필터는 WHERE, 조인 대상 필터는 ON 절** 에 둔다.

# 셀프 조인 vs 윈도우 함수

실무에서는 셀프 조인 대신 **윈도우 함수** 를 쓰는 것이 훨씬 빠르다.

|구분|셀프 조인|윈도우 함수|
|---|---|---|
|코드 길이|길다|짧다|
|성능|나쁨 (데이터 폭증)|좋음 (테이블 1회 스캔)|
|지원 환경|모든 SQL|대부분의 현대 DB|
|시험 출제|⭐️ SQLD 빈출|실무 우선|

```sql
-- 셀프 조인으로 누적합
SELECT A.일자, SUM(B.매출액) AS 누적매출액
FROM 일자별매출 A
JOIN 일자별매출 B ON A.일자 >= B.일자
GROUP BY A.일자;

-- ✅ 윈도우 함수로 동일한 결과를 단 한 줄에
SELECT 일자, SUM(매출액) OVER (ORDER BY 일자) AS 누적매출액
FROM 일자별매출;
```

> SQLD 시험에서는 셀프 조인이 출제되지만, 실무에서는 윈도우 함수를 먼저 고려하자.
