# SQL Self Join (셀프 조인)

> 하나의 테이블에 별칭(Alias)을 두 개 붙여, 마치 두 개의 테이블인 것처럼 자기 자신과 JOIN하는 기법

---

## 왜 필요한가?

일반 JOIN은 **서로 다른 두 테이블**을 합칠 때 쓴다.  
하지만 아래처럼 **같은 테이블 안에서** 데이터끼리 비교해야 할 때는 테이블을 가상으로 두 개로 나눠야 한다.

|사용 사례|예시|
|---|---|
|계층 구조 탐색|사원-매니저 관계, 카테고리 대분류-소분류|
|누적 집계|특정 일자 이전의 모든 누적 매출|
|상대적 비교|어제 매출 vs 오늘 매출 비교|

> 윈도우 함수(`LAG`, `LEAD`, `SUM OVER`)가 지원되지 않는 환경에서 특히 유용하다.

---

##  핵심 동작 원리 (누적합 예제)

**원본 데이터 (테이블 1개)**

|일자|매출액|
|---|---|
|2015.11.01|1000|
|2015.11.02|1000|
|2015.11.03|1000|

```sql
SELECT 
    A.일자,
    SUM(B.매출액) AS 누적매출액
FROM 일자별매출 A
JOIN 일자별매출 B
    ON A.일자 >= B.일자   -- ★ 핵심: A의 날짜 기준으로, 같거나 과거인 B의 데이터를 모두 매칭
GROUP BY A.일자
ORDER BY A.일자;
```

**실행 결과**

|일자|누적매출액|
|---|---|
|2015.11.01|1000|
|2015.11.02|2000|
|2015.11.03|3000|

---

##  내부 동작 단계별 추적

`A.일자 = 11.03` 일 때를 기준으로 DB 엔진이 어떻게 처리하는지 추적해보자.

**[1단계] 테이블 복제**

같은 테이블에 별칭 A, B를 각각 붙여 두 개처럼 취급한다.

| A.일자  | A.매출액 | B.일자  | B.매출액 |
| ----- | ----- | ----- | ----- |
| 11.01 | 1000  | 11.01 | 1000  |
| 11.02 | 1000  | 11.02 | 1000  |
| 11.03 | 1000  | 11.03 | 1000  |

**[2단계] ON 조건으로 매칭 (`A.일자 >= B.일자`)**

A가 `11.03`이면 조건을 만족하는 B는 `11.01, 11.02, 11.03` 3건이 달라붙는다.

|A.일자|B.일자|B.매출액|
|---|---|---|
|11.03|11.01|1000|
|11.03|11.02|1000|
|11.03|11.03|1000|

**[3단계] GROUP BY + SUM**

`A.일자`로 묶고 `B.매출액`을 모두 더한다 → `1000 + 1000 + 1000 = 3000`

##  조직도 조회 예제 (등가 조인)

누적합처럼 범위 조건(`>=`)이 아닌 **등호(`{text}=`) 조인**으로도 셀프 조인을 많이 쓴다.  
사원 테이블에서 각 사원의 매니저 이름을 찾는 경우다.

```sql
SELECT 
    E.사원명    AS 사원,
    M.사원명    AS 매니저
FROM 사원 E
LEFT JOIN 사원 M          -- 같은 테이블을 M이라는 별칭으로 한 번 더 사용
    ON E.매니저번호 = M.사원번호;  -- E의 매니저번호 = M의 사원번호
```

|사원|매니저|
|---|---|
|홍길동|NULL (최상위)|
|이순신|홍길동|
|이민정|홍길동|

> `LEFT JOIN`을 쓰는 이유: 최상위 사원(매니저 없음)도 결과에 포함시키기 위해서다.  
> `INNER JOIN`을 쓰면 매니저가 없는 행은 제외된다.

##  시험 핵심 포인트

### ① 부등호 방향이 바뀌면 결과가 뒤집힌다

|조건|동작|
|---|---|
|`ON A.일자 >= B.일자`|A 기준 **과거** B를 합산 → 날짜가 갈수록 누적합 증가|
|`ON A.일자 <= B.일자`|A 기준 **미래** B를 합산 → 날짜가 갈수록 누적합 감소|

`<=` 조건으로 바꾸면 `11.01`에 전체 총합이, `11.03`에 1000이 찍히는 반대 결과가 나온다.

###  별칭(Alias)은 필수, 선택이 아니다

```sql
-- ❌ 에러 발생: 어느 테이블의 일자인지 알 수 없음
SELECT 일자별매출.일자 FROM 일자별매출 JOIN 일자별매출 ...

-- ✅ 별칭으로 구분
SELECT A.일자 FROM 일자별매출 A JOIN 일자별매출 B ...
```

테이블 이름이 같기 때문에 DB 엔진이 어느 쪽을 참조할지 구분할 수 없다.  
셀프 조인에서 별칭은 **필수 생존 규칙**이다.

---

##  셀프 조인 vs 윈도우 함수

실무에서는 셀프 조인 대신 **윈도우 함수**를 쓰는 것이 훨씬 빠르다.

|구분|셀프 조인|윈도우 함수|
|---|---|---|
|코드 길이|길다|짧다|
|성능|나쁨 (데이터 폭증)|좋음 (테이블 1회 스캔)|
|지원 환경|모든 SQL|대부분의 현대 DB|

```sql
-- ✅ 윈도우 함수로 동일한 누적합을 단 한 줄에
SELECT 일자, SUM(매출액) OVER (ORDER BY 일자) AS 누적매출액
FROM 일자별매출;
```

> SQLD 시험에서는 셀프 조인이 출제되지만, 실무에서는 윈도우 함수를 먼저 고려하자.
