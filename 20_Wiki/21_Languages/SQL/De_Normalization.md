## 개념 한 줄 요약

**"조회(SELECT) 속도를 극대화하기 위해, 정규화 원칙을 깨고 고의적으로 데이터 중복을 허용하는 기법."** 
(정규화가 '정리정돈'이라면, 반정규화는 '자주 쓰는 물건을 책상 위에 꺼내놓는 것'과 같다.)

>선 정규화, 후 반정규화

---
## 왜 필요한가? (Why)

**문제점 (Over-Normalization):**

- 정규화를 완벽하게(3차 정규화 이상) 하면 테이블이 너무 잘게 쪼개진다.
- 데이터를 조회할 때마다 **`JOIN`을 10번씩** 해야 한다면? 서버 CPU가 터지고 응답 속도가 느려진다.

**해결책:**

- 자주 같이 쓰이는 컬럼은 **그냥 한 테이블에 중복해서 넣어두자.**
- 매번 계산하기 힘든 통계값(합계, 평균)은 **미리 계산해서 저장해두자.**

---
## 핵심 기법 3가지 (Techniques)

### 컬럼 중복 (가장 흔함)

**"조인(JOIN)을 줄이기 위해 옆 동네 데이터를 가져온다."**

- **상황:** 게시판(`Board`) 목록을 보여줄 때 작성자 이름(`UserName`)이 필요한데, 이름은 유저(`User`) 테이블에만 있다.
- **정규화:** `Board` 테이블에는 `UserID`만 있고, 조회할 때마다 `JOIN User`를 해야 함.
- **반정규화:** `Board` 테이블에 `UserName` 컬럼을 아예 추가해버림. (데이터가 중복되지만, 조인 없이 조회 가능)

### 2. 파생 컬럼 추가 (Derived Column)

**"계산된 결과값을 미리 저장한다."**

- **상황:** 주문 내역을 볼 때마다 `주문총액`을 알고 싶다.
- **정규화:** 주문상세(`OrderDetail`) 테이블을 다 뒤져서 `SUM(Price * Qty)`를 매번 계산해야 함.
- **반정규화:** 주문(`Order`) 테이블에 `TotalAmount` 컬럼을 만들고, 주문이 들어올 때 계산해서 박아넣음. (조회 시 계산 부하 0)

### 3. 테이블 병합 (Table Merging)

- **1:1 관계 병합:** 너무 자주 같이 쓰이는 두 테이블은 그냥 하나로 합친다.
- **슈퍼/서브 타입 병합:** 전체 직원, 정규직, 계약직 테이블이 따로 있는데 조회하기 힘들면 그냥 '통합 직원 테이블' 하나로 합친다.

## 트레이드오프 (Trade-off) ⭐️

**"조회 속도(SELECT)를 얻는 대신, 입력/수정/삭제(DML) 성능과 데이터 무결성을 포기한다."**

반정규화는 **공짜 점심이 아닙니다.** 치명적인 단점이 있습니다.

| **구분** | **정규화 (Normalization)**          | **반정규화 (De-normalization)**          |
| ------ | -------------------------------- | ------------------------------------ |
| **장점** | 데이터 중복 없음, 무결성 유지 쉬움, 저장 공간 절약   | **조회(SELECT) 속도가 엄청 빠름**, 조인 감소      |
| **단점** | **조인(JOIN)이 많아 조회 느림**, SQL 복잡해짐 | **데이터 중복 발생**, 데이터 수정 시 여러 곳 다 고쳐야 함 |
| **비유** | 서류를 종류별로 완벽하게 파일철에 보관함           | 자주 보는 서류 복사본을 내 책상에도 둠               |


> **⚠️ 위험성 예시:** 유저가 닉네임을 바꿨다.
> - **정규화:** `User` 테이블 1곳만 고치면 끝.
> - **반정규화:** `Board`, `Order`, `Review` 등 닉네임을 복사해둔 **모든 테이블**을 다 찾아서 고쳐야 함. (하나라도 빼먹으면 데이터 불일치 버그 발생!)

---
## 실무 적용 가이드 (When to use)

**무조건 반정규화를 하는 게 아닙니다.** 다음 순서를 따르세요.

1. 일단 **정규화**를 충실히 수행한다. (기본 원칙)
2. SQL을 짰는데 **조인이 너무 많거나(5개 이상), 대용량 데이터**라서 조회가 느리다.
3. 인덱스(Index) 튜닝으로 해결해 본다.
4. 그래도 느리면 **최후의 수단으로 반정규화**를 적용한다.

## 초보자가 자주 하는 실수 (Misconceptions)

### ① "반정규화는 나쁜 건가요?"

- 아닙니다.
- **성능 최적화의 핵심 기술**입니다. 조회 성능이 중요한 서비스(예: 쇼핑몰 상품 목록, 뉴스 피드)에서는 필수입니다.

### ② "처음부터 반정규화해서 설계하면 안 되나요?"

- **안 됩니다.** 
- 처음부터 중복을 허용하면 나중에 데이터가 꼬였을 때 어디가 원본인지 모르게 됩니다. "선 정규화, 후 반정규화" 순서를 꼭 지키세요.

### ③ "JSON 컬럼을 쓰는 것도 반정규화인가요?"

- 맞습니다. 
- 별도 테이블로 쪼개야 할 속성들을 하나의 JSON 컬럼에 몰아넣는 것도 대표적인 반정규화 기법입니다. (스키마 유연성 + 조회 편의성)
