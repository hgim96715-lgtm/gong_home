#  SQL 윈도우 함수 (Window Functions) 

## 한 줄 요약

`GROUP BY`처럼 행을 압축해서 날려버리지 않고, **원본 행을 그대로 유지한 채로** 계산 결과를 옆에 나란히 붙여주는 강력한 분석 함수다.

---

## 왜 필요한가?

**문제점:** "부서별 평균 연봉"을 구하려고 `GROUP BY 부서`를 쓰면, 부서별로 딱 1줄씩만 남고 사원들의 개별 데이터(이름, 직급 등)는 모두 사라져 버린다.

**해결책:** 윈도우 함수를 쓰면 개별 사원의 데이터를 그대로 보존하면서, 옆 컬럼에 "해당 사원이 속한 부서의 평균 연봉"이나 "부서 내 연봉 순위"를 계산해서 붙여넣을 수 있다.

---

## 실무 맥락

데이터 엔지니어링 및 분석 실무에서 가장 많이 쓰이는 문법이다. "웹사이트 접속자의 이전/현재 페이지 체류 시간 비교(행 순서)", "게시글의 일자별 누적 조회수(집계)", "상위 10% 우수 고객 추출(비율)" 등 복잡한 요건을 단일 쿼리로 해결할 때 사용한다. SQLD 시험에서도 배점이 가장 높은 핵심 출제 영역이다.

---
## 기본 문법 구조

윈도우 함수는 함수명 뒤에 반드시 **`OVER()`** 절이 따라붙는 것이 특징이다.

```sql
함수명(인자) OVER (
    PARTITION BY 그룹핑할_컬럼   -- 1. 무엇을 기준으로 그룹(파티션)을 나눌 것인가? (선택)
    ORDER BY 정렬할_컬럼         -- 2. 파티션 내에서 어떤 순서로 정렬할 것인가? (선택/함수에 따라 필수)
    ROWS BETWEEN 시작 AND 끝     -- 3. 연산할 행의 범위를 어디까지로 제한할 것인가? (선택)
)
```

### PARTITION BY vs GROUP BY — 가장 많이 헷갈리는 포인트

```sql
-- GROUP BY: 그룹을 "합쳐서" 1줄로 압축한다 (원본 행이 사라짐)
SELECT 부서, AVG(급여) FROM 사원 GROUP BY 부서;
-- 결과: 부서별로 딱 1줄씩만 나옴 (행 수가 줄어듦)

-- PARTITION BY: 그룹 내에서 계산하되 "원본 행을 그대로 유지"한다
SELECT 사원명, 부서, 급여,
       AVG(급여) OVER (PARTITION BY 부서) AS 부서평균
FROM 사원;
-- 결과: 원본 행 수 그대로 + 옆에 부서평균 컬럼이 추가됨
```

```
GROUP BY   →  줄이는 것   (합쳐서 요약, 행 수 감소)
PARTITION BY →  붙이는 것  (원본 유지하면서 계산값 추가)
```

##  순위 함수

> 데이터의 순위를 매길 때 사용한다. **공동 순위 발생 시 그다음 번호를 어떻게 처리할지**에 따라 함수를 골라 쓴다.

### 함수 한눈에 보기

|함수|동점자 처리|순위 번호|출력 예시|
|---|---|:-:|---|
|`ROW_NUMBER()`|동점 무시, 무조건 고유 번호|연속|1, 2, 3, 4|
|`RANK()`|동점자 같은 등수, 다음 번호 건너뜀|불연속|1, 2, 2, **4**|
|`DENSE_RANK()`|동점자 같은 등수, 번호 안 건너뜀|연속|1, 2, 2, **3**|

### ROW_NUMBER() — 무조건 고유한 번호

동점자고 뭐고 상관없이, **무조건 1, 2, 3, 4 고유한 순번**을 부여한다.

> 💡 언제 쓰나? → "공동 순위 없이, **정확히 N명만** 뽑아줘" / 페이지네이션(행 번호 매기기)

```sql
ROW_NUMBER() OVER (ORDER BY 급여 DESC)
-- 같은 급여여도: 1, 2, 3, 4 (무조건 고유 번호)
```

### RANK() — 공동 순위 허용, 번호는 건너뜀 (올림픽 메달 방식)

동점자에게 **같은 순위**를 주되, 그 다음 번호는 **건너뛴다**.  
공동 2위가 2명이면 → 1, 2, 2, **4** (3등이 없어짐)

> 💡 언제 쓰나? → "올림픽처럼 공동 2등 인정, 단 3등 자리는 없애도 됨"

```sql
RANK() OVER (ORDER BY 급여 DESC)
-- 동점자: 1, 2, 2, 4, 5 (3번 번호가 사라짐)
```

### DENSE_RANK() — 공동 순위 허용, 번호는 연속 (밀집 순위)

동점자에게 **같은 순위**를 주되, 그 다음 번호는 **건너뛰지 않는다**.  
공동 2위가 2명이면 → 1, 2, 2, **3** (번호가 빠지지 않음)

> 💡 언제 쓰나? → "`<= 3`으로 잘랐는데 **3위가 여러 명이면 전부 포함**하고 싶을 때"

```sql
DENSE_RANK() OVER (ORDER BY 급여 DESC)
-- 동점자: 1, 2, 2, 3, 4 (번호 연속, 빈 자리 없음)
```

### 결과 비교표 (급여: 300, 200, 200, 100인 경우)

|급여|ROW_NUMBER()|RANK()|DENSE_RANK()|
|:-:|:-:|:-:|:-:|
|300|1|1|1|
|200|2|2|2|
|200|3|2|2|
|100|4|**4** (3 없음)|**3** (연속)|

### Top-N 뽑기 핵심 패턴

> ⚠️ **중요:** 윈도우 함수는 `WHERE` 절에서 바로 쓸 수 없다!  
> **CTE(WITH절) 또는 인라인 뷰**로 한 번 감싼 뒤, 바깥에서 필터링해야 한다.

```sql
-- 급여 순위 번호표를 먼저 붙이고, 바깥에서 3등 이하만 필터링!
WITH RankedEmp AS (   -- 💡 이렇게 한 번 감싸는 게 핵심!
    SELECT 
        사원명, 급여,
        ROW_NUMBER() OVER (ORDER BY 급여 DESC) AS rn  -- 얄짤없이 딱 3명
        -- RANK()       OVER (ORDER BY 급여 DESC) AS rn  -- 공동 2등 있으면 3명
        -- DENSE_RANK() OVER (ORDER BY 급여 DESC) AS rn  -- 3위까지 동점자 전원 포함
    FROM 사원
)
SELECT 사원명, 급여, rn AS 순위
FROM RankedEmp
WHERE rn <= 3;  -- 바깥 쿼리에서 3위까지만 싹둑!
```

### 부서별 Top-N — PARTITION BY 활용

```sql
-- 부서별 연봉 탑 3 뽑기
WITH RankedEmp AS (
    SELECT
        부서, 사원명, 급여,
        DENSE_RANK() OVER (
            PARTITION BY 부서    -- 부서가 바뀔 때마다 순위를 1부터 리셋!
            ORDER BY 급여 DESC
        ) AS rn
    FROM 사원
)
SELECT 부서, 사원명, 급여, rn AS 순위
FROM RankedEmp
WHERE rn <= 3;
```

### 언제 뭘 써야 하나? 실무 선택 기준

|상황|추천 함수|이유|
|---|---|---|
|무조건 1명만 뽑아야 할 때 (중복 허용 X)|`ROW_NUMBER()`|동점이어도 1명만 나옴|
|동점자도 같은 등수로 인정해야 할 때|`RANK()`|올림픽처럼 공동 2위 인정|
|등수 번호가 연속이어야 할 때|`DENSE_RANK()`|빈 등수 없이 1, 2, 3 연속|
|상위 N명 자르되 동점자 모두 포함|`RANK()`|공동 3위 2명이면 둘 다 포함|
|페이지네이션 / 행 번호 매기기|`ROW_NUMBER()`|고유 번호가 필요하니까|

### RANK vs COUNT — 어떤 도구를 꺼낼까?

```
"순위 번호 (1등, 2등...)" 가 결과에 필요한가?
        ↓ YES                        ↓ NO
     RANK 써라               COUNT(*) + GROUP BY + HAVING
```

|문제에 이 단어가 보이면|꺼낼 도구|
|---|---|
|"1위", "상위 N위", "순위"|`RANK()` / `DENSE_RANK()`|
|"횟수", "몇 번", "등재", "건수"|`COUNT(*)`|
|"중복 없이 한 명만"|`ROW_NUMBER()`|
|"각 그룹에서 최고값"|`RANK()` + `WHERE rnk = 1`|

```sql
-- ✅ "상위 50위 이내 2회 이상 등재" → COUNT로 충분
SELECT title, COUNT(*) AS cnt
FROM bestseller_top50
GROUP BY title
HAVING COUNT(*) >= 2
ORDER BY cnt DESC;

-- ✅ "각 부서에서 급여 1위만 뽑기" → RANK가 필요
WITH ranked AS (
    SELECT *, RANK() OVER (PARTITION BY 부서 ORDER BY 급여 DESC) AS rnk
    FROM 사원
)
SELECT * FROM ranked WHERE rnk = 1;
```

> 📎 COUNT 삼형제(`COUNT(*)` / `COUNT(컬럼)` / `COUNT(DISTINCT)`) 차이가 헷갈릴 때 →[[SQL_Aggregate_GROUP_BY#① COUNT 삼형제 (가장 많이 헷갈림)| COUNT 삼형제]] 참고

## 집계 함수

> 우리가 아는 `SUM`, `AVG`, `MAX`, `MIN`, `COUNT`를 **원본 행 유지 상태**로 계산한다.  
> `ORDER BY`를 넣느냐 빼느냐에 따라 **전체 합산** 또는 **누적 합산**이 된다.

- **`ORDER BY` 없음:** 파티션 전체의 총합·최댓값 등을 구해서 **같은 파티션의 모든 행에 똑같은 값을 복사**해 붙인다.
- **`ORDER BY` 있음:** 첫 행부터 현재 행까지 차곡차곡 **누적**해서 더하거나 갱신해 나간다.

```sql
SELECT 
    부서, 사원명, 입사일자, 주문금액, 급여,

    -- 1. [전체 합산] ORDER BY 없음 → 총합이 모든 행에 동일하게 찍힘
    SUM(주문금액) OVER() AS 총_주문금액,

    -- 2. [파티션 평균] PARTITION BY만 있고 ORDER BY 없음
    --    같은 부서라면 모두에게 동일한 부서 평균 급여가 찍힘
    ROUND(AVG(급여) OVER (PARTITION BY 부서), 2) AS 부서평균급여,

    -- 3. [파티션 최댓값] ORDER BY 없음
    --    "우리 부서에서 제일 높은 급여"가 부서원 모두의 행에 동일하게 출력됨
    MAX(급여) OVER (PARTITION BY 부서) AS 부서내_최고급여,

    -- 4. [누적 합산] ORDER BY 있음 → 입사일자 순으로 한 줄씩 누적
    SUM(주문금액) OVER (ORDER BY 입사일자) AS 누적_주문금액,

    -- 5. [누적 최솟값] ORDER BY 있음 → 날짜가 지날 때마다 역대 최저 급여 갱신
    MIN(급여) OVER (ORDER BY 입사일자) AS 지금까지_최저급여

FROM 사원테이블;
```

### 결과 이해 예시 (영업부 사원 4명)

|사원명|입사일|급여|MAX(급여) OVER(PARTITION BY 부서)<br>※ ORDER BY 없음|MIN(급여) OVER(ORDER BY 입사일)<br>※ ORDER BY 있음|💡 계산 원리|
|---|---|---|:-:|:-:|---|
|김신입|1/1|300|**500**|**300**|MAX: 영업부 전체 최고 = 500, 전원 동일값<br>MIN: 첫 행 → 본인 300|
|이대리|1/2|500|**500**|**300**|MAX: 여전히 500<br>MIN: 1/1~1/2 중 최저 = 300 유지|
|박과장|1/3|400|**500**|**300**|MAX: 계속 500 복붙<br>MIN: 최저는 여전히 300|
|최인턴|1/4|200|**500**|**200 🔥**|MAX: ORDER BY 없으므로 동일<br>MIN: 더 낮은 값 등장 → **역대 최저 갱신**|

## 행 순서 함수 (SQL Server 미지원)

> **현재 행을 기준으로, 이전/다음 행의 값을 끌어올 때 사용한다.**  
> 시계열 데이터(날짜별 변화량) 분석의 핵심 도구다.

### 함수 한눈에 보기

|함수|역할|SQL Server 지원|
|---|---|:-:|
|`LAG()`|이전 행 값 가져오기|✅|
|`LEAD()`|다음 행 값 가져오기|✅|
|`FIRST_VALUE()`|파티션 내 첫 번째 값|❌|
|`LAST_VALUE()`|파티션 내 마지막 값|❌|

### LAG() — 이전 행 가져오기

`LAG(컬럼명, [건너뛸_행_수], [NULL 대체값])`

- **건너뛸_행_수** 생략 → 기본값 `1` (바로 윗줄)
- **NULL 대체값** 생략 → 값이 없으면 `NULL` 출력

```sql
-- 기본: 바로 어제 값 (1칸 위, 없으면 NULL)
SELECT 접속일자, 접속자수,
    LAG(접속자수) OVER (ORDER BY 접속일자) AS 전일_접속자수
FROM 트래픽로그;

-- 심화: 7일 전 값, 없으면 0으로 대체
SELECT 접속일자, 접속자수,
    LAG(접속자수, 7, 0) OVER (ORDER BY 접속일자) AS 일주일전_접속자수
FROM 트래픽로그;
```

### LEAD() — 다음 행 가져오기

`LEAD(컬럼명, [건너뛸_행_수], [NULL 대체값])` — LAG와 구조 동일, 방향만 반대

```sql
SELECT 접속일자, 접속자수,
    LEAD(접속자수) OVER (ORDER BY 접속일자) AS 익일_접속자수
FROM 트래픽로그;
```

### FIRST_VALUE() — 파티션 내 첫 번째 값

첫 번째 값은 항상 윈도우 범위 안에 존재하므로 **별도 설정 없이 정상 작동**한다.

```sql
SELECT 접속일자, 접속자수,
    FIRST_VALUE(접속자수) OVER (PARTITION BY 월 ORDER BY 접속일자) AS 월초_접속자수
FROM 트래픽로그;
```

### LAST_VALUE() — 파티션 내 마지막 값 ⚠️ 주의 필요!

윈도우 기본 범위는 **"현재 행까지"** 다. 그냥 쓰면 **자기 자신을 마지막 값으로** 출력해버린다.  
✅ **해결책: `UNBOUNDED FOLLOWING`으로 범위를 끝까지 열어야 한다!**

```sql
SELECT 접속일자, 접속자수,
    LAST_VALUE(접속자수) OVER (
        PARTITION BY 월
        ORDER BY 접속일자
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING  -- ← 이게 핵심!
    ) AS 월말_접속자수
FROM 트래픽로그;
```

### 🚨 자주 하는 실수 — 연속성 문제

**❌ 잘못된 접근: COUNT로 연속 출전 판단**

```sql
HAVING COUNT(*) >= 2  -- 이건 틀렸다!
```

> **두 번 이상 참가 ≠ 연속 참가**  
> 연속성은 **"개수"** 의 문제가 아니라, **"시간(연도) 간격"** 의 문제다!

**✅ 올바른 접근: LAG로 연속성 판단**

연속성 판별 공식: `현재값 - 이전값 = 일정 간격`

|상황|공식|기댓값|
|---|---|---|
|올림픽 연속 출전|`현재연도 - LAG(연도) = 4`|4년 간격|
|날짜 연속|`현재날짜 - LAG(날짜) = 1`|1일 간격|
|월 연속|`현재월 - LAG(월) = 1`|1개월 간격|

```sql
SELECT 선수명, 참가연도,
    LAG(참가연도) OVER (PARTITION BY 선수명 ORDER BY 참가연도) AS 이전_참가연도,
    참가연도 - LAG(참가연도) OVER (PARTITION BY 선수명 ORDER BY 참가연도) AS 간격
FROM 올림픽참가기록;
```

**문제 키워드 신호**

|키워드|필요한 함수|
|---|---|
|**연속** (연속 출전, 연속 달성)|`LAG` + 간격 비교|
|**이전** (이전 달 대비, 전날 대비)|`LAG`|
|**다음** (다음 달 예측, 익일 비교)|`LEAD`|
|**변화** (증가율, 변화량)|`LAG` + 뺄셈|
|**증가** (연속 증가, 꾸준히 상승)|`LAG` + 조건 비교|

## 비율 함수 (SQL Server 미지원)

> 데이터를 **비율·분포·등급**으로 표현할 때 사용한다.

### 함수 한눈에 보기

|함수|결과값 범위|핵심 역할|
|---|:-:|---|
|`RATIO_TO_REPORT()`|0 ~ 1 (합산 = 1)|전체 합 대비 내 비율|
|`PERCENT_RANK()`|0.0 ~ 1.0 (0부터 시작)|상대적 순위 비율 (상위 몇 %)|
|`CUME_DIST()`|(0, 1] (0 불가)|누적 분포 비율 (나 포함)|
|`NTILE(N)`|1 ~ N (정수)|N개 등급으로 균등 분할|

### RATIO_TO_REPORT() — 전체 합 대비 내 비율

파티션 내 전체 합계 대비 현재 행의 값이 차지하는 **백분율(비율)** 을 구한다.

> 💡 Oracle 전용 문법. PostgreSQL·SQL Server에서는 직접 나눗셈으로 대체한다.  
> ⚠️ 괄호 안에 타겟 컬럼을 넣고, `OVER()` 안에는 `ORDER BY`를 쓰지 않는다!

```sql
SELECT 사원명, 급여,

    -- [Oracle 전용] RATIO_TO_REPORT 오리지널 문법
    RATIO_TO_REPORT(급여) OVER() AS 오라클_급여비율,

    -- [실무용 / PostgreSQL, SQL Server] 대체 구현
    ROUND(급여 / SUM(급여) OVER(), 2) AS 포그레_급여비율

FROM 사원;
```

### PERCENT_RANK() — 상대적 위치를 비율로 표현

순위를 **0.0 ~ 1.0 사이의 비율**로 나타낸다.  
공식: `(현재 순위 - 1) / (전체 행 수 - 1)`  
→ 1등은 항상 `0.0`, 꼴찌는 항상 `1.0`

> 💡 언제 쓰나? → "**상위 30% 이내**인 직원만 골라줘" 같은 백분위 필터링

```sql
WITH RankedEmp AS (
    SELECT 사원명, 급여,
        PERCENT_RANK() OVER (ORDER BY 급여 DESC) AS pct_rank
    FROM 사원
)
SELECT 사원명, 급여, ROUND(pct_rank * 100, 1) AS 상위_퍼센트
FROM RankedEmp
WHERE pct_rank <= 0.3;  -- 상위 30% 이내만
```

### CUME_DIST() — 누적 분포 비율

나보다 작거나 같은 값을 가진 행이 전체의 몇 %인지, **나 자신을 포함한 누적 비율**을 구한다.

> 💡 PERCENT_RANK와의 차이: PERCENT_RANK는 나보다 앞에 있는 사람 기준, CUME_DIST는 나를 포함한 누적 기준

```sql
SELECT 사원명, 급여,
    CUME_DIST() OVER (ORDER BY 급여 DESC) AS 누적_분포비율
FROM 사원;
-- 첫 번째 행도 최소 '1/전체행수' 이상 (절대 0이 될 수 없음)
```

### NTILE(N) — N개 등급으로 균등 분할

전체 데이터를 N개의 바구니로 **균등하게 쪼개서** 그룹 번호(1~N)를 부여한다.

```sql
SELECT 사원명, 급여,
    NTILE(4) OVER (ORDER BY 급여 DESC) AS 급여_등급_4분위
FROM 사원;
```

**⚠️ NTILE의 무자비한 분배 원리 2가지**

**① 나머지 우선 할당:** 행 수가 N으로 딱 안 떨어지면, 맨 위 그룹부터 순서대로 1개씩 더 욱여넣는다.

**② 동점자 강제 이산가족:** 급여가 완전히 같은 동점자라도 경계선에 걸리면 얄짤없이 다른 그룹으로 찢어버린다.

|사원명|급여|NTILE(2)|💡 해석|
|---|---|:-:|---|
|A|500|1그룹|총 5명 ÷ 2 = 몫 2, 나머지 1 → 1그룹에 3명 배정|
|B|400|1그룹|1그룹 정원 채우는 중|
|C|**300**|1그룹|💥 C와 D는 급여 동점이지만 1그룹 정원(3명)이 여기서 꽉 참|
|D|**300**|2그룹|동점임에도 정원 초과로 2그룹으로 밀려남|
|E|100|2그룹|나머지 없으므로 원래 몫인 2명 배정 완료|

### 비율 함수 결과값 범위 비교표

|함수|결과값 범위|1등 값|꼴찌 값|핵심 특징|
|---|:-:|:-:|:-:|---|
|`RATIO_TO_REPORT()`|0 ~ 1|-|-|파티션 내 합은 항상 1|
|`PERCENT_RANK()`|0.0 ~ 1.0|**0.0**|1.0|1등은 무조건 0|
|`CUME_DIST()`|(0, 1]|0 불가|**1.0**|나 포함 누적 → 0이 절대 안 됨|
|`NTILE(N)`|1 ~ N (정수)|1|N|결과는 항상 정수|

##  Windowing 절 — 세밀한 범위 지정

`ORDER BY`로 줄을 세웠다면, 이제 **"현재 행을 기준으로 어디서부터 어디까지 연산할 것인가?"** 를 돋보기처럼 미세하게 조절하는 기능이다. 이동 평균(최근 3일 평균 매출)을 구할 때 필수다.

### 키워드 암기 대신 해석하기

|키워드|해석|실무 의미|
|---|---|---|
|`UNBOUNDED`|경계가 없는, 무한한|**맨 처음** 또는 **맨 끝**|
|`PRECEDING`|이전의, 앞선|**~앞 (이전 행)**|
|`FOLLOWING`|다음의, 뒤따르는|**~뒤 (다음 행)**|
|`CURRENT ROW`|현재의 행|**현재 행**|

**조합 예시:**

- `UNBOUNDED PRECEDING` → 파티션의 **맨 첫 행**
- `UNBOUNDED FOLLOWING` → 파티션의 **맨 끝 행**
- `1 PRECEDING` → **바로 1줄 앞(위) 행**
- `BETWEEN 1 PRECEDING AND 1 FOLLOWING` → **앞 1줄 ~ 뒤 1줄** (총 3줄)

### ROWS vs RANGE

|구분|기준|특징|
|---|---|---|
|`ROWS`|**물리적 줄 수**|값이 같든 다르든 상관없이 진짜 '줄 수'로 셈|
|`RANGE`|**논리적 값**|현재 행과 '값이 같은' 동점자들을 하나의 덩어리로 묶어서 셈|

```sql
SELECT 날짜, 매출,
    -- ROWS: 물리적으로 정확히 '앞 1줄 ~ 뒤 1줄' (총 3줄) 이동 평균
    AVG(매출) OVER (
        ORDER BY 날짜
        ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING
    ) AS 이동평균_3일,

    -- RANGE: 날짜 중복이 있을 때, 같은 날짜 데이터를 묶어서 누적합 계산
    SUM(매출) OVER (
        ORDER BY 날짜
        RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS 완벽한_누적합

FROM 매출테이블;
```

### RANGE에서 숫자의 의미

`RANGE BETWEEN 숫자 PRECEDING`은 줄 수가 아니라 **값의 차이**를 기준으로 범위를 잡는다.

```sql
-- 나와 급여 차이가 2000 이내인 사람이 몇 명인지 센다
SELECT 사원명, 급여,
    COUNT(*) OVER (
        ORDER BY 급여
        RANGE BETWEEN 2000 PRECEDING AND CURRENT ROW
    ) AS 나와_급여비슷한_사람수
FROM 사원;
```

|사원명|급여|COUNT 결과|💡 계산 원리|
|---|---|:-:|---|
|박신입|2000|1|범위: 0 ~ 2000 → 본인 1명|
|김대리|3500|2|범위: 1500 ~ 3500 → 박신입, 본인|
|나보통|5000|2|범위: 3000 ~ 5000 → 김대리, 본인 (2000은 미포함)|
|최과장|8000|1|범위: 6000 ~ 8000 → 본인만|

## ⚡ [매우 중요] 기본값의 배신: 누적합이 한 줄씩 안 더해지고 점프하는 이유

`ORDER BY`만 적고 범위 지정을 생략하면, 컴퓨터는 암묵적으로 **`RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`** 를 기본으로 발동시킨다.

이 때문에 **같은 정렬 기준값(급여, 날짜 등)을 가진 행들이 덩어리로 묶여 한 번에 더해지면서 숫자가 껑충 점프**하는 현상이 발생한다.

**해결책:** `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`로 명시하면 물리적으로 한 줄씩 누적된다.

```sql
SELECT 사원명, 급여,
    -- 기본값 (RANGE) → 같은 급여는 묶어서 한 번에 더해버림
    SUM(급여) OVER (ORDER BY 급여) AS 누적합_기본값,

    -- 해결책 (ROWS) → 무조건 1줄씩 차곡차곡 더함
    SUM(급여) OVER (ORDER BY 급여 ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS 누적합_해결책
FROM 사원;
```

|사원명|급여|누적합 (RANGE 기본)|누적합 (ROWS 해결)|💡 왜 다를까?|
|---|---|:-:|:-:|---|
|박신입|100|100|100|첫 행, 동일|
|**나보통**|**200**|**500 💥**|**300**|**RANGE**: 동점자(200×2) 한꺼번에 → `100+(200×2)`<br>**ROWS**: 한 줄씩 → `100+200`|
|**김대리**|**200**|**500**|**500**|RANGE: 동점자 결과 유지 / ROWS: `300+200`|
|최과장|300|800|800|최종 누적합은 동일 (중간 과정만 다름)|

## 🚨 초보자 실수 모음

**"윈도우 함수를 `WHERE` 절에 넣어서 순위 1등인 것만 필터링할 수 있다?"** → 절대 불가능하다. 윈도우 함수는 `WHERE`·`GROUP BY`가 모두 끝난 뒤 `SELECT` 절에서만 사용할 수 있다. CTE 또는 인라인 뷰로 감싼 뒤 바깥에서 `WHERE rn = 1` 형태로 필터링해야 한다.

**"`SUM() OVER(ORDER BY 날짜)`를 썼는데 왜 누적합이 이상하게 나오나요?"** → 정렬 기준에 **중복된 날짜**가 있으면, 같은 날짜를 하나의 덩어리로 묶어 한 번에 더해버린다. 완벽한 행별 누적합을 원하면 값이 고유한 컬럼(로그 ID, 타임스탬프)으로 정렬 기준을 잡거나 `ROWS`를 명시해야 한다.
