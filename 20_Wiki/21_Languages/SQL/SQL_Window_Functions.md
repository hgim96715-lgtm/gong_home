# 윈도우 함수 (Window Functions)

## 한줄요약

행(Row)과 행 간의 관계를 정의하거나 그룹 내에서 연산을 수행하면서도, `GROUP BY`처럼 원본 데이터를 압축하여 날려버리지 않고 **원본 행을 그대로 유지한 채로** 결과를 옆에 나란히 붙여주는 강력한 분석 함수이다.

---
## 왜 필요한가?

**문제점:** 
- "부서별 평균 연봉"을 구하기 위해 `GROUP BY 부서`를 쓰면 부서별로 딱 1줄씩만 남고 사원들의 개별 데이터(이름, 직급 등)는 모두 사라져 버린다.

 **해결책:**
 -  윈도우 함수를 사용하면 개별 사원의 데이터를 그대로 보존하면서, 그 옆 컬럼에 "해당 사원이 속한 부서의 평균 연봉"이나 "부서 내 연봉 순위"를 계산해서 붙여넣을 수 있다.

---
## 실무맥락

데이터 엔지니어링 및 분석 실무에서 가장 많이, 그리고 가장 중요하게 쓰이는 문법이다. "웹사이트 접속자의 이전 페이지와 현재 페이지 체류 시간 비교(행순서)", "게시글의 일자별 누적 조회수(집계)", "상위 10% 우수 고객 추출(비율)" 등 복잡한 데이터 파이프라인(ETL) 요건을 단일 쿼리로 해결할 때 사용한다. SQLD 시험에서도 배점이 가장 높은 핵심 출제 영역이다.

---
## Code Core Points (기본 문법 구조)

윈도우 함수는 함수명 뒤에 반드시 **`OVER()`** 절이 따라붙는 것이 특징이다.

```sql
-- 기본 뼈대
함수명(인자) OVER (
    PARTITION BY 그룹핑할_컬럼      -- 1. 무엇을 기준으로 그룹(파티션)을 나눌 것인가? (선택)
    ORDER BY 정렬할_컬럼            -- 2. 파티션 내에서 어떤 순서로 정렬할 것인가? (선택/함수에 따라 필수)
    ROWS BETWEEN 시작 AND 끝        -- 3. 연산할 행의 범위를 어디까지로 제한할 것인가? (선택) - Windowing절
)
```

#### (`PARTITION BY` vs `GROUP BY`)

- `GROUP BY 부서`: 100명의 사원 데이터가 '영업부 1줄, 마케팅부 1줄'로 납작하게 압축되어 개별 사원 정보가 날아간다.
- `PARTITION BY 부서`: 100명의 사원 데이터(100줄)가 그대로 유지된다. 원본 데이터를 보존한 상태에서 내부적으로 부서별 칸막이만 쳐두고, 그 칸막이 안에서 1등부터 순위를 매기거나 평균을 구한다.

## Code Core Points (1. 순위 함수)

데이터의 순위를 매길 때 사용한다. 
공동 1위가 발생했을 때 그다음 순위를 어떻게 매길 것인지에 따라 함수를 골라 쓴다.

- **`ROW_NUMBER()`**: 중복을 무시하고 무조건 1, 2, 3, 4 고유한 순번을 부여
- **`RANK()`**: 공동 2위가 2명이면 다음은 4위가 된다. (1, 2, 2, 4 - 올림픽 메달 방식)
- **`DENSE_RANK()`**: 공동 2위가 2명이어도 다음은 3위가 된다. (1, 2, 2, 3 - 밀집된 순위)

```sql
SELECT 
    사원명, 부서, 급여,
    -- 1) ROW_NUMBER: 중복 무시하고 무조건 1, 2, 3, 4 고유한 순번 부여
    ROW_NUMBER() OVER(PARTITION BY 부서 ORDER BY 급여 DESC) AS row_num,
    
    -- 2) RANK: 공동 2위가 2명이면 다음은 4위 (1, 2, 2, 4) -> (올림픽 메달 방식)
    RANK() OVER(PARTITION BY 부서 ORDER BY 급여 DESC) AS rnk,
    
    -- 3) DENSE_RANK: 공동 2위가 2명이어도 다음은 3위 (1, 2, 2, 3) -> (밀집된 순위)
    DENSE_RANK() OVER(PARTITION BY 부서 ORDER BY 급여 DESC) AS dense_rnk
FROM 사원;
```

**결과 형태 비교 (부서가 같고, 급여가 300, 200, 200, 100인 경우)**

|**급여**|**ROW_NUMBER()**|**RANK()**|**DENSE_RANK()**|
|---|---|---|---|
|300|1|1|1|
|200|2|**2**|**2**|
|200|3|**2**|**2**|
|100|4|**4**|**3**|

### 언제 뭘 써야 하나? 실무 선택 기준

| 상황                        | 추천 함수          | 이유                |
| ------------------------- | -------------- | ----------------- |
| 무조건 1명만 뽑아야 할 때 (중복 허용 X) | `ROW_NUMBER()` | 동점이어도 1명만 나옴      |
| 동점자도 같은 등수로 인정해야 할 때      | `RANK()`       | 올림픽처럼 공동 2위 인정    |
| 등수 번호가 연속이어야 할 때          | `DENSE_RANK()` | 빈 등수 없이 1,2,3 연속  |
| 상위 N명 자르되 동점자 모두 포함       | `RANK()`       | 공동 3위 2명이면 둘 다 포함 |
| 페이지네이션 / 행 번호 매기기         | `ROW_NUMBER()` | 고유 번호가 필요하니까      |

### PARTITION BY vs GROUP BY — 가장 많이 헷갈리는 포인트

```sql
-- GROUP BY: 그룹을 "합쳐서" 1줄로 만든다 (원본 행이 사라짐)
SELECT 부서, AVG(급여)
FROM 사원
GROUP BY 부서;
-- 결과: 부서별로 딱 1줄씩만 나옴

-- PARTITION BY: 그룹 내에서 계산하되 "원본 행을 유지"한다
SELECT 사원명, 부서, 급여,
       AVG(급여) OVER(PARTITION BY 부서) AS 부서평균
FROM 사원;
-- 결과: 원본 행 수 그대로 유지되면서 옆에 부서평균이 붙어서 나옴
```

>**암기법:** `GROUP BY` → **줄이는** 것 (합쳐서 요약) `PARTITION BY` → **나란히 붙이는** 것 (원본 유지하면서 계산값 추가)

### ❗ RANK vs COUNT — 어떤 도구를 꺼낼까?

```sql
"순위 번호(1등, 2등...)" 가 결과에 필요한가?
        ↓ YES                        ↓ NO
     RANK 써라               COUNT(*) + GROUP BY + HAVING
```

| 문제에 이 단어가 보이면           | 꺼낼 도구                      |
| ----------------------- | -------------------------- |
| "1위", "상위 N위", "순위"     | `RANK` / `DENSE_RANK`      |
| "횟수", "몇 번", "등재", "건수" | `COUNT(*)`                 |
| "중복 없이 한 명만"            | `ROW_NUMBER()`             |
| "각 그룹에서 최고값"            | `RANK()` + `WHERE rnk = 1` |

```sql
-- ✅ "상위 50위 이내 2회 이상 등재" → COUNT로 충분
SELECT title, COUNT(*) AS cnt
FROM bestseller_top50
GROUP BY title
HAVING COUNT(*) >= 2
ORDER BY cnt DESC;

-- ✅ "각 부서에서 급여 1위만 뽑기" → RANK가 필요
WITH ranked AS (
    SELECT *, RANK() OVER(PARTITION BY 부서 ORDER BY 급여 DESC) AS rnk
    FROM 사원
)
SELECT * FROM ranked WHERE rnk = 1;
```

>📎더 자세한 내용은 [[SQL_Aggregate_GROUP_BY#① COUNT 삼형제 (가장 많이 헷갈림)|count 삼형제]] 참고
> COUNT 삼형제(`COUNT(*)` / `COUNT(컬럼)` / `COUNT(DISTINCT)`) 차이가 헷갈릴 때 확인.

## Code Core Points (2. 집계 함수)

우리가 아는 `SUM`, `AVG`, `MAX`, `MIN`, `COUNT`를 원본 행 유지 상태로 계산한다. 
`ORDER BY`를 넣느냐 빼느냐에 따라 **전체 합산**이 될 수도 있고 **누적 합산**이 될 수도 있다.

- **파티션 전체 집계 (`ORDER BY` 없음):** 파티션(또는 전체)의 총합, 최댓값 등을 구해서 **같은 파티션(그룹)에 속한 모든 행에 완벽하게 똑같은 값을 복사해서 붙여넣는다.**
- **누적 집계 (`ORDER BY` 있음):** 첫 행부터 현재 행까지 차곡차곡 누적해서 더하거나(누적합), 갱신(누적 최댓값/최솟값)해 나간다.
- **파티션 평균**: 부서별 평균 급여 등을 개인 데이터 옆에 붙여서 비교할 때 쓴다.

```sql
SELECT 
    부서, 사원명, 입사일자, 주문금액, 급여,
    
    -- 1. [파티션 전체 합산: SUM] ORDER BY 없음. 전체 총합이 모든 행에 동일하게 찍힌다.
    SUM(주문금액) OVER() AS 총_주문금액,
    
    -- 2. [파티션 평균: AVG] PARTITION BY만 있고 ORDER BY는 없음.
    -- 같은 부서 소속이라면, 사원명이나 급여가 달라도 '부서평균급여' 컬럼에는 모두 같은 숫자가 찍힌다.
    ROUND(AVG(급여) OVER(PARTITION BY 부서), 2) AS 부서평균급여,
    
    -- 3. [파티션 전체 최댓값: MAX] ORDER BY 없음.
    -- "우리 부서에서 제일 돈 많이 받는 사람의 월급"이 부서원 모두의 행에 똑같이 출력된다.
    -- (내 급여와 부서 내 최고 급여를 비교할 때 매우 유용함)
    MAX(급여) OVER(PARTITION BY 부서) AS 부서내_최고급여,
    
    -- 4. [누적 합산: SUM] ORDER BY 있음. (입사일자 순으로 누적)
    SUM(주문금액) OVER(ORDER BY 입사일자) AS 누적_주문금액,
    
    -- 5. [누적 최솟값: MIN] ORDER BY 있음. 
    -- "오늘(입사일자)까지 입사한 사람 중 역대 가장 적은 급여"가 얼마인지 날짜가 지날 때마다 갱신된다.
    MIN(급여) OVER(ORDER BY 입사일자) AS 지금까지_최저급여
FROM 사원테이블;
```

| 사원명     | 입사일 | 급여  | MAX(급여)<br>OVER(PARTITION BY 부서)<br>※ ORDER BY 없음 | MIN(급여)<br>OVER(ORDER BY 입사일)<br>※ ORDER BY 있음 | 💡 계산 원리                                                                                              |
| ------- | --- | --- | ------------------------------------------------- | ---------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| **김신입** | 1/1 | 300 | **500**                                           | **300**                                        | • **MAX**: 영업부 전체 최고 급여 = 500<br>• ORDER BY 없음 → 전원 동일 값<br><br>• **MIN**: 첫 행<br>→ 비교 대상 없음 → 본인 300 |
| **이대리** | 1/2 | 500 | **500**                                           | **300**                                        | • **MAX**: 여전히 전체 최고 = 500<br><br>• **MIN**: 1/1~1/2 중 최저<br>→ 김신입 300 유지                             |
| **박과장** | 1/3 | 400 | **500**                                           | **300**                                        | • **MAX**: 계속 500 복붙<br><br>• **MIN**: 1/3까지도 최저는 300                                                 |
| **최인턴** | 1/4 | 200 | **500**                                           | **200 🔥**                                     | • **MAX**: ORDER BY 없으므로 동일<br><br>• **MIN**: 더 낮은 값 등장<br>→ **역대 최저 갱신 (200)**                       |

## Code Core Points (3. 행 순서 함수) -> (SQL Server는 미지원)

현재 행을 기준으로, 이전 행이나 다음 행의 값을 당겨오거나 밀어낼 때 사용한다. 
시계열 데이터(날짜별 변화량) 분석의 핵심이다.

- **`LAG(컬럼명, [건너뛸_행_수], [NULL대체_기본값])`**: 이전 행의 값을 끌어온다. 숫자를 생략하면 무조건 바로 윗줄(1칸 전)을 가져오며, 값이 없으면 NULL을 띄운다. (예: 7을 넣으면 일주일 전 데이터를 가져온다.)
- **`LEAD(컬럼명, [건너뛸_행_수], [NULL대체_기본값])`**: 다음 행의 값을 끌어온다.
- **`FIRST_VALUE()`**: 파티션 내에서 정렬 기준 첫 번째 값을 가져온다.
- **`LAST_VALUE()`**: 파티션 내에서 정렬 기준 마지막 값을 가져온다.
	-  윈도우 함수는 현재 행까지만 제한하기 때문에, LAST_VALUE는 범위 지정 키워드 사용 필요
	- ⚠️ **[매우 중요]** 진정한 의미의 '마지막 값'을 가져오려면 윈도우 범위를 반드시 **`UNBOUNDED FOLLOWING`** 으로 끝까지 열어주어야 한다!

```sql
SELECT 
    접속일자, 접속자수,
    -- 1) 기본 LAG: 바로 윗줄(어제) 가져오기 (생략 시 1칸 전, 대체값 NULL)
    LAG(접속자수) OVER(ORDER BY 접속일자) AS 전일_접속자수,
    
    -- 💡 심화 LAG: 7칸 위(일주일 전) 데이터 당겨오기. 만약 7일 전 기록이 없어서 NULL이 뜬다면 숫자 0으로 덮어써라!
    LAG(접속자수, 7, 0) OVER(ORDER BY 접속일자) AS 일주일전_접속자수,
    
    -- 2) LEAD: 바로 아랫줄(내일) 가져오기
    LEAD(접속자수) OVER(ORDER BY 접속일자) AS 익일_접속자수,
    
    -- 3) FIRST_VALUE: (SQL Server는 미지원)
    -- 윈도우 기본값이 '현재 행까지'라도, 어차피 '첫 번째 값'은 무조건 범위 안에 있으므로 정상 작동한다.
    FIRST_VALUE(접속자수) OVER(PARTITION BY 월 ORDER BY 접속일자) AS 월초_접속자수,
    
    -- 4) 💥 LAST_VALUE: (SQL Server는 미지원)
    -- 범위를 '파티션의 끝(UNBOUNDED FOLLOWING)'까지 강제로 열어주지 않으면, 
    -- 윈도우가 '현재 행'에서 닫혀버려서 결국 자기 자신을 마지막 값으로 출력해버린다!
    LAST_VALUE(접속자수) OVER(
        PARTITION BY 월 
        ORDER BY 접속일자 
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS 월말_접속자수
FROM 트래픽로그;
```

## Code Core Points (4. 비율 함수)-> (SQL Server는 미지원)


1. **`RATIO_TO_REPORT()`**: 파티션 내 전체 합계(SUM) 대비 현재 행의 값이 차지하는 **백분율(비율)** 을 구한다.
2. **`NTILE(N)`**: 전체 데이터를 N개의 등급(바구니)으로 **균등하게 쪼갠다.** (예: 4등분 하여 1분위~4분위 나누기)
3. **`PERCENT_RANK()`**: 그룹 내의 **백분율 순위**를 구한다. (1등이 0, 꼴찌가 1로 계산됨)
4. **`CUME_DIST()`**: 현재 행보다 작거나 같은 건수에 대한 **누적 분포 비율**을 구한다. (0초과 ~ 1이하 반환)

```sql
SELECT 
    사원명, 급여,
    -- 1) [SQLD 시험용 / Oracle 전용] RATIO_TO_REPORT 오리지널 문법
    -- 💡 주의: 괄호 안에 타겟 컬럼(급여)을 넣고, OVER() 안에는 절대 ORDER BY를 쓰지 않는다!
    RATIO_TO_REPORT(급여) OVER() AS 오라클_급여비율,

    -- 1-1) [실무용 / PostgreSQL, SQL Server] RATIO_TO_REPORT 대체 구현
    -- 내 급여를 전체 급여 합계(SUM OVER)로 직접 나눈다! (소수점 보기 좋게 ROUND 처리)
    ROUND(급여 / SUM(급여) OVER(), 2) AS 포그레_급여비율,
    
    -- 2) NTILE(N): 전체 데이터를 4개의 바구니로 쪼개서 그룹 번호(1~4) 부여
    NTILE(4) OVER(ORDER BY 급여 DESC) AS 급여_등급_4분위,
    
    -- 3) PERCENT_RANK(): 1등은 0, 꼴찌는 1로 계산되는 백분율 순위
    PERCENT_RANK() OVER(ORDER BY 급여 DESC) AS 백분율_순위,
    
    -- 4) CUME_DIST(): 나와 급여가 같거나 큰 사람이 전체의 몇 %인지 누적 분포 계산
    CUME_DIST() OVER(ORDER BY 급여 DESC) AS 누적_분포비율
FROM 사원;
```

### NTILE(N)의 무자비한 분배 원리

`NTILE`은 값이 같든 다르든 상관없이, 오직 **물리적인 줄 수(Row)** 를 기준으로 칼같이 잘라버리는 무서운 특징 2가지가 있어.

**나머지 우선 할당:**
행의 수가 N으로 딱 안 떨어지면(나머지가 생기면), 맨 위의 1그룹부터 순서대로 1개씩 더 욱여넣는다.

**동점자 강제 이산가족:**
급여가 완벽히 똑같은 동점자라도 하필 바구니를 나누는 경계선에 걸려버리면, 얄짤없이 하나는 앞 그룹, 하나는 뒷 그룹으로 찢어버린다.

| **사원명** | **급여**  | **NTILE(2)** | **💡 어떻게 쪼개진 걸까? (해석)**                                          |
| ------- | ------- | ------------ | ---------------------------------------------------------------- |
| **A**   | 500     | **1그룹**      | 총 5명을 2조로 나눔 (몫: 2, 나머지: 1).<br>1그룹에 **기본 2명 + 남은 1명**을 추가 배정.   |
| **B**   | 400     | **1그룹**      | (1그룹 정원 채우는 중)                                                   |
| **C**   | **300** | **1그룹**      | 💥 **[동점자 이산가족 발생]** C와 D는 급여가 300으로 같지만, 1그룹 정원(3명)이 여기서 꽉 차버림! |
| **D**   | **300** | **2그룹**      | 정원이 끝났으므로, 동점자임에도 불구하고 D는 얄짤없이 2그룹으로 밀려난다.                       |
| **E**   | 100     | **2그룹**      | 남은 나머지가 없으므로 원래 몫인 2명 배정 완료.                                     |

### 비율 함수 4대장 결과값 범위(Range) 비교표

| 함수명                     | 결과값 범위                | 💡 실무 / 시험 해석                                               |
| ----------------------- | --------------------- | ----------------------------------------------------------- |
| **`RATIO_TO_REPORT()`** | **0 ~ 1**<br>(총합 = 1) | • 전체 합을 1(100%)로 가정<br>• 내가 차지하는 비율<br>• 파티션 내 합은 항상 1      |
| **`NTILE(N)`**          | **1 ~ N**<br>(정수)     | • 데이터를 N개 그룹으로 분할<br>• 결과는 항상 정수<br>• `NTILE(4)` → 1~4 중 하나 |
| **`PERCENT_RANK()`**    | **0 ~ 1**<br>(0부터 시작) | • 내 위에 몇 명이 있는지<br>• 1등 → 0<br>• 꼴찌 → 1                     |
| **`CUME_DIST()`**       | **(0, 1]**<br>(0 불가)  | • 나 포함 누적 비율<br>• 최소값도 나 자신 포함<br>• 마지막 행은 항상 1             |

## Code Core Points (5. 세밀한 범위 지정: Windowing 절)

`ORDER BY`로 파티션 내에서 줄을 세웠다면, 이제 **"현재 행을 기준으로 도대체 어디서부터 어디까지 연산할 것인가?"** 그 범위를 돋보기처럼 미세하게 조절하는 기능이다. 이동 평균선(예: 최근 3일 평균 매출)을 구할 때 필수적이다.

**암기 대신 해석하기**

|**영단어 (Keyword)**|**해석**|**실무 의미**|
|---|---|---|
|**UNBOUNDED**|경계가 없는, 무한한|**맨 처음** 또는 **맨 끝**|
|**PRECEDING**|이전의, 앞선|**~앞 (이전 행)**|
|**FOLLOWING**|다음의, 뒤따르는|**~뒤 (다음 행)**|
|**CURRENT ROW**|현재의 행|**현재 행**|

**조합 예시:**

- `UNBOUNDED PRECEDING` 👉 한계 없이 이전으로 = **"파티션의 맨 첫 행"**
- `UNBOUNDED FOLLOWING` 👉 한계 없이 이후로 = **"파티션의 맨 끝 행"**
- `1 PRECEDING` 👉 **"바로 1줄 앞(위) 행"**
- `BETWEEN 1 PRECEDING AND 1 FOLLOWING` 👉 **"바로 1줄 앞부터 ~ 바로 1줄 뒤까지"** (총 3줄)

### **ROWS vs RANGE**

범위를 지정할 때 단위를 `ROWS`로 할지, `RANGE`로 할지 결정해야 한다.

- **`ROWS` (물리적 행 기준):** 값이 같든 다르든 상관없이, 내 눈에 보이는 **진짜 '줄 수'** 를 기준으로 셈을 한다.
- **`RANGE` (논리적 값 기준):** 현재 행과 **'값이 같은'** 동점자들은 모두 하나의 덩어리로 묶어서 셈을 한다.

```sql
SELECT 
    날짜, 매출,
    -- 1) [ROWS] 물리적으로 정확히 '최근 3일(어제, 오늘, 내일)'의 평균 구하기
    AVG(매출) OVER(
        ORDER BY 날짜 
        ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING
    ) AS 이동평균_3일,
    
    -- 2) [RANGE] 날짜가 같은(중복된) 데이터가 있을 때, 그 데이터들을 묶어서 누적합 구하기
    SUM(매출) OVER(
        ORDER BY 날짜 
        RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS 완벽한_누적합
FROM 매출테이블;
```

### `ROWS`에서의 숫자 vs `RANGE`에서의 숫자

단위가 `ROWS`인지 `RANGE`인지에 따라 뒤에 붙는 '숫자'의 의미가 완전히 달라집니다

- **`ROWS 1 PRECEDING` (물리적 줄 수):** 내 값과 상관없이, 내 눈에 보이는 **"내 바로 위쪽 1줄"** 을 뜻합니다
- **`RANGE 2000 PRECEDING` (논리적 값의 차이):** 물리적인 줄 수가 아닙니다! 정렬 기준 컬럼(예: 급여)을 기준으로 **"내 급여보다 딱 2000 적은 값까지"** 를 뜻합니다. 즉, 내 급여가 5000이라면 3000 ~ 5000 사이의 값을 가진 모든 사람을 범위로 묶으라는 뜻이 됩니다.

```sql
SELECT 
    사원명, 
    급여,
    -- 내 급여를 기준으로, [내 급여 - 2000] 부터 [내 급여] 범위 안에 들어오는 사람이 몇 명인지 센다.
    COUNT(*) OVER(
        ORDER BY 급여 
        RANGE BETWEEN 2000 PRECEDING AND CURRENT ROW
    ) AS "나와_급여비슷한_사람수"
FROM 사원;
```

**예시 표 (RANGE 2000 PRECEDING)**

| 사원명 | 급여 | COUNT(*) 결과 | 💡 계산 원리 (해석) |
|------|----|-------------|------------------|
| 박신입 | 2000 | 1 | • 기준 급여: 2000<br>• 범위: 0 ~ 2000<br>• 포함 인원: 본인 1명 |
| 김대리 | 3500 | 2 | • 기준 급여: 3500<br>• 범위: 1500 ~ 3500<br>• 포함 인원: 박신입, 본인 |
| **나보통** | **5000** | **2** | • 기준 급여: 5000<br>• 범위: **3000 ~ 5000**<br>• 포함 인원: 김대리, 본인<br>• 2000은 범위 미포함 |
| 최과장 | 8000 | 1 | • 기준 급여: 8000<br>• 범위: 6000 ~ 8000<br>• 포함 인원: 본인 |

>**요약:** `RANGE BETWEEN 숫자 PRECEDING`은 무조건 맨 위쪽 끝행부터 전부 다 비교하는 게 아니라, 오직 **"(내 값 - 숫자) 이상 ~ (내 값) 이하"** 라는 조건 필터를 걸어 그 범위에 해당하는 행만 쏙쏙 골라 묶는 아주 스마트한 기능입니다.

## [매우 중요] 기본값의 배신: 왜 누적합이 한 줄씩 안 더해지고 점프할까?

윈도우 함수에서 `ORDER BY`만 적고 뒤에 범위 지정을 생략하면, 컴퓨터는 암묵적으로 **`RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`** 를 기본값으로 발동시킵니다.

이 때문에 **"같은 정렬 기준값(급여, 날짜 등)을 가진 행들은 서로 손을 잡고 하나의 덩어리가 되어 한 방에 더해진다(숫자가 껑충 점프한다)"** 는 현상이 발생합니다.

**해결책:** 중복된 값이 있더라도 물리적으로 얄짤없이 '진짜 한 줄씩' 누적합을 구하고 싶다면, 괄호 안에 `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` 라고 **`ROWS`** 옵션을 직접 명시해 주어야 합니다.

```sql
SELECT 
    사원명, 
    급여,
    -- 1. [기본값 발동 - RANGE] 범위 지정 생략. 같은 급여는 묶어서 한 번에 더해버린다.
    SUM(급여) OVER(ORDER BY 급여) AS 누적합_기본값,
    
    -- 2. [해결책 - ROWS] 무조건 내 눈에 보이는 1줄씩 차곡차곡 더한다.
    SUM(급여) OVER(ORDER BY 급여 ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS 누적합_해결책
FROM 사원;
```

**결과 비교 (급여가 200으로 같은 '나보통'과 '김대리'의 누적합 차이 보기)**

| 사원명 | 급여 | 누적합 (RANGE 기본) | 누적합 (ROWS 해결) | 💡 왜 숫자가 다를까? |
|------|----|------------------|-----------------|------------------|
| 박신입 | 100 | 100 | 100 | • 첫 행<br>• RANGE / ROWS 동일 |
| **나보통** | **200** | **500 💥** | **300** | • **RANGE**: 급여 200 동점자 묶음<br>→ `100 + (200 × 2)`<br><br>• **ROWS**: 앞 행 + 현재 행<br>→ `100 + 200` |
| **김대리** | **200** | **500** | **500** | • **RANGE**: 동점자 포함 결과 유지<br><br>• **ROWS**: `300 + 200` |
| 최과장 | 300 | 800 | 800 | • 최종 누적합은 동일<br>• 중간 과정만 다름 |

## 초보자 실수

**"윈도우 함수를 `WHERE` 절에 넣어서 순위가 1등인 것만 필터링할 수 있다?"**
- **절대 불가능하다.** 윈도우 함수는 SQL 실행 순서상 `WHERE`나 `GROUP BY`가 모두 끝난 뒤인 `SELECT` 절이나 `ORDER BY` 절에서만 사용할 수 있다.
- 1등만 뽑고 싶다면, 윈도우 함수를 쓴 쿼리 전체를 인라인 뷰(서브쿼리)나 `WITH` 절(CTE)로 한 번 감싼 뒤, 바깥쪽 쿼리의 `WHERE` 절에서 `WHERE row_num = 1` 형태로 필터링해야 한다.

**"`SUM() OVER(ORDER BY 날짜)`를 썼는데 왜 누적합이 안 나오고 이상하게 계산되나요?"**
- 정렬 기준(`ORDER BY`) 데이터에 **중복된 날짜**가 있을 경우, 윈도우 함수는 같은 날짜를 하나의 덩어리로 묶어서 한 번에 더해버린다. 
- 완벽하게 한 행씩 누적합을 구하려면 값이 고유한 컬럼(예: 로그 ID, 시간단위 타임스탬프)으로 정렬 기준을 꼼꼼하게 잡아주어야 한다.
