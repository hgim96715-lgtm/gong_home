## 한줄 요약 

**"현실 세계의 복잡한 정보를 약속된 표기법(ERD)으로 단순화, 추상화, 명확화하여 설계도를 그리는 과정."** 

>"데이터베이스의 모델링은 '현실세계를 단순화하여 표현하는 기법'"⭐️

---
##  Why: 왜 필요한가?

* **문제:** 설계도 없이 DB를 만들면 데이터 중복, 불일치(모순), 성능 저하가 발생하며, 업무가 바뀌었을 때 수정하기가 매우 어렵습니다. 
* **해결:** 모델링을 통해 업무 흐름을 명확히 정의하고, 시스템의 안정성과 유연성을 확보합니다. 개발자와 현업 담당자가 소통하는 "공통 언어" 역할을 합니다.

---
##  Practical Context (실무 활용)

* **신규 프로젝트:** "수강신청 시스템"을 만들 때, 바로 코딩(SQL)부터 치는 게 아니라 "학생", "강의", "신청"이라는 개념을 먼저 그림(ERD)으로 그립니다.
* **리버스 엔지니어링:** 이미 돌아가는 DB를 보고 "이게 도대체 무슨 구조지?" 하고 분석하여 다시 설계도로 역추적할 때 사용합니다.

##  Theory Core Points (핵심 이론)

### A. 데이터 모델링의 3대 특징 (Characteristics) ⭐️

현실 세계를 모델로 만들 때 반드시 지켜야 할 3가지 규칙입니다.

> **💡 암기 팁: "추.단.명"

1. **추상화 (Abstraction):**
    - 불필요한 세부 사항은 버리고, **핵심적인 특징(공통점)만 뽑아서 간략하게** 표현합니다.
    - "현실 세계를 일정한 형식에 맞춰 표현하는 것" 자체가 추상화입니다.

2. **단순화 (Simplicity):**
    - 복잡한 현실 세계를 약속된 규약(Notation)이나 표기법으로 표현하여, 누구나 **쉽게 이해할 수 있도록** 만듭니다.

3. **명확화 (Clarity):**
    - 애매모호함을 제거하고, **누가 봐도 딱 한 가지 의미로만 해석되게** 정확하게 정의합니다.
    - "이게 A야, B야?" 하는 모호성(Ambiguity)이 없어야 합니다.

### B.데이터 모델링의 3대 관점 (View)

모델링은 단순히 데이터만 보는 게 아니라, **"데이터가 어떻게 흐르는지"** 까지 봐야 합니다.

1. **데이터 관점 (Data):** "무엇(What)을 저장하는가?"
    - 업무와 데이터 간의 관계를 분석 (Structure).

2. **프로세스 관점 (Process):** "무엇(What)을 하는가?"
    - 업무가 실제로 어떻게 흘러가는지 분석 (Flow).

3. **데이터와 프로세스의 상관 관점 (Interaction):** "누가(Who) 무엇을(What) 건드리는가?"
    - 프로세스와 데이터 간의 상호작용 분석 (CRUD 분석).

### C. 데이터 모델링의 유의사항 (데이터 품질 보장) ⭐️

데이터 품질을 망치는 **3가지 적(중복, 비유연성, 비일관성)** 을 피하고, 다음 3가지를 지켜야 합니다.

> 암기 팁 ! "유.유.일 (유일, 유연, 일관)'

1. **유일성 (Uniqueness):** "중복 저장 금지"
    - 같은 데이터가 여러 곳에 저장되면 안 됩니다. (하나를 고쳤는데 다른 곳은 안 고쳐지는 '모순' 방지)
  
2. **유연성 (Flexibility):** "변화에 유연하게"
    - 데이터와 프로세스를 분리해서 설계합니다. 업무 프로세스가 조금 바뀌어도 데이터 구조가 와르르 무너지지 않아야 합니다.

3. **일관성 (Consistency):** "앞뒤가 맞아야 함"
    - 데이터 간의 관계(Relationship)를 명확히 정의해서, 데이터끼리 모순이 없어야 합니다.

### D. 모델링의 3단계 (추상화 수준: 높음 → 낮음) 

> 암기팁 "개논물" , **개념적(Abstract) → 논리적(Standard) → 물리적(Concrete)**

1.  **개념적 모델링 (Conceptual):** "무엇(What)을 할 것인가?"
    * 핵심 엔터티(Student, Course)와 관계 도출.
    * 추상화 수준이 가장 높음.
    * 산출물: 주제 영역 정의서, 핵심 ERD.

2.  **논리적 모델링 (Logical):** "어떻게(How) 구조화할 것인가?" **(가장 중요!⭐️)**
    * 식별자 확정, 정규화(Normalization) 수행.
    * 누가 봐도 이해할 수 있는 표준 구조 (재사용성 높음).

3.  **물리적 모델링 (Physical):** "실제 성능(Performance)은?"
    * 실제 특정 DBMS(Oracle, MySQL)에 맞춰 테이블 생성.
    * 인덱스, 파티셔닝, 반정규화 수행.

### E. 스키마의 3계층 (ANSI-SPARC)

데이터베이스를 보는 **"관점(View)"** 에 따라 3가지로 나눕니다. 

>암기팁: "외.개.내"

1.  **외부 스키마 (External):** 사용자 관점. (나한테 보이는 화면)
    * 예: 학생은 '내 성적'만 보이고, 교수는 '전체 학생 성적'이 보임.

2.  **개념 스키마 (Conceptual):** 조직 전체 관점. (통합된 설계도)
    * 예: 학교 전체의 모든 데이터 구조와 관계. (DBA의 관점)

3.  **내부 스키마 (Internal):** 저장 장치 관점. (물리적 저장)
    * 예: 데이터가 하드디스크 어디에, 어떤 인덱스로 저장되는지.

| **계층**                            | **키워드 (암기)** | **누가 보나요?**    | **몇 개인가?**    | **설명**                                                  |
| --------------------------------- | ------------ | -------------- | ------------- | ------------------------------------------------------- |
| **1. 외부 스키마**<br><br>(External)   | **개인, 서브**   | **사용자, 프로그래머** | **여러 개 (N)**  | "나는 내 것만 볼래."<br><br>개인별로 필요한 데이터만 보여주는 사용자 관점 (서브 스키마) |
| **2. 개념 스키마**<br><br>(Conceptual) | **통합, 전체**   | **조직 전체, DBA** | **오직 하나 (1)** | "학교 전체의 설계도."<br><br>모든 사용자가 공유하는 **통합된** 데이터 구조        |
| **3. 내부 스키마**<br><br>(Internal)   | **물리, 저장**   | **시스템 설계자**    | **오직 하나 (1)** | "하드디스크에 어떻게 넣지?"<br><br>데이터가 물리적으로 **저장**되는 구조          |

**⚠️주의 (시험 함정):**

- "모든 사용자가 공유하는 뷰"라고 하면 **개념 스키마**입니다.
- "개별 사용자가 보는 뷰"라고 하면 **외부 스키마**입니다.
- **DBA**는 전체를 봐야 하니까 **개념 스키마**를 관리하고, **프로그래머**는 자기 프로그램만 짜면 되니까 **외부 스키마**를 봅니다.


### F. 데이터 독립성 (Data Independence) ⭐️

3계층 구조가 보장하는 **"방화벽"** 같은 역할입니다.
하위 단계의 구조를 변경해도, 상위 단계는 영향을 받지 않아야 한다는 원칙입니다.

| **구분**                           | **설명**                                  | **변경 영향 범위**                     | **예시 (상황)**                                           |
| -------------------------------- | --------------------------------------- | -------------------------------- | ----------------------------------------------------- |
| **1. 논리적 독립성**<br><br>(Logical)  | **개념 스키마**가 바뀌어도<br><br>**외부 스키마**는 멀쩡함 | **개념 ↔ 외부**<br><br>(논리적 구조 변경 시) | "테이블을 2개로 쪼개거나 통합해도,<br><br>사용자가 보는 화면(View)은 그대로다."  |
| **2. 물리적 독립성**<br><br>(Physical) | **내부 스키마**가 바뀌어도<br><br>**개념 스키마**는 멀쩡함 | **내부 ↔ 개념**<br><br>(물리적 구조 변경 시) | "데이터를 HDD에서 SSD로 옮기거나<br><br>인덱스를 바꿔도, 테이블 구조는 그대로다." |

#### 💡 [암기 비법] "논.개.외" vs "물.내.개"

시험에서는 "어떤 스키마가 바뀌었을 때 어디가 보호받느냐?"를 물어봅니다.

1. **논리적 독립성 (논.개.외):**
    - **논**리적 구조(**개**념 스키마)를 고쳐도, **외**부 스키마(사용자)는 모르게 해라.
    - _비유:_ 식당 주방(개념) 리모델링을 해도, 손님 메뉴판(외부)은 그대로 유지됨.

2. **물리적 독립성 (물.내.개):**
    - **물**리적 구조(**내**부 스키마)를 고쳐도, **개**념 스키마(논리)는 모르게 해라.
    - _비유:_ 식재료 창고 위치(내부)를 지하 1층에서 2층으로 옮겨도, 주방(개념) 요리 방식은 안 바뀜.

##  Common Beginner Misconceptions (주의사항)

* **오해:** "모델링은 DB 개발자만 하는 것이다?"
    * 👉 **아닙니다.** 모델링은 업무(Business)를 분석하는 과정이므로, 기획자나 현업 담당자의 업무 이해도가 가장 중요합니다.

* **오해:** "무조건 정규화를 많이 하면 좋다?"
    * 👉 **아닙니다.** 논리적 모델링에서는 정규화가 필수지만, 물리적 모델링 단계에서는 성능을 위해 **반정규화(De-normalization)** 를 하기도 합니다.
    * (조인이 너무 많아지면 느려지니까요!)
