# Data Modeling: 데이터 모델링의 모든 것

> **핵심 요약** **"현실 세계의 복잡한 정보를 약속된 표기법(ERD)으로 단순화, 추상화, 명확화하여 설계도를 그리는 과정"** "데이터베이스의 모델링은 현실세계를 단순화하여 표현하는 기법" ⭐

---

## Why? 왜 필요한가?

- **문제:** 설계도 없이 DB를 만들면 데이터 중복, 불일치(모순), 성능 저하가 발생하며 업무가 바뀌었을 때 수정하기가 매우 어렵습니다.
- **해결:** 모델링을 통해 업무 흐름을 명확히 정의하고, 시스템의 안정성과 유연성을 확보합니다. 개발자와 현업 담당자가 소통하는 **"공통 언어"** 역할을 합니다.

**실무 활용:**

- **신규 프로젝트:** "수강신청 시스템"을 만들 때 바로 SQL부터 치는 게 아니라 "학생", "강의", "신청"이라는 개념을 먼저 ERD로 그립니다.
- **리버스 엔지니어링:** 이미 돌아가는 DB를 보고 다시 설계도로 역추적할 때 사용합니다.

---

## 데이터 모델링의 3대 특징 ⭐

현실 세계를 모델로 만들 때 반드시 지켜야 할 3가지 규칙입니다.

> 💡 **암기 팁: "추.단.명"**

|특징|설명|
|:--|:--|
|**추상화** (Abstraction)|불필요한 세부 사항은 버리고, **핵심적인 특징(공통점)만 뽑아서** 간략하게 표현|
|**단순화** (Simplicity)|복잡한 현실 세계를 약속된 규약(Notation)으로 표현하여 **누구나 쉽게 이해**할 수 있도록 만듦|
|**명확화** (Clarity)|애매모호함을 제거하고, **누가 봐도 딱 한 가지 의미**로만 해석되게 정확히 정의|

---

## 데이터 모델링의 3대 관점

모델링은 단순히 데이터만 보는 게 아니라, **"데이터가 어떻게 흐르는지"** 까지 봐야 합니다.

|관점|핵심 질문|분석 내용|
|:--|:--|:--|
|**데이터 관점** (Data)|무엇(What)을 저장하는가?|업무와 데이터 간의 관계 분석 (Structure)|
|**프로세스 관점** (Process)|무엇(What)을 하는가?|업무가 실제로 어떻게 흘러가는지 분석 (Flow)|
|**상관 관점** (Interaction)|누가(Who) 무엇을(What) 건드리는가?|프로세스와 데이터 간의 상호작용 분석 (CRUD 분석)|

---

##  데이터 모델링의 유의사항 ⭐

데이터 품질을 망치는 **3가지 적(중복, 비유연성, 비일관성)** 을 피해야 합니다.

> 💡 **암기 팁: "유.유.일" (유일, 유연, 일관)**

|유의사항|설명|위반 시 문제|
|:--|:--|:--|
|**유일성** (Uniqueness)|같은 데이터가 여러 곳에 저장되면 안 됨|하나를 고쳤는데 다른 곳은 안 고쳐지는 '모순' 발생|
|**유연성** (Flexibility)|데이터와 프로세스를 분리해서 설계|업무 프로세스가 조금 바뀌어도 데이터 구조가 와르르 무너짐|
|**일관성** (Consistency)|데이터 간의 관계를 명확히 정의|데이터끼리 모순이 생겨 앞뒤가 안 맞는 상황 발생|

## 5모델링의 3단계 

> 💡 **암기 팁: "개.논.물"** — 추상화 수준: 높음 → 낮음

```
[현실 세계]
     ↓
┌─────────────────────────────────┐
│  STEP 1. 개념적 모델링           │  ← 추상화 수준 가장 높음
│  "무엇을 저장할까?" (업무 중심)  │
└──────────────┬──────────────────┘
               ↓
┌─────────────────────────────────┐
│  STEP 2. 논리적 모델링    ⭐    │  ← 가장 중요!
│  "어떻게 구조화할까?" (표준)    │
└──────────────┬──────────────────┘
               ↓
┌─────────────────────────────────┐
│  STEP 3. 물리적 모델링           │  ← 추상화 수준 가장 낮음
│  "실제 DB에 어떻게 넣을까?"     │
└─────────────────────────────────┘
     ↓
[실제 데이터베이스]
```

---

## ① 개념적 모델링 (Conceptual) — "무엇(What)을 할 것인가?"

**업무 중심적, 전사적 관점에서 핵심 개념만 뽑아내는 단계**입니다.

- 구체적인 기술 용어 없이 **핵심 엔터티(Entity)와 관계(Relationship)** 만 도출합니다.
- 시스템으로 구축하고자 하는 업무 전체를 **큰 그림**으로 표현합니다.
- **추상화 수준이 가장 높습니다.** (세부 속성, PK 같은 건 신경 안 씀)
- 주요 표기법: **Peter Chen 표기법** (마름모 관계 표현)

|구분|내용|
|:--|:--|
|**핵심 질문**|"이 시스템에서 관리해야 할 핵심 대상이 무엇인가?"|
|**주요 작업**|핵심 엔터티 도출, 엔터티 간 관계 정의|
|**산출물**|주제 영역 정의서, 핵심 ERD (개념 ERD)|
|**참여자**|업무 전문가(현업), 데이터 아키텍트|
|**예시**|`학생`, `강의`, `교수` 라는 개념과 "학생은 강의를 수강한다" 는 관계만 표현|

---

## ② 논리적 모델링 (Logical) — "어떻게(How) 구조화할 것인가?" ⭐ 가장 중요

**Key, 속성(Attribute) 등을 구체적이고 정확하게 표현하는 단계**입니다.

- 특정 DBMS에 종속되지 않는 **표준화된 구조**로 표현합니다.
- **식별자(PK/FK) 확정, 정규화(Normalization) 수행**이 핵심 작업입니다.
- 누가 봐도 이해할 수 있어 **재사용성이 높습니다.**
- 주요 표기법: **Barker 표기법**, IE 표기법

|구분|내용|
|:--|:--|
|**핵심 질문**|"각 엔터티에 어떤 속성이 있고, 어떻게 식별하며, 어떻게 연결되는가?"|
|**주요 작업**|속성 정의, PK/FK 확정, 정규화, 관계 차수(1:1 / 1:N / M:N) 정의|
|**산출물**|논리 ERD, 속성 정의서|
|**참여자**|데이터 모델러, DBA|
|**예시**|`학생(학번 PK, 이름, 학과코드 FK)`, `수강(학번 PK·FK, 과목코드 PK·FK, 성적)` 으로 구체화|

---

## ③ 물리적 모델링 (Physical) — "실제 성능(Performance)은?"

**실제 데이터베이스에 반영할 수 있도록 물리적 저장 구조까지 설계하는 단계**입니다.

- **특정 DBMS(Oracle, MySQL 등)의 문법과 특성**에 맞춰 테이블을 생성합니다.
- 성능을 위한 **인덱스, 파티셔닝, 반정규화**를 수행합니다.
- **추상화 수준이 가장 낮고**, 가장 구체적입니다.

|구분|내용|
|:--|:--|
|**핵심 질문**|"이 구조가 실제 운영 환경에서 빠르게 동작하는가?"|
|**주요 작업**|테이블/컬럼 데이터 타입 확정, 인덱스 설계, 파티셔닝, 반정규화|
|**산출물**|물리 ERD, DDL(CREATE TABLE 스크립트)|
|**참여자**|DBA, 개발자|
|**예시**|`VARCHAR2(20)`, `NUMBER(10)` 같은 실제 타입 지정, 인덱스 추가|

---

## 3단계 한눈에 비교

|구분|개념적|논리적 ⭐|물리적|
|:--|:-:|:-:|:-:|
|**추상화 수준**|높음|중간|낮음|
|**핵심 작업**|엔터티/관계 도출|속성·식별자·정규화|타입·인덱스·반정규화|
|**DBMS 종속**|❌ 독립|❌ 독립|✅ 종속|
|**정규화**|❌|✅ 수행|반정규화 가능|
|**산출물**|개념 ERD|논리 ERD|DDL 스크립트|
|**주요 표기법**|Peter Chen|Barker / IE|DBMS별|

##  스키마의 3계층 (ANSI-SPARC)

데이터베이스를 보는 **관점(View)** 에 따라 3가지로 나눕니다.

> 💡 **암기 팁: "외.개.내"**

|계층|키워드|누가 보나?|몇 개?|설명|
|:--|:--|:--|:-:|:--|
|**외부 스키마** (External)|개인, 서브|사용자, 프로그래머|**여러 개 (N)**|"나는 내 것만 볼래." 개인별로 필요한 데이터만 보여주는 사용자 관점|
|**개념 스키마** (Conceptual)|통합, 전체|조직 전체, DBA|**오직 하나 (1)**|"학교 전체의 설계도." 모든 사용자가 공유하는 통합된 데이터 구조|
|**내부 스키마** (Internal)|물리, 저장|시스템 설계자|**오직 하나 (1)**|"하드디스크에 어떻게 넣지?" 데이터가 물리적으로 저장되는 구조|

> ⚠️ **시험 함정 주의:**
> 
> - "모든 사용자가 공유하는 뷰" → **개념 스키마**
> - "개별 사용자가 보는 뷰" → **외부 스키마**
> - **DBA** → 전체를 봐야 하니까 **개념 스키마** 관리
> - **프로그래머** → 자기 프로그램만 짜면 되니까 **외부 스키마** 참조

## 7데이터 독립성 (Data Independence) 

3계층 구조가 보장하는 **"방화벽"** 같은 역할입니다. 하위 단계의 구조를 변경해도, 상위 단계는 영향을 받지 않아야 한다는 원칙입니다.

> 💡 **암기 팁: "논.개.외" / "물.내.개"**

|구분|설명|변경 영향 범위|예시|
|:--|:--|:--|:--|
|**논리적 독립성** (Logical)|**개념 스키마**가 바뀌어도 **외부 스키마**는 멀쩡함|개념 ↔ 외부|테이블을 2개로 쪼개거나 통합해도, 사용자가 보는 화면(View)은 그대로|
|**물리적 독립성** (Physical)|**내부 스키마**가 바뀌어도 **개념 스키마**는 멀쩡함|내부 ↔ 개념|HDD에서 SSD로 옮기거나 인덱스를 바꿔도 테이블 구조는 그대로|

**비유로 이해하기:**

```
논리적 독립성 (논.개.외)
  식당 주방(개념) 리모델링을 해도 → 손님 메뉴판(외부)은 그대로 유지됨

물리적 독립성 (물.내.개)
  식재료 창고 위치(내부)를 옮겨도 → 주방(개념) 요리 방식은 안 바뀜
```

## 주의사항 (Common Misconceptions)

|오해|진실|
|:--|:--|
|"모델링은 DB 개발자만 하는 것이다?"|**아닙니다.** 모델링은 업무(Business)를 분석하는 과정이므로 기획자나 현업 담당자의 업무 이해도가 가장 중요합니다.|
|"무조건 정규화를 많이 하면 좋다?"|**아닙니다.** 논리적 모델링에서는 정규화가 필수지만, 물리적 모델링 단계에서는 성능을 위해 **반정규화**를 하기도 합니다.|
